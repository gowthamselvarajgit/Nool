The bridge between auth and business

A User row knows you can log in. But it doesn't know what you are in the business — a worker? a saree owner? UserProfile is the tiny table that answers that question, linking auth identity to business identity.

🤔 The Problem UserProfile Solves

We have three entities that represent people:

  • User — has login credentials (mobile, password, role)
  • Employee — has worker business data (name, joining date, polish rate)
  • SareeOwner — has client business data (name, rate per saree)

When the backend looks up a User by mobile during login, it needs to know: which Employee or SareeOwner is this person? Without that answer, the employeeId and ownerId claims in the JWT can't be populated, and the rest of the system can't filter "show me my data".

UserProfile is the answer. One small table with three columns — user_id, employee_id, owner_id — that maps each User to either an Employee row or a SareeOwner row (or neither, for ADMINs).

💭 Why Not Just Put These IDs on the User Table?

A fair question. We could have added employee_id and owner_id columns directly to users. Three reasons we didn't:

ReasonExplanation
Separation of concerns The users table is sacred — it's the authentication record. Adding business-related foreign keys to it would mix two concerns (auth + business mapping) into one table.
Easier evolution If we later add a fourth role with a different business entity, we just add a column to user_profiles — no migration to the touchier users table.
Independent lifecycle A profile mapping can be created or modified after the User is set up. Two separate steps with two clean audit trails.

📜 The Full File — Every Line Explained

auth/entity/UserProfile.java Java 
package com.nool.backend.auth.entity;

import jakarta.persistence.*;
import lombok.*;
import java.time.LocalDateTime;

@Entity
@Table(name = "user_profiles")            // SQL table = user_profiles (plural)
@Getter @Setter
@NoArgsConstructor
@AllArgsConstructor
@Builder
public class UserProfile {

    // ── PRIMARY KEY ───────────────────────────────────────────────
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    @Column(name = "profile_id")
    private Long id;

    // ── LINK TO THE USER ──────────────────────────────────────────
    @OneToOne(optional = false)                // every profile MUST belong to a user
    @JoinColumn(name = "user_id",
                nullable = false,             // FK column cannot be NULL
                unique   = true)              // each User has AT MOST one profile
    private User user;
    // ★ This is the FK side of the OneToOne relationship. User.java had
    //   mappedBy="user" on its side — that means the FK column user_id
    //   lives HERE, in user_profiles. Hibernate creates a `user_id` column
    //   referencing users.user_id.

    // ── EMPLOYEE LINK (nullable — only set if this is a WORKER) ───
    @Column(name = "employee_id")
    private Long employeeId;
    // ★ Just a Long, NOT a @ManyToOne to Employee. This is a deliberate
    //   "loose reference" — we store the ID but don't navigate to the
    //   Employee object from here. Why? See below ("Why no @ManyToOne?").

    // ── OWNER LINK (nullable — only set if this is a SAREE_OWNER) ─
    @Column(name = "owner_id")
    private Long ownerId;
    // Same pattern as employeeId — just an ID, no @ManyToOne navigation.

    // ── AUDIT TIMESTAMPS ──────────────────────────────────────────
    @Column(name = "created_at", nullable = false, updatable = false)
    private LocalDateTime createdAt;

    @Column(name = "updated_at")
    private LocalDateTime updatedAt;

    // ── LIFECYCLE HOOKS ───────────────────────────────────────────
    @PrePersist
    protected void onCreate(){
        LocalDateTime now = LocalDateTime.now();
        this.createdAt = now;
        this.updatedAt = now;
    }

    @PreUpdate
    protected void onUpdate(){
        this.updatedAt = LocalDateTime.now();
    }
}

⚡ Why employeeId Is Just a Long (Not a @ManyToOne)

Notice the design choice: employeeId is a plain Long, not a @ManyToOne relationship to the Employee entity. The same for ownerId. This is intentional.

The trade-off

Plain Long (what we chose)@ManyToOne (alternative)
StorageJust a number in the columnFK with referential constraint
NavigationNone — code looks up Employee by ID separatelyprofile.getEmployee().getName()
Lookup speedSame — index lookup either waySame
CouplingLoose — UserProfile doesn't know what an Employee isTight — Hibernate metadata links the two
FK validationNot enforced at DB levelEnforced — DB refuses orphan IDs
JWT buildingEasy — just profile.getEmployeeId(), no extra DB queryWould trigger a JOIN or lazy load just to read the ID
For UserProfile's job, "plain Long" wins. The class only exists to answer "what's the employee/owner ID for this user?" during login. We never need to navigate from a profile to the full Employee object. Storing just the ID keeps the design simple, avoids unnecessary JOINs, and lets the auth layer stay independent from the business modules.

The trade-off: there's no database-level FK constraint, so theoretically the employee_id could point to a non-existent employee. In practice, this is always set by the same service method that creates the Employee, so it's safe — but a stricter implementation would add the FK.

🎭 The Three Role Patterns in UserProfile

A UserProfile row looks slightly different depending on what kind of user it belongs to. Three patterns:

Pattern 1 — ADMIN profile

-- Mom & Dad's profile: just a User link, no business mapping
profile_id | user_id | employee_id | owner_id
    1      |    1    |    NULL     |   NULL

Admins don't have an Employee row or a SareeOwner row — they ARE the business. So both ID fields are NULL.

Pattern 2 — WORKER profile

-- Prema's profile: links User #3 to Employee #5
profile_id | user_id | employee_id | owner_id
    2      |    3    |     5       |   NULL

Pattern 3 — SAREE_OWNER profile

-- Rahul's profile: links User #2 to SareeOwner #1
profile_id | user_id | employee_id | owner_id
    3      |    2    |    NULL     |    1
Invariant the code enforces: a profile should have at most one of employee_id / owner_id set. Both being non-null doesn't make business sense (you can't be both a worker AND a client of yourself). This isn't enforced at the DB level — the service code is responsible.

🔑 How It's Used During Login

Inside AuthServiceImpl.login() (we'll deep-dive that file in Chapter 25), after BCrypt confirms the password, this happens:

Excerpt from AuthServiceImpl.login()Java
// Fetch the linked profile via the OneToOne navigation on User
UserProfile profile = user.getUserProfile();    // LAZY-loaded ONE time here

// Build the JWT claims using the profile's IDs
Map<String, Object> claims = new HashMap<>();
claims.put("userId",     user.getId());
claims.put("role",       user.getRole().name());
claims.put("employeeId", profile != null ? profile.getEmployeeId() : null);
claims.put("ownerId",    profile != null ? profile.getOwnerId()    : null);

// Mint the JWT
String token = jwtUtil.generateToken(claims, user.getMobileNumber());

So whenever any worker or owner makes a request, their JWT already contains their Employee/Owner ID — no extra DB lookup is needed on every request. Endpoints like GET /api/employees/me just read the employeeId claim from the JWT and use it directly.

🎬 Real-World Scenarios

Scenario 1: Dad creates a new worker (Prema)

The "Add Employee" flow in fact creates three rows across three tables:

  1. Create a User row: mobile=9000000002, password=<BCrypt hash>, role=WORKER. New ID = 3.
  2. Create an Employee row: name=Prema, polishRate=30.0, .... New ID = 5.
  3. Create a UserProfile row: user_id=3, employee_id=5, owner_id=NULL.

Now everything is linked. When Prema logs in later, the JWT will contain employeeId=5.

Scenario 2: Dad creates a new client (Rahul)
  1. User: mobile=9000000001, role=SAREE_OWNER. ID = 2.
  2. SareeOwner: ownerName="Rahul Textiles", polishRatePerSaree=65.0. ID = 1.
  3. UserProfile: user_id=2, employee_id=NULL, owner_id=1.

When Rahul logs in, the JWT contains ownerId=1. The endpoint POST /api/inventory/my-ledger reads it and fetches only Rahul's ledger entries.

Scenario 3: A worker transitions to a saree owner

Imagine Prema decides to start her own saree-trading business and brings sarees back to my parents' workshop as a client. Two options:

  • Option A: Create a separate User + SareeOwner + UserProfile triple for her business identity. She logs in with a different mobile.
  • Option B: Update her existing UserProfile to fill both employee_id and owner_id (and switch User.role to SAREE_OWNER).

NOOL's current implementation favors Option A — cleaner. Option B is technically possible but would confuse the rest of the system, which assumes profile fields are mutually exclusive.

Common UserProfile mistakes
Setting both employeeId and ownerId at the same time
The system assumes mutual exclusion. Only one of them should be non-null per row. The code enforces this in service methods; don't bypass.
Creating a UserProfile without a User
@OneToOne(optional = false) plus nullable=false on user_id means the FK is mandatory. Trying to save a profile with a null User throws at INSERT time.
Forgetting to create a UserProfile when adding an Employee or SareeOwner
Without the profile, login still works (User exists), but the JWT won't have employeeId/ownerId claims — every "my data" endpoint will break for that user. The service code must always create all three rows together inside one transaction.
Using @ManyToOne to Employee/SareeOwner here (with cascade)
Plain Long is intentional — keeps UserProfile decoupled from the business modules. A @ManyToOne would create cascade-delete risks and tight coupling.
Adding business logic methods (isEmployee(), getEmployee()) to this entity
Entities should stay anaemic — plain data, no business behavior. Lookup helpers belong in service classes.
Interview questions about UserProfile
Why have a bridge table at all? Why not just put employeeId and ownerId on the User entity?
Three reasons. (1) Separation of concerns — User is purely an auth entity; business links live elsewhere. (2) Lifecycle independence — a profile can be re-mapped without touching the User row. (3) Easier to extend — if we ever add a fourth role with a different business entity, we add a column to user_profiles, not the touchier users table.
Why is the OneToOne to User optional = false but the inverse on User isn't?
The owning side (UserProfile.user) is mandatory — every profile must link to a User. The inverse side (User.userProfile) is just a navigational helper; it can be null for an admin who has no profile yet, or it could be missing during a partial migration. Marking the owning side as optional=false enforces the data rule at the JPA level. The inverse side stays flexible.
Why are employeeId and ownerId plain Long fields instead of @ManyToOne?
To keep the auth module decoupled from the business modules. UserProfile only needs to ANSWER "what's the ID?", not navigate to the full entity. Using plain Long avoids unnecessary JOIN/lazy-load complexity, eliminates cascade risks, and lets the business entities evolve independently. The trade-off is no DB-level FK validation, which we accept because service code always sets these IDs correctly.
What happens if I try to load a User with no UserProfile?
user.getUserProfile() returns null. The auth code explicitly handles that case (profile != null ? profile.getEmployeeId() : null), which is right for admins (they have no profile). For workers/owners, it would mean something's broken in the data — but the system fails gracefully (JWT just gets null for those claims).
Could you replace UserProfile with a discriminator column on User?
You could use JPA's @Inheritance(strategy = JOINED) with subclass entities (AdminUser, WorkerUser, OwnerUser). It's cleaner type-wise but more complex structurally — three tables instead of two, more careful with persist/merge cascading. NOOL's pragmatic flat approach is simpler for a small project and gives the same effective result.
Why is the table called user_profiles (plural) but you used user_profile in the chapter 04 diagram?
The chapter 04 description had a small inaccuracy — the actual table name is plural (user_profiles) as Hibernate generates it from the @Table(name = "user_profiles") annotation here. This is more consistent with the other plural table names (users, employees, policies etc.). It's a small detail but worth getting right.
