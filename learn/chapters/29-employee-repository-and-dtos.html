1️⃣ EmployeeRepository.java

repository/employee/EmployeeRepository.javaJava
package com.nool.backend.repository.employee;

import com.nool.backend.entity.employee.Employee;
import com.nool.backend.enums.EmployeeStatus;
import org.springframework.data.domain.Page;
import org.springframework.data.domain.Pageable;
import org.springframework.data.jpa.repository.JpaRepository;
import org.springframework.data.jpa.repository.Query;
import org.springframework.data.repository.query.Param;
import org.springframework.stereotype.Repository;
import java.util.List;

@Repository
public interface EmployeeRepository extends JpaRepository<Employee, Long> {

    // ── 1. Find employees by status (ACTIVE / LEFT) ─────────────
    List<Employee> findByStatus(EmployeeStatus status);

    // ── 2. Count employees by status ────────────────────────────
    long countByStatus(EmployeeStatus status);

    // ── 3. Custom JPQL — all active employees ───────────────────
    @Query("SELECT e FROM Employee e WHERE e.status = 'ACTIVE'")
    List<Employee> findAllActiveEmployees();

    // ── 4. Duplicate-check by mobile ────────────────────────────
    boolean existsByMobileNumber(String mobileNumber);

    // ── 5. Search by name OR mobile (paginated) ─────────────────
    @Query("SELECT e FROM Employee e WHERE " +
           "LOWER (e.name) LIKE LOWER(CONCAT('%', :keyword, '%')) OR " +
           "e.mobileNumber LIKE CONCAT('%', :keyword, '%')")
    Page<Employee> searchEmployees(@Param("keyword") String keyword,
                                    Pageable pageable);
}

Method-by-method reference

MethodGenerated/declared SQLUsed for

countByStatus(status)SELECT COUNT(*) FROM employees WHERE status = ?Admin dashboard count widgets

Two ways to declare queries — both used here

Spring Data JPA gives us two ways to define repository methods, and EmployeeRepository uses both:

ApproachHow it worksWhen to use
Method-name parsing Spring parses the method name (findByStatus) and generates SQL automatically Simple queries on one or two fields. findByX, existsByX, countByX.
Explicit @Query (JPQL) You write the query yourself in JPQL (a Java version of SQL) Anything beyond trivial — OR conditions, LOWER(), LIKE patterns, JOINs.

JPQL vs SQL — a quick note

JPQL (in @Query)JPQL
SELECT e FROM Employee e WHERE e.status = 'ACTIVE'
Translated SQL Hibernate generatesSQL
SELECT * FROM employees WHERE status = 'ACTIVE'

JPQL works on Java entities (Employee, field status); Hibernate translates to SQL using the entity → table mapping. Slightly more abstract — and DB-agnostic.

Why not use findByNameContainingIgnoreCaseOrMobileNumberContaining(...)? You could! Spring Data's name-parser supports those keywords. But once a query has OR and pattern matching, the method name becomes very long. The @Query version is more readable. Both styles work.

2️⃣ CreateEmployeeRequestDto.java

dto/employee/CreateEmployeeRequestDto.javaJava
package com.nool.backend.dto.employee;

import jakarta.validation.constraints.*;
import lombok.Data;
import java.time.LocalDate;

@Data
public class CreateEmployeeRequestDto {

    @NotBlank(message = "Employee name is required")
    private String employeeName;

    @NotNull(message = "Joining date is required")
    private LocalDate joiningDate;

    @NotNull(message = "Polishing rate is required")
    @Positive(message = "Polishing rate must be greater than zero")
    private Double polishingRate;

    @NotBlank(message = "Mobile number is required")
    @Pattern(
            regexp = "^[6-9][0-9]{9}$",
            message = "Mobile number must be a valid 10-digit Indian mobile number"
    )
    private String mobileNumber;

    @NotBlank(message = "Password is required")
    private String password;
}

The validation rules at a glance

FieldAnnotationsValid exampleRejected example
employeeName@NotBlank"Prema""" or " " or null
joiningDate@NotNull"2026-05-15"null
polishingRate@NotNull, @Positive30.00, -1, null
mobileNumber@NotBlank, @Pattern(^[6-9][0-9]{9}$)"9000000002""123", "5000000000", "abc"
password@NotBlank"prema123""" or null

The mobile-number regex explained

That ^[6-9][0-9]{9}$ deserves a closer look — it enforces Indian mobile format:

  • ^ — start of string
  • [6-9] — first digit MUST be 6, 7, 8, or 9 (Indian mobile prefixes)
  • [0-9]{9} — followed by exactly 9 more digits
  • $ — end of string (so total length is exactly 10)

This catches typos like "5xxxxxxxxx" (landline-prefix), short numbers, and any non-digit characters.

@Data
public class UpdateEmployeeRequestDto {

    private String employeeName;

    @Positive(message = "Polishing rate must be greater than zero")
    private Double polishingRate;

    @NotBlank(message = "Mobile number is required")

    )

}

How update differs from create

CreateUpdate
No employeeId — server assigns itemployeeId required — points to the row to edit
Has passwordNO password (passwords are not updated through this endpoint)
Has joiningDateNO joining date (historical — never edit)
Status auto-set to ACTIVEStatus NOT in this DTO (separate endpoint for status flips)
Why are these omissions deliberate?
  • password — security. Password changes deserve their own endpoint with extra checks (old-password verification, etc.). Bundling it with name edits is a recipe for accidents.
  • joiningDate — it's historical fact. Should never be edited.
  • status — status changes have business consequences (worker can/can't log in). A separate endpoint isolates that intent.

What's NOT in the response

  • password — never expose
  • user_id — internal foreign-key, not useful to the frontend
  • createdAt / updatedAt — internal audit, not shown in worker lists
  • The User object — would leak the BCrypt password hash
DTO as a security boundary: the response DTO controls EXACTLY what data the frontend (and therefore anyone with browser dev tools) can see. Returning the entity directly would expose the BCrypt password hash and other internal fields. The DTO is the wall between "what we store" and "what we expose".

6️⃣ EmployeeListResponse.java

dto/employee/EmployeeListResponse.javaJava
package com.nool.backend.dto.employee;

import lombok.Builder;
import lombok.Data;
import java.time.LocalDate;

@Data @Builder
public class EmployeeListResponse {
    private Long      employeeId;
    private String    employeeName;
    private LocalDate joiningDate;
    private Double    polishingRate;
    private String    status;
    private String    mobileNumber;
}

This DTO is currently identical to EmployeeResponseDto. Why two classes with the same fields?

Two reasons.

(1) Future divergence. List endpoints often want LESS detail than single-row endpoints (e.g., omit the mobile from a public list view to save bandwidth). Keeping a separate class lets us tune that later without touching all the single-row consumers.

(2) Type-level clarity. When a function signature returns PaginationResponseDto<EmployeeListResponse>, the reader instantly knows "this is a list page", not a single employee. Saves a comment.

🎬 Real-World Scenarios

Scenario 1: Dad searches for "Pre" in the employee list
  1. Frontend posts POST /api/employees/list with body { "page": 0, "size": 10, "searchKeyword": "Pre", "sortBy": "createdAt", "sortingDirection": "DESC" }.
  2. Service builds a PageRequest, calls employeeRepository.searchEmployees("Pre", pageable).
  3. Repository's @Query fires: SELECT e FROM Employee e WHERE LOWER(e.name) LIKE '%pre%' OR e.mobileNumber LIKE '%Pre%'.
  4. Hibernate translates to MySQL, executes, returns a Page of Employee entities.
  5. Service maps each Employee → EmployeeListResponse (excluding password, audit fields).
  6. Wraps in PaginationResponseDto, returns to controller, serialized to JSON.
  7. Frontend renders the table with pagination.
Scenario 2: Dad tries to add a worker with mobile "1234567890"
  1. Frontend posts POST /api/employees with bad mobile.
  2. Spring deserializes into CreateEmployeeRequestDto.
  3. @Valid kicks in. @Pattern on mobileNumber: "1234567890" → first digit is "1", not in [6-9]. FAILS.
  4. Spring throws MethodArgumentNotValidException before the controller body runs.
  5. GlobalExceptionHandler returns 400 with: "mobileNumber: Mobile number must be a valid 10-digit Indian mobile number".
  6. Service is never called — no wasted DB hit.
Scenario 3: Admin counts active workers for the dashboard
  1. Dashboard service calls employeeRepository.countByStatus(EmployeeStatus.ACTIVE).
  2. SQL: SELECT COUNT(*) FROM employees WHERE status = 'ACTIVE'.
  3. Returns 7 — the count is rendered as a dashboard stat card.
Scenario 4: Quick mobile-uniqueness check before insert
  1. EmployeeServiceImpl.createEmployee receives a new request.
  2. Calls employeeRepository.existsByMobileNumber("9000000002").
  3. Returns true → throws DuplicateResourceException BEFORE attempting any insert.
  4. This is faster than letting MySQL's UNIQUE constraint throw — the friendly check up-front gives a better error message and avoids transaction rollback.
Common mistakes
Returning Employee (the entity) directly from controllers
Always map to a DTO. The entity exposes the BCrypt password hash, internal timestamps, and lazy-loaded relationships that can trigger N+1 queries.
Including password in the Update DTO
Password changes deserve their own endpoint with extra checks. Bundling password edits with name/rate edits invites accidents.
Using the same DTO for create and update
They have different rules — create needs password + joining date, update doesn't. Separate DTOs make the rules visible at the type level.
Forgetting @Repository on the interface
Spring auto-detects JpaRepository-derived interfaces, but @Repository makes the role explicit and provides exception translation (converts JDBC SQLExceptions into Spring's DataAccessException hierarchy).
Writing the search query as findByNameOrMobileNumberContainingIgnoreCase(...)
For complex queries, the method name becomes hard to read. @Query with named parameters is clearer for OR + LIKE + LOWER.
Returning List<Employee> from a search/list endpoint
Return a paginated response. The employees table could grow to thousands of rows; loading them all blows up memory. Page<Employee> + PaginationResponseDto handle pagination naturally.
Interview questions
Why are there two DTOs (EmployeeResponseDto and EmployeeListResponse) with the same fields?
Future-proofing and type-level clarity. The fields are identical today, but list endpoints frequently diverge later (e.g., trim to a few columns for performance). Keeping them separate now means we can tune one without affecting the other. The type also documents intent at function signatures.
When should you use @Query vs method-name derivation?
Use method-name derivation for simple queries on 1-2 fields. Switch to @Query when the method name would become unwieldy — OR conditions, function calls like LOWER(), JOINs across entities, or anything more complex than a few finder keywords.
Why existsByMobileNumber instead of findByMobileNumber + isPresent?
Performance. existsBy generates a query that the database can short-circuit at the first matching row (SELECT 1 ... LIMIT 1). findBy SELECTs all columns of all matching rows, which is wasteful when you only need a boolean.
What does @Pattern validate?
It runs the field's value against a regex. If the regex matches, the field is valid. Used here to enforce Indian mobile format (10 digits starting with 6-9). The error message in the annotation is what GlobalExceptionHandler returns when validation fails.
Why is the password field in CreateEmployeeRequestDto a plain String, not a special type?
The DTO is just a transport object. The plain-text password lives in memory only for the duration of the request, then is hashed in AdminUserService before any persistence. Adding a "SecureString" type would be over-engineering for a single transient field.
Why does the search method return Page<Employee> instead of List<Employee>?
Spring Data's Page includes total count, current page index, page size, total pages — useful for pagination UIs. The service later maps this to PaginationResponseDto which exposes the same info to the frontend. List would lose the totals.
