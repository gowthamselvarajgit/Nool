Custom Exceptions &
GlobalExceptionHandler

Every real application needs a clear, consistent way to say "that didn't work". NOOL has three small custom exception classes and one big handler that catches everything and turns it into a clean JSON error response.

🧠 Why Custom Exceptions?

Java has plenty of built-in exceptions (NullPointerException, IllegalArgumentException, etc.) — why invent our own?

Two reasons:

  1. Clarity in service code. When I write throw new ResourceNotFoundException("Owner not found"), anyone reading the line instantly knows what kind of error happened. Compare that to throw new RuntimeException("Owner not found") — vague.
  2. Different exception types map to different HTTP status codes. A "not found" should return 404. A "duplicate" should return 409 Conflict. A "bad input" should return 400. The handler uses the exception's class to pick the right status automatically.
The architecture: Service code throws specific custom exceptions. GlobalExceptionHandler intercepts every exception that escapes a controller, looks at its class, and produces a uniform JSON response with the right HTTP status. The frontend then reads that JSON and shows the user a friendly toast.

1️⃣ The Three Custom Exceptions

All three are deliberately tiny — just 7 lines each. They exist purely to give errors a meaningful name that the handler can pattern-match on.

1.1 BadRequestException — "You gave me bad input"

exception/BadRequestException.javaJava
package com.nool.backend.exception;

public class BadRequestException extends RuntimeException {
    public BadRequestException(String message) {
        super(message);  // pass the message up to RuntimeException
    }
}

When I throw this: when the request is structurally valid but the values are wrong for business reasons. Example: "Cannot return 150 sarees for Rahul — only 30 in hand." The HTTP method and URL are fine; the data inside isn't.

Maps to HTTP status: 400 Bad Request

1.2 ResourceNotFoundException — "That thing doesn't exist"

exception/ResourceNotFoundException.javaJava
package com.nool.backend.exception;

public class ResourceNotFoundException extends RuntimeException {

    }

When I throw this: when looking up a specific resource (an Owner, an Employee, a Ledger entry) and the database returns nothing. Common pattern: repository.findById(id).orElseThrow(() -> new ResourceNotFoundException("Owner not found")).

2️⃣ The Shape of Every Error — ApiErrorResponse

Before we look at the handler, let's see what shape the handler will produce. Every error response — no matter which exception triggered it — has exactly the same five fields. That uniformity is what lets the frontend handle errors generically.

dto/common/ApiErrorResponse.javaJava
package com.nool.backend.dto.common;

import lombok.Builder;
import lombok.Data;
import java.time.LocalDateTime;

@Data      // Lombok: @Getter + @Setter + toString + equals + hashCode
@Builder   // Lombok: enables ApiErrorResponse.builder()...build()
public class ApiErrorResponse {
    private LocalDateTime timeStamp;   // when the error happened
    private int          status;      // HTTP status code (400, 404, 409, etc.)
    private String        error;       // short label like "Bad Request"
    private String        message;     // human-friendly description
    private String        path;        // the URL that was being called
}

3️⃣ GlobalExceptionHandler — The Catch-All

This is the file that intercepts every uncaught exception across the whole backend. It's ~200 lines, but the structure is simple: one method per kind of exception. Each method builds an ApiErrorResponse with the appropriate status and message.

The annotation that makes it work — @RestControllerAdvice

A regular @RestController handles HTTP requests. @RestControllerAdvice is its sibling: a class that handles things across all controllers — exceptions, response post-processing, etc. Spring auto-detects it at startup and registers each @ExceptionHandler method as a global interceptor.

The class header

exception/GlobalExceptionHandler.java — top of fileJava
package com.nool.backend.exception;

import com.nool.backend.dto.common.ApiErrorResponse;
import jakarta.servlet.http.HttpServletRequest;
import org.springframework.dao.DataIntegrityViolationException;
import org.springframework.http.HttpStatus;
import org.springframework.http.ResponseEntity;
import org.springframework.http.converter.HttpMessageNotReadableException;
import org.springframework.web.bind.MethodArgumentNotValidException;
import org.springframework.web.bind.annotation.ExceptionHandler;
import org.springframework.web.bind.annotation.RestControllerAdvice;
import org.springframework.web.servlet.NoHandlerFoundException;
import java.time.LocalDateTime;

@RestControllerAdvice   // ★ Spring auto-registers this for global exception handling
public class GlobalExceptionHandler {
    // ... handler methods below
}

3.1 Handler — ResourceNotFoundException → 404

GlobalExceptionHandler — 404 Not FoundJava
@ExceptionHandler(ResourceNotFoundException.class)
public ResponseEntity<ApiErrorResponse> handleNotFound(
        ResourceNotFoundException exception,
        HttpServletRequest        request) {

    ApiErrorResponse errorResponse = ApiErrorResponse.builder()
            .timeStamp(LocalDateTime.now())                 // when
            .status(HttpStatus.NOT_FOUND.value())           // 404
            .error("Not Found")                          // label
            .message(exception.getMessage())                // "Owner not found"
            .path(request.getRequestURI())                  // /api/owners/99
            .build();

    return ResponseEntity.status(HttpStatus.NOT_FOUND).body(errorResponse);
}

How it gets called: Anywhere in NOOL's service code, when I write throw new ResourceNotFoundException("Owner not found"), Spring sees the exception bubble up, finds this handler (registered for that exception class), invokes it, and the client receives a 404 JSON response.

3.3 Handler — Validation failures → 400

When a request DTO has @NotBlank, @Min etc. validation annotations and the input fails them, Spring throws MethodArgumentNotValidException. We catch that and pull the first field-level error message out.

GlobalExceptionHandler — 400 ValidationJava
@ExceptionHandler(MethodArgumentNotValidException.class)
public ResponseEntity<ApiErrorResponse> handleValidation(
        MethodArgumentNotValidException exception,
        HttpServletRequest              request) {

    // Pick the FIRST field error to show the user.
    // (Could concatenate them all, but one is enough to act on.)
    String message = exception.getBindingResult()
            .getFieldErrors()
            .stream()
            .map(error -> error.getField() + ": " + error.getDefaultMessage())
            .findFirst()
            .orElse("Validation failed");

    ApiErrorResponse errorResponse = ApiErrorResponse.builder()
            .timeStamp(LocalDateTime.now())
            .status(HttpStatus.BAD_REQUEST.value())
            .error("Bad Request")
            .message(message)                               // e.g., "mobileNumber: must be 10 digits"
            .path(request.getRequestURI())
            .build();

    return ResponseEntity.badRequest().body(errorResponse);
}

3.6 Handler — Database integrity errors → smart messages ⭐

This is the most sophisticated handler in the file. When the database itself rejects something — usually a foreign-key violation or a unique-constraint violation — Hibernate wraps it in a DataIntegrityViolationException. The raw error message is cryptic ("Duplicate entry '9000000001' for key 'mobile_number'"). We translate it into something a non-technical user can act on.

GlobalExceptionHandler — DataIntegrityViolationJava
@ExceptionHandler(DataIntegrityViolationException.class)
public ResponseEntity<ApiErrorResponse> handleDataIntegrityViolation(
        DataIntegrityViolationException exception,
        HttpServletRequest              request) {

    String path       = request.getRequestURI();
    String rootCause  = extractRootCauseMessage(exception);              // dig past wrapper layers
    String lowerCause = rootCause == null ? "" : rootCause.toLowerCase();

    String     message;
    HttpStatus status = HttpStatus.CONFLICT;

    // ── Inspect the actual database error first, not just the URL ────────
    if (lowerCause.contains("foreign key")
            || lowerCause.contains("a foreign key constraint fails")) {
        status = HttpStatus.CONFLICT;
        if (path.contains("/employees")) {
            message = "Cannot delete this employee — they have attendance, " +
                      "salary, or daily-work history. Mark them as 'Left Job' instead.";
        } else if (path.contains("/owners")) {
            message = "Cannot delete this owner — they have transaction or " +
                      "payment history. Mark them as 'Not Working' instead.";
        } else {
            message = "Cannot complete the action: related records exist. " +
                      "Underlying error: " + rootCause;
        }
    } else if (lowerCause.contains("duplicate entry")
            || lowerCause.contains("unique constraint")) {
        if (lowerCause.contains("mobile") || lowerCause.contains("mobile_number")) {
            message = "This mobile number is already registered. " +
                      "Please use a different one.";
        } else if (lowerCause.contains("attendance_date") || path.contains("/attendance")) {
            message = "Attendance is already marked for this employee on this date.";
        } else if (lowerCause.contains("work_date") || path.contains("/employee-daily-working")) {
            message = "A daily work entry already exists for this employee on this date.";
        } else if (path.contains("/salary")) {
            message = "A salary payment already exists for this period.";
        } else {
            message = "Duplicate entry detected. Underlying error: " + rootCause;
        }
    } else if (lowerCause.contains("cannot be null")
            || lowerCause.contains("not-null property")) {
        status = HttpStatus.BAD_REQUEST;
        message = "A required field is missing. Underlying error: " + rootCause;
    } else {
        message = "Data conflict. Underlying error: " + rootCause;
    }

    ApiErrorResponse errorResponse = ApiErrorResponse.builder()
            .timeStamp(LocalDateTime.now())
            .status(status.value())
            .error(status.getReasonPhrase())
            .message(message)
            .path(path)
            .build();

    return ResponseEntity.status(status).body(errorResponse);
}

// Helper: drill down through wrapper exceptions to the real root cause
private String extractRootCauseMessage(Throwable t) {
    Throwable cur = t;
    Throwable last = t;
    int depth = 0;
    while (cur != null && depth++ < 10) {      // max 10 levels — safety
        last = cur;
        cur = cur.getCause();
    }
    return last.getMessage();
}
Why this matters so much: a raw MySQL error like "Duplicate entry '9000000001-2026-05-14' for key 'attendance.UK6e5a...'" means nothing to my mom. But "Attendance is already marked for this employee on this date" — she immediately knows what to do. This handler is the bridge between database reality and user-friendly messages.

3.7 Handler — Anything else → 500

For exceptions we didn't anticipate (a real bug, a server error), there's a last-resort handler. It deliberately re-throws Spring Security exceptions so they're handled by Spring's own filters (which produce proper 401/403 responses).

GlobalExceptionHandler — 500 fallbackJava
@ExceptionHandler(Exception.class)
public ResponseEntity<ApiErrorResponse> handleGenric(
        Exception          exception,
        HttpServletRequest request) throws Exception {

    // Let Spring Security handle its own access-denied/authentication errors
    if (exception instanceof org.springframework.security.access.AccessDeniedException
            || exception instanceof org.springframework.security.core.AuthenticationException) {
        throw exception;       // re-throw — Spring Security has its own response logic
    }

    ApiErrorResponse errorResponse = ApiErrorResponse.builder()
            .timeStamp(LocalDateTime.now())
            .status(HttpStatus.INTERNAL_SERVER_ERROR.value())
            .error("Internal Server Error")
            .message(exception.getMessage() != null
                     ? exception.getMessage()
                     : "An unexpected error occurred")
            .path(request.getRequestURI())
            .build();

    return ResponseEntity.internalServerError().body(errorResponse);
}

🎬 Real Scenarios

Scenario 1: Dad tries to delete a worker who has salary history
  1. Dad clicks "Delete" on Prema's row in the Employees page.
  2. Frontend sends DELETE /api/employees/3.
  3. Backend issues DELETE FROM employees WHERE employee_id=3.
  4. MySQL refuses — Prema has rows in employee_daily_work referencing her.
  5. MySQL throws a foreign-key violation, Hibernate wraps it in DataIntegrityViolationException.
  6. handleDataIntegrityViolation reads the root cause, sees "foreign key" and path "/employees".
  7. Returns: "Cannot delete this employee — they have attendance, salary, or daily-work history. Mark them as 'Left Job' instead."
  8. Dad reads the message and flips status to LEFT instead.
Scenario 2: Admin marks Prema present twice on the same day
  1. Mom marks Prema PRESENT for 2026-05-15. Saves.
  2. She doesn't realize, and a few minutes later marks her PRESENT for the same date again.
  3. Backend tries to INSERT a second row.
  4. The UNIQUE constraint (employee_id, attendance_date) blocks it.
  5. handleDataIntegrityViolation sees "duplicate entry" + "attendance_date" in the cause.
  6. Returns: "Attendance is already marked for this employee on this date."
  7. The UI shows a friendly toast — Mom realizes her mistake and moves on.
Scenario 3: Trying to receive sarees for an owner that doesn't exist
  1. Frontend sends POST /api/inventory/receipt with ownerId: 99.
  2. SareeInventoryServiceImpl.addReceipt() calls findByIdForUpdate(99).
  3. Repository returns empty Optional. .orElseThrow(...) fires.
  4. Service throws new ResourceNotFoundException("Owner not found").
  5. Handler returns 404 with body: {"status": 404, "message": "Owner not found", "path": "/api/inventory/receipt"}.
  6. Frontend's handleResponse in api.js reads it, throws JS Error, the React component shows the error in a toast.
Common exception-handling mistakes
Catching exceptions inside controllers / services and silently swallowing them
Let them propagate. The handler exists precisely so each layer doesn't have to know about HTTP. try { ... } catch(Exception e) { return null; } is almost always a bug.
Throwing raw RuntimeException("Owner not found")
Throw the specific subclass. The handler chooses 404 vs 400 vs 409 based on exception class. A raw RuntimeException would hit the 500 fallback and look like a server bug.
Including the stack trace or internal class names in user-facing messages
The message field is what users read. Keep it human-friendly. Internal details belong in the server logs (which Spring Boot logs automatically).
Letting Spring Security exceptions get caught by the generic 500 handler
Notice we explicitly re-throw AccessDeniedException and AuthenticationException. Spring Security has its own response writer that produces proper 401/403 — don't intercept those.
Hard-coding the same string in two handlers
If the same message pattern appears 3+ times, extract a helper method. (NOOL's handler is small enough that direct duplication is OK.)
Interview questions about exception handling
What does @RestControllerAdvice do?
It marks a class as a global exception handler / response advisor that applies to all @RestController-annotated controllers. Methods annotated with @ExceptionHandler(SomeException.class) inside it are auto-registered. When any controller's method throws SomeException, Spring routes it here, runs the method, and uses the returned ResponseEntity as the HTTP response.
Why does NOOL have custom exceptions instead of using Spring's ResponseStatusException?
Two reasons. First, custom classes are more expressive — ResourceNotFoundException reads better than new ResponseStatusException(NOT_FOUND, "..."). Second, they keep HTTP concerns out of services — services throw business-level exceptions, and only the handler maps them to HTTP. That separation matters if I ever expose the same services through a different protocol (gRPC, messaging).
Why does DataIntegrityViolationException need its own handler instead of being caught by the generic one?
Because the raw exception message is cryptic — something like "Duplicate entry '9000000001' for key 'mobile_number'". For technical users that's enough; for my mom using NOOL on a Saturday morning it's useless. The handler parses the root cause and produces a human-friendly translation. This is one of the most impactful pieces of UX work in the whole backend.
Why is extractRootCauseMessage capped at depth 10?
Pure defensive programming. Exception chains can in theory have cycles (rare but possible with bad code). Without the cap, this loop could spin forever. 10 levels is way more than any real chain ever has — typical depth is 2-3.
What happens if I throw an exception not handled by any @ExceptionHandler method?
The fallback @ExceptionHandler(Exception.class) catches it and returns 500. Without that, Spring's default would return a 500 with a generic error page — but the JSON shape would differ from the rest of our responses. The fallback ensures uniformity.
Where does the handler log the actual stack trace?
It doesn't — currently. Spring Boot's default error logging still records the stack trace to the console because the exception bubbles up. If I wanted explicit log lines per handler, I'd inject a Logger and call log.error(...) at the top of each method. For NOOL's scale, the default is fine.
