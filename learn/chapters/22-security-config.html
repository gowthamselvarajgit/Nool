SecurityConfig.java —
Who can access what

This is where NOOL declares every authorization rule: who can add employees, who can approve payments, who can view inventory. One Java file controls the entire security policy. Read it carefully — it's the law of the application.

🧠 What This File Controls

SecurityConfig is a Spring @Configuration class that defines:

  1. The password hasher (BCrypt) — used to hash + verify passwords
  2. CORS settings — which frontend origins can call the backend
  3. The security filter chain — the master URL → role mapping
  4. The JWT filter slot — where our custom filter plugs into Spring's chain
One file, the whole authorization policy. If a rule needs to change (e.g., "let workers also view their own owner payments"), you edit one line here. No annotations scattered across 50 controllers, no copy-paste mistakes.

⚙️ The Three Beans + One Filter Chain

SecurityConfig defines four things. We'll go through each.

Part 2 — The password encoder bean

BCrypt for password hashingJava
@Bean
public PasswordEncoder passwordEncoder() {
    return new BCryptPasswordEncoder();
}

Exposes a single BCrypt bean used throughout the auth code. Whenever a service does passwordEncoder.encode(plainPassword) or passwordEncoder.matches(plain, hash), this is the bean that runs. BCrypt was covered in detail in Chapter 11 (User entity) — it's one-way, adds a salt automatically, and is industry-standard.

Part 3 — CORS configuration

CORS: which frontend origins can call this backendJava
@Bean
public CorsConfigurationSource corsConfigurationSource() {
    CorsConfiguration configuration = new CorsConfiguration();
    configuration.setAllowedOrigins(List.of(
            "http://localhost:5173",         // Vite dev server default port
            "http://localhost:5174",         // alternate dev port
            "http://localhost:4173",         // Vite preview server
            "http://127.0.0.1:5173",
            "http://127.0.0.1:5174",
            "http://127.0.0.1:4173"
    ));
    configuration.setAllowedMethods(List.of(
            "GET", "POST", "PUT",
            "DELETE", "PATCH", "OPTIONS"));    // all standard HTTP verbs
    configuration.setAllowedHeaders(List.of("*"));   // allow Authorization, Content-Type, etc.
    configuration.setAllowCredentials(true);          // allow cookies if any (we use JWT in headers)

    UrlBasedCorsConfigurationSource source =
            new UrlBasedCorsConfigurationSource();
    source.registerCorsConfiguration("/**", configuration);   // apply to all paths
    return source;
}

What's CORS? Cross-Origin Resource Sharing. By default, browsers refuse to let a page from http://localhost:5173 (frontend) call APIs on http://localhost:8083 (backend) — different ports = different origins = blocked. This bean tells the browser "allow these origins". Without it, the React app couldn't talk to the backend at all in dev mode.

Part 4 — The security filter chain (the heart of the file)

This is where every URL pattern is mapped to a required role. It's long, so we'll go through it section by section.

SecurityFilterChain beanJava
@Bean
public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {

    http
        // ── Wire up CORS using the bean above ─────────────────────────
        .cors(cors -> cors.configurationSource(corsConfigurationSource()))

        // ── Disable CSRF — not needed for stateless REST APIs ─────────
        .csrf(csrf -> csrf.disable())

        // ── No HTTP sessions — every request authenticates via its JWT ─
        .sessionManagement(session ->
                session.sessionCreationPolicy(SessionCreationPolicy.STATELESS))

        // ── Disable browser-default auth UIs we don't use ─────────────
        .httpBasic(basic -> basic.disable())
        .formLogin(form -> form.disable())

The setup so far: CORS on, CSRF off (we don't need it for stateless APIs), no sessions (every request is independent), no default login forms (we have our own /api/auth/login endpoint).

🎯 How Rule Matching Actually Works

Rules are evaluated top to bottom. The FIRST rule whose pattern + method matches is the one that decides. That's why ordering matters.

Example: a worker calls GET /employees/me.

  1. Does /auth/** match? No, URL is /employees/me.
  2. Does /admin/** match? No.
  3. Does POST /employees match? No, method is GET not POST.
  4. Does PUT /employees match? No.
  5. ... continues down ...
  6. Does GET /employees/me match? Yes! Rule says: ADMIN or WORKER. Worker passes. Request allowed.
Ordering matters! If GET /employees/** came BEFORE GET /employees/me, the more general rule would match first and the specific one would never be reached. Specific rules must come before catch-alls.

🛡️ The Final Catch-All

The last rule is .anyRequest().authenticated(). This means: "any URL not matched above must at least be logged in". It's the safety net.

Without this rule, an URL not listed above would be reachable WITHOUT any auth at all — a massive security hole. The catch-all says "by default, deny unless authenticated".

🌱 Why Beans?

Each @Bean-annotated method returns an object that Spring registers in its IoC container. Other parts of the app can then inject these beans by type:

  • PasswordEncoder — injected into AuthServiceImpl and AdminUserServiceImpl
  • CorsConfigurationSource — used by the SecurityFilterChain bean
  • SecurityFilterChain — picked up automatically by Spring Security

The Spring framework discovers all @Bean methods at startup, creates the objects, and wires them where they're needed. No manual new BCryptPasswordEncoder() calls anywhere else in the codebase.

🎬 Real-World Scenarios

Scenario 1: Worker tries to add an employee (should fail)
  1. Prema (WORKER) sends POST /api/employees trying to add a fake employee.
  2. JwtAuthenticationFilter verifies her token, sets security context with role WORKER.
  3. SecurityConfig rules evaluate. Matching rule: POST /employees → hasAuthority("ADMIN").
  4. Prema's authority is WORKER, not ADMIN. Rule rejects.
  5. Response: 403 Forbidden.
Scenario 2: Worker views her own profile (should succeed)
  1. Prema sends GET /api/employees/me.
  2. Filter sets context with role WORKER.
  3. SecurityConfig: matching rule is GET /employees/me → hasAnyAuthority("ADMIN", "WORKER").
  4. Prema is WORKER → passes.
  5. Controller runs, uses CurrentUserUtil.getEmployeeId() to find Prema's row, returns it.
Scenario 3: Owner accesses inventory of another owner (should fail at service layer, not security)
  1. Rahul (SAREE_OWNER) somehow sends POST /api/inventory/owner/2/ledger (someone else's owner_id).
  2. SecurityConfig: matching rule is POST /inventory/owner/*/ledger → hasAuthority("ADMIN").
  3. Rahul is not ADMIN → blocked with 403.
  4. Note: Even if Rahul tried /inventory/my-ledger, the service layer would scope it to his own owner ID via CurrentUserUtil.getOwnerId() — defense in depth.
Scenario 4: Login attempt (no token yet)
  1. Frontend sends POST /api/auth/login with mobile + password.
  2. JwtAuthenticationFilter's shouldNotFilter sees /api/auth/ prefix → SKIP.
  3. SecurityConfig rule: /auth/** → permitAll().
  4. Request reaches AuthController.login().
  5. Service verifies password, mints JWT, returns it.
Common SecurityConfig mistakes
Putting catch-all rules BEFORE specific ones
Rules are matched top-down, first wins. Put specific rules first, catch-alls last. Otherwise the catch-all swallows everything.
Using permitAll() too liberally to "make things work"
permitAll() means literally anyone can call it — even unauthenticated requests. Use sparingly: only login, register, public health-check endpoints.
Disabling CSRF for stateful session-based apps
CSRF is needed for cookie-based auth. NOOL is stateless + JWT in headers → CSRF disabled is correct. But don't blindly copy this to a session-based app — you'd open an attack vector.
Wildcarding the CORS origin to *
Explicit origins only. Wildcard origins + credentials = security hole. NOOL lists each allowed dev port explicitly, and production would add the production frontend URL.
Forgetting .addFilterBefore(jwtAuthenticationFilter, ...)
Without this line, our JWT filter wouldn't run — Spring would never know about our custom auth flow. The order matters: must be before UsernamePasswordAuthenticationFilter.
Mixing hasRole() and hasAuthority() inconsistently
In Spring Security, hasRole("X") implicitly looks for authority "ROLE_X" (adds the prefix). hasAuthority("X") checks for the literal string. NOOL stores roles WITHOUT the ROLE_ prefix, so we use hasAuthority() consistently.
Interview questions about SecurityConfig
What's the difference between hasRole and hasAuthority in Spring Security?
hasRole("ADMIN") implicitly looks for an authority called "ROLE_ADMIN" — Spring auto-prefixes "ROLE_". hasAuthority("ADMIN") looks for the literal authority "ADMIN". NOOL stores roles without the prefix (the JWT claim is just "role": "ADMIN"), so we use hasAuthority consistently to avoid confusion.
Why is CSRF disabled?
CSRF (Cross-Site Request Forgery) protection guards against malicious websites tricking a logged-in user's browser into submitting requests using their session cookies. NOOL uses JWT tokens in the Authorization header, NOT cookies — so the CSRF attack vector doesn't apply. Disabling it is correct here. For a session-based app, you'd KEEP CSRF on.
Why SessionCreationPolicy.STATELESS?
To tell Spring Security: "don't create HTTP sessions for me; don't try to remember users between requests." Each request must independently authenticate via its JWT. This is the foundation of NOOL's stateless architecture — multiple backend instances behind a load balancer all work the same way, no session affinity needed.
Could you replace this file with method-level annotations like @PreAuthorize?
You could — Spring Security supports both. @PreAuthorize("hasAuthority('ADMIN')") on each controller method achieves the same per-endpoint rule. The trade-off: URL rules are centralized in ONE file (easier to audit), but method annotations live next to the methods (easier to spot when reading a controller). NOOL chose URL rules for the audit benefit, plus they apply even if you forget the annotation.
What does the asterisk in /attendance/employee/*/summary mean?
A single * matches one path segment (no slashes). So /attendance/employee/5/summary matches but /attendance/employee/5/something/else doesn't. A ** would match any number of segments. NOOL uses single-star for path variables and double-star for "any subpath" patterns.
How would you add a new endpoint?
Three steps. (1) Add the controller method. (2) Add a rule in SecurityConfig specifying who can call it. (3) Test with each role. The catch-all .anyRequest().authenticated() protects you against forgetting step 2 — unmatched URLs require at least authentication, never anonymous.
Why declare a PasswordEncoder bean in this file?
Because security-related beans naturally live in the security config. Spring Security looks for a PasswordEncoder bean by type when wiring its AuthenticationProvider. Putting it here keeps all security policy in one file.
