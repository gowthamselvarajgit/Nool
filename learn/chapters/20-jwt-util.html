🧠 What Is a JWT, Really?

JWT stands for JSON Web Token. It's a long string with three parts separated by dots:

A real JWT — three parts joined by dotsJWT
eyJhbGciOiJIUzI1NiJ9.eyJ1c2VySWQiOjEsInJvbGUiOiJBRE1JTiIsImlhdCI6MTc0NzIw....YkE9Yz3jZ-XwQ7tR8KvLp9N2sM4...
└─── HEADER ─────────┘ └─── PAYLOAD ──────────────────────────────────┘ └─── SIGNATURE ──────────┘
PartWhat's in itWho can read it?
HEADERThe algorithm used (HS256 for NOOL)Anyone — it's just Base64
PAYLOADThe actual data: userId, role, employeeId, ownerId, expiryAnyone — Base64 again
SIGNATUREA cryptographic stamp created with the server's secretAnyone can read; only the secret-holder can produce a valid one
The magic of JWTs: the payload is readable but not forgeable. Anyone can decode and see the contents (no encryption — don't put passwords inside!). But if anyone tries to change the contents, the signature no longer matches, and verification fails. It's like a passport: anyone can read it, only the government can issue one.

auth/security/JwtUtil.java Java 
package com.nool.backend.auth.security;

import io.jsonwebtoken.Claims;                // the decoded payload object
import io.jsonwebtoken.Jwts;                  // factory for building/parsing JWTs
import io.jsonwebtoken.SignatureAlgorithm;    // HS256, HS384, HS512, RS256, etc.
import io.jsonwebtoken.security.Keys;         // safe key creation helpers
import org.springframework.beans.factory.annotation.Value;
import org.springframework.stereotype.Component;

import javax.crypto.SecretKey;
import java.nio.charset.StandardCharsets;
import java.util.Date;
import java.util.Map;

@Component                                   // Spring auto-creates a singleton of this class
public class JwtUtil {

    // ── INJECTED SETTINGS (from application.properties) ───────────
    @Value("${jwt.secret}")
    private String secret;                     // the master secret. Long random string.

    @Value("${jwt.expiration}")
    private long expiration;                   // 86_400_000 ms = 24 hours

    // ── HELPER: turn the secret String into a real crypto SecretKey ─
    private SecretKey getSigningKey() {
        return Keys.hmacShaKeyFor(secret.getBytes(StandardCharsets.UTF_8));
    }
    // ★ Keys.hmacShaKeyFor validates the key length (must be ≥ 256 bits for HS256).
    //   It also catches "weak secret" bugs at startup instead of mysterious
    //   runtime errors. UTF-8 byte encoding ensures consistent results across
    //   any operating system.

    // ── MINT A NEW JWT ────────────────────────────────────────────
    public String generateToken(Map<String, Object> claims, String subject) {

        return Jwts.builder()
                .setClaims(claims)                                // the payload data (userId, role, ...)
                .setSubject(subject)                              // the "sub" claim — usually the mobile
                .setIssuedAt(new Date())                      // "iat" claim — now
                .setExpiration(new Date(                      // "exp" claim — now + 24h
                    System.currentTimeMillis() + expiration))
                .signWith(getSigningKey(),
                          SignatureAlgorithm.HS256)              // HMAC-SHA-256 signature
                .compact();                                       // produce the final dotted string
    }

    // ── PARSE & VERIFY A JWT ──────────────────────────────────────
    public Claims extractAllClaims(String token) {
        return Jwts.parserBuilder()
                .setSigningKey(getSigningKey())   // SAME key used for signing
                .build()
                .parseClaimsJws(token)            // ★ throws if signature is invalid
                .getBody();                       // returns the Claims (payload)
    }
    // ★ The library does the signature check INSIDE parseClaimsJws.
    //   If anyone tampered with the payload, the recomputed signature
    //   won't match and an exception is thrown.

    // ── HAS THE TOKEN EXPIRED? ────────────────────────────────────
    public boolean isTokenExpired(String token) {
        return extractAllClaims(token)
                .getExpiration()
                .before(new Date());
    }
}

🔍 What's Actually Inside a NOOL JWT

A typical token decodes to this:

-- HEADER (Base64 decoded):
{
  "alg": "HS256",        -- algorithm
  "typ": "JWT"           -- token type
}

-- PAYLOAD (Base64 decoded) — for Rahul after he logs in:
{
  "userId":     2,                       -- his User.id
  "role":       "SAREE_OWNER",             -- his User.role
  "employeeId": null,                    -- not a worker
  "ownerId":    1,                       -- his SareeOwner.id
  "sub":        "9000000001",              -- mobile (subject)
  "iat":        1747203131,                 -- issued-at (Unix seconds)
  "exp":        1747289531                  -- expires (24h later)
}

-- SIGNATURE:
-- HMAC-SHA-256( base64(HEADER) + "." + base64(PAYLOAD),  jwt.secret )
-- → a fixed 32-byte hash, Base64-encoded.

You can paste any JWT into jwt.io and see exactly this structure. Try it during development — but never share a real production token publicly, because while the payload is readable, the token is still a valid credential.

✍️ How Signing Actually Protects the Payload

The magic is HMAC-SHA-256. Here's a simplified view of what signWith() does:

  1. Take the base64-encoded HEADER + "." + base64-encoded PAYLOAD.
  2. Run HMAC-SHA-256 over that string, using the secret as the key.
  3. Get a fixed 32-byte hash. Base64-encode it. That's the SIGNATURE.

When extractAllClaims() verifies a token:

  1. Split the token into HEADER, PAYLOAD, SIGNATURE.
  2. Recompute HMAC-SHA-256 over HEADER + "." + PAYLOAD using the secret.
  3. Compare the recomputed hash to the token's SIGNATURE.
  4. Match? Token is authentic. Mismatch? Tampered or forged — throw.
An attacker tries to escalate to ADMIN:
  1. Rahul (a SAREE_OWNER) base64-decodes his token, edits the payload: "role": "ADMIN".
  2. Re-encodes the modified payload, glues it back with the ORIGINAL signature.
  3. Sends a request to /api/employees (admin-only).
  4. Server recomputes the signature using the modified payload + the server's secret.
  5. Recomputed signature ≠ token's signature. Verification fails. 401 returned.
Without the secret, an attacker cannot produce a signature that matches modified content. That's the whole reason JWTs are secure.

🤔 Why HS256 (Symmetric) Instead of RS256 (Asymmetric)?

HS256 (NOOL's choice)RS256 (the asymmetric alternative)
TypeSymmetric — same key signs and verifiesAsymmetric — private key signs, public key verifies
KeysOne shared secretPublic/private key pair
SpeedFasterSlower
Best forSingle backend (NOOL)OAuth servers issuing tokens to many verifiers
Key distributionKeep secret on the server onlyDistribute public key freely

NOOL has one backend that both issues AND verifies tokens. HS256 is simpler, faster, and adequate. If we ever needed third-party services to verify NOOL-issued tokens without sharing the secret, we'd switch to RS256.

🎬 Real-World Scenarios

Scenario 1: Rahul logs in successfully
  1. Rahul types mobile + password. Frontend posts to /api/auth/login.
  2. AuthServiceImpl.login() verifies the password with BCrypt — passes.
  3. Service builds claims map: { userId: 2, role: "SAREE_OWNER", ownerId: 1, employeeId: null }.
  4. Calls jwtUtil.generateToken(claims, "9000000001").
  5. JwtUtil:
    • Sets iat = now (e.g., 1747203131).
    • Sets exp = now + 24 hours (1747289531).
    • Signs with HS256 + the configured secret.
    • Returns the dotted JWT string.
  6. Service returns to controller → controller returns to frontend.
  7. Frontend stores the token in localStorage as nool_token.
  8. For the next 24 hours, every API request carries Authorization: Bearer eyJhbGc....
Scenario 2: Token verification on a protected endpoint
  1. Rahul makes a request to /api/inventory/my-ledger.
  2. Browser includes the Authorization header automatically.
  3. Backend's JwtAuthenticationFilter (Chapter 21) strips the "Bearer " prefix.
  4. Calls jwtUtil.isTokenExpired(token) first — false (still 23 hours left).
  5. Calls jwtUtil.extractAllClaims(token) — returns the payload object.
  6. Filter extracts role = "SAREE_OWNER" and sets it in Spring's security context.
  7. The request continues. SecurityConfig rule for that endpoint requires ADMIN or SAREE_OWNER — passes.
  8. Controller runs, fetches Rahul's ledger, returns JSON.
Scenario 3: Token expires
  1. Rahul logged in Monday at 10 AM. Expiry = Tuesday 10 AM.
  2. Tuesday 11 AM, he clicks "View Inventory".
  3. JwtUtil's isTokenExpired compares exp claim to "now" → true.
  4. Filter rejects with 401.
  5. Frontend's handleResponse in api.js sees 401, wipes localStorage, redirects to /login.
  6. Rahul logs in again, gets a fresh 24-hour token, continues.
Scenario 4: Trying to verify a tampered token
  1. Attacker copies Rahul's token, changes "role" from "SAREE_OWNER" to "ADMIN" in the payload.
  2. Re-Base64-encodes the modified payload but keeps the OLD signature (because they don't have the secret).
  3. Sends a request to POST /api/employees (admin-only).
  4. jwtUtil.extractAllClaims() is called.
  5. Internally, the library recomputes signature using modified payload + secret.
  6. Recomputed signature ≠ token's signature.
  7. Throws SignatureException. Filter falls through without setting any authentication. The request reaches the .anyRequest().authenticated() rule, fails, returns 401.
Common JwtUtil mistakes
Hard-coding the secret in this file
Keep it in application.properties and inject via @Value — exactly what NOOL does. In production, override via environment variable.
Using a short secret like "mysecret"
HMAC-SHA-256 requires the key to be at least 256 bits = 32 bytes = 32 ASCII characters. Keys.hmacShaKeyFor throws WeakKeyException if you try a shorter one. NOOL's secret is intentionally long.
Storing sensitive data inside the JWT payload (passwords, API keys, etc.)
The payload is base64-encoded, NOT encrypted. Anyone can decode and read it. Only put non-sensitive identifiers (userId, role, foreign-key IDs).
Setting expiration to "forever" or several months
If a token leaks, attackers can use it for the entire lifetime. 24 hours is a reasonable balance. For higher security, use short access tokens (15 min) + refresh tokens.
Catching JWT exceptions and silently allowing the request through
Any verification failure means "I don't know who this is" — never grant access. Let the exception propagate; the filter chain will return 401.
Forgetting to rotate the secret if it leaks
Rotating the secret instantly invalidates every issued token. Users have to log in again — but the leaked token becomes useless. Plan rotation: change the property, restart the backend.
Interview questions about JwtUtil
Why JWT and not server-side sessions?
Statelessness. The server doesn't have to keep a "logged-in users" table — the user carries proof of who they are. That means I can run multiple backend instances behind a load balancer without sharing session state, and restarts don't kick anyone out. Easier to scale, easier to debug.
What's the difference between Base64-encoding and encryption?
Encoding (Base64) just makes binary data ASCII-friendly. It's NOT secret — anyone can decode it instantly. Encryption transforms data such that only someone with the right key can read it. JWTs use Base64 for compactness and signing for integrity. The payload is readable; what matters is that it's not forgeable.
Why HS256 and not RS256?
HS256 is symmetric — same key signs and verifies. Simple and fast, perfect for a single backend like NOOL. RS256 is asymmetric (private signs, public verifies), useful when multiple services need to verify but only one issues — for example, an OAuth provider giving tokens to third parties. NOOL doesn't have that requirement.
If a user logs out, how do you invalidate their JWT?
In NOOL, we don't invalidate it server-side — the frontend just deletes the token from localStorage. The token remains technically valid on the server until its expiry, but with no one possessing it, that's effectively fine. If we needed real server-side revocation (e.g., "force logout this stolen token NOW"), we'd maintain a blacklist in Redis and check it in the filter — but that adds back the stateful complexity JWTs are designed to avoid.
What happens if the secret changes after users log in?
Every existing token becomes instantly invalid — signature verification needs the same secret used to sign. Users get 401 on their next request and have to log in again. This is actually a useful emergency lever — if you suspect a secret leak, rotate it.
What's the role of Keys.hmacShaKeyFor?
It validates the key length and converts the byte array into a SecretKey that the JWT library can use. The crucial bit: it refuses keys shorter than 256 bits for HS256, catching weak-secret bugs early (at startup) instead of after deployment.
Can a JWT be re-used multiple times?
Yes — that's the whole point. A single login produces one token, which is then sent on every request until it expires. JWTs are inherently bearer tokens: whoever holds them can use them. That's why HTTPS in production is non-negotiable — anyone who intercepts a token can impersonate the user.
