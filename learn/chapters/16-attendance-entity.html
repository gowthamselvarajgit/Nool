🧠 What This Entity Tracks

Every morning my mom opens NOOL and ticks who's at the workshop today. Each tick (or cross) becomes one row in the attendance table. That's it. Simple in concept — but the design has one important guarantee: you cannot mark the same worker present (or absent) twice on the same date.

The DB enforces this, not just the code. A UNIQUE constraint on (employee_id, attendance_date) means MySQL itself refuses to insert a duplicate. Even if a bug in the service layer tries, the database is the last line of defense. We saw the friendly error message back in Chapter 09 — "Attendance is already marked for this employee on this date."

📜 The Full File — Every Line Explained

entity/employee/Attendance.java Java 
package com.nool.backend.entity.employee;

import com.nool.backend.enums.AttendanceStatus;     // PRESENT or ABSENT
import jakarta.persistence.*;
import lombok.*;
import java.time.LocalDate;
import java.time.LocalDateTime;

@Entity
@Table(
    name = "attendance",
    uniqueConstraints = {                                      // ★ THE key design point
        @UniqueConstraint(columnNames = {"employee_id", "attendance_date"})
    }
)
@Getter @Setter
@NoArgsConstructor
@AllArgsConstructor
@Builder
public class Attendance {

    // ── PRIMARY KEY ───────────────────────────────────────────────
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    @Column(name = "attendance_id")
    private Long id;

    // ── THE DATE (day-precision, no time) ─────────────────────────
    @Column(name = "attendance_date", nullable = false)
    private LocalDate attendanceDate;
    // LocalDate (not LocalDateTime) — only the day matters.

    // ── PRESENT / ABSENT ──────────────────────────────────────────
    @Enumerated(EnumType.STRING)
    @Column(name = "attendance_status", nullable = false)
    private AttendanceStatus attendanceStatus;
    // Stored as text — "PRESENT" or "ABSENT".

    // ── REMARKS (optional note) ───────────────────────────────────
    @Column(name = "remarks")
    private String remarks;
    // "Half day for doctor appointment", "Festival leave", etc.

    // ── EMPLOYEE LINK ─────────────────────────────────────────────
    @ManyToOne
    @JoinColumn(name = "employee_id", nullable = false)
    private Employee employee;
    // Many attendance rows belong to one employee — across days.

    // ── AUDIT TIMESTAMPS ──────────────────────────────────────────
    @Column(name = "created_at", nullable = false, updatable = false)
    private LocalDateTime createdAt;

    @Column(name = "updated_at")
    private LocalDateTime updatedAt;

    // ── LIFECYCLE HOOKS ───────────────────────────────────────────
    @PrePersist
    protected void onCreate(){
        LocalDateTime now = LocalDateTime.now();
        this.createdAt = now;
        this.updatedAt = now;
    }

    @PreUpdate
    protected void onUpdate(){
        this.updatedAt = LocalDateTime.now();
    }
}

🗄️ What This Becomes in MySQL

CREATE TABLE attendance (
    attendance_id     BIGINT      NOT NULL AUTO_INCREMENT,
    employee_id       BIGINT      NOT NULL,
    attendance_date   DATE        NOT NULL,
    attendance_status VARCHAR(20) NOT NULL,        -- PRESENT or ABSENT
    remarks           VARCHAR(255),
    created_at        DATETIME(6) NOT NULL,
    updated_at        DATETIME(6),

    PRIMARY KEY (attendance_id),
    UNIQUE KEY uk_attendance_emp_date (employee_id, attendance_date), -- ★ THE GUARD
    FOREIGN KEY (employee_id) REFERENCES employees(employee_id)
);

-- Mom marks Prema present today:
INSERT INTO attendance
    (employee_id, attendance_date, attendance_status, remarks, created_at, updated_at)
VALUES
    (5, '2026-05-15', 'PRESENT', NULL,
     '2026-05-15 09:15:00', '2026-05-15 09:15:00');

-- Mom tries to mark Prema present AGAIN for the same date (by mistake):
INSERT INTO attendance
    (employee_id, attendance_date, attendance_status, ...)
VALUES (5, '2026-05-15', 'PRESENT', ...);
-- ERROR 1062: Duplicate entry '5-2026-05-15' for key 'uk_attendance_emp_date'
-- → GlobalExceptionHandler turns this into a friendly 409 message.

-- Mom changes Prema's status from PRESENT to ABSENT (she went home sick):
UPDATE attendance
   SET attendance_status = 'ABSENT',
       remarks           = 'Went home sick at 11 AM',
       updated_at        = '2026-05-15 11:30:00'      -- auto by @PreUpdate
 WHERE attendance_id = 142;

🔐 The UNIQUE Constraint — Why It's So Important

The line that makes this entity safe is this one in the @Table annotation:

The critical declarationJava
@Table(
    name = "attendance",
    uniqueConstraints = {
        @UniqueConstraint(columnNames = {"employee_id", "attendance_date"})
    }
)

It says: "the combination of employee_id + attendance_date must be unique across the entire table."

Without it, what could go wrong?

Without the constraintWith the constraint
Two rows for Prema on May 15 (one PRESENT, one ABSENT) — which is correct? Cannot happen. Second INSERT fails immediately.
Salary calculation double-counts working days. Salary calculation is correct by construction.
Attendance calendar shows confusing duplicates. Clean one row per day per worker.
The bug only shows up much later when totals don't match. The error message at insertion time tells the user exactly what's wrong.
Defense in depth: Service code should check before insert ("is there already a row for this employee + date?"), but bugs happen. The DB-level constraint is the safety net that catches what the application code might miss. Two layers of protection — exactly what you want for important data.

⚙️ A Subtle Detail — The @ManyToOne Fetch Strategy

Notice the employee link doesn't have fetch = FetchType.LAZY:

Default fetch behaviorJava
@ManyToOne                                // default fetch = EAGER for ManyToOne
@JoinColumn(name = "employee_id", nullable = false)
private Employee employee;

In JPA, @ManyToOne defaults to FetchType.EAGER — meaning every time we load an Attendance row, Hibernate also JOINs in the full Employee row.

Is this a bug? Not necessarily. Attendance rows are almost always displayed alongside the employee name (calendar views, lists). EAGER loading saves separate queries. But if a hot path ever loads many Attendance rows just to count them, adding fetch = FetchType.LAZY would improve performance. NOOL accepts the EAGER default for simplicity at the current scale.

🎬 Real-World Scenarios

Scenario 1: Mom marks the morning roll call
  1. 9:15 AM. Mom logs in, opens Admin → Attendance → today.
  2. The page lists all active workers with a PRESENT/ABSENT toggle.
  3. She ticks PRESENT for Prema and Ravi, ABSENT for Suresh (he called in sick).
  4. She clicks Save.
  5. Frontend sends three POST requests to /api/attendance.
  6. For each one:
    • Service checks: does a row already exist for (employee, today)? No.
    • Inserts a new Attendance row.
    • @PrePersist stamps timestamps.
  7. Three new rows in the attendance table for 2026-05-15.
Scenario 2: Mom accidentally tries to mark Prema again
  1. A few minutes later, Mom returns to the attendance page (browser tab was still open).
  2. She doesn't realize she already saved, and clicks Save again for Prema.
  3. Backend attempts another INSERT for (employee_id=5, attendance_date=2026-05-15).
  4. UNIQUE constraint rejects: ERROR 1062: Duplicate entry '5-2026-05-15'.
  5. Hibernate throws DataIntegrityViolationException.
  6. GlobalExceptionHandler.handleDataIntegrityViolation sees "duplicate entry" + "attendance_date" in the cause.
  7. Returns HTTP 409 with message: "Attendance is already marked for this employee on this date."
  8. Frontend shows the toast. Mom realizes her mistake — no duplicate row created.
Scenario 3: Prema goes home sick at 11 AM
  1. Prema came in at 9 AM (marked PRESENT) but feels unwell at 11 AM and leaves.
  2. Mom opens attendance, finds Prema's row, changes status to ABSENT, adds remark "Went home sick at 11 AM".
  3. Backend issues UPDATE attendance SET attendance_status='ABSENT', remarks='...', updated_at=NOW() WHERE attendance_id=142.
  4. @PreUpdate sets updatedAt. Created_at stays unchanged.
  5. The audit log now shows: created at 09:15 with PRESENT, updated at 11:30 with ABSENT.
Common mistakes around Attendance.java
Skipping the UNIQUE constraint because "the service code already checks"
Service code can have bugs. Constraints in the database are the last line of defense. Both layers should exist — defense in depth.
Using LocalDateTime for attendance_date
Attendance is a day-precision concept. LocalDate matches that exactly and avoids weird "9:15 AM" vs "10:32 AM" comparisons that break the UNIQUE constraint.
Hard-deleting an attendance row to "fix a mistake"
UPDATE the status instead. Deleting destroys the audit trail. If Prema's PRESENT was wrong, change it to ABSENT — don't erase the row.
Adding extra status values (HALF_DAY, LEAVE, HOLIDAY) without updating the enum
If you need new statuses, add them to AttendanceStatus in enums/. The DB column is wide enough (VARCHAR 20), so no migration needed. The frontend's friendlyStatus() helper already has display labels for several extra values, ready for when you add them.
Storing the employee's NAME on each attendance row (denormalization for "speed")
Use the FK to employees and JOIN. Denormalization makes name changes a nightmare ("Prema gets married and updates her name — now we have to UPDATE all 200 attendance rows?"). Normalize. Trust the JOIN.
Interview questions about Attendance.java
Why a UNIQUE constraint on (employee_id, attendance_date)?
To make it physically impossible for a worker to be marked twice on the same day. The application code does check before inserting, but constraints in the database are the last line of defense — they catch bugs the service might miss. Defense in depth. The friendly error message you saw in Chapter 09 comes from translating the resulting DataIntegrityViolationException.
Why LocalDate not LocalDateTime?
Because attendance is a day-precision concept. We care that Prema was present on May 15 — not at what hour. Using LocalDateTime would mean two rows at 9:15 AM and 10:00 AM would both pass the UNIQUE constraint (different timestamps), which is wrong. LocalDate guarantees comparisons are by day, exactly what we want.
Why is the @ManyToOne not LAZY-fetched here?
JPA's default for @ManyToOne is EAGER. We accept this default because attendance rows are almost always rendered alongside the employee's name, so eager loading saves separate round-trips. For pure-count queries (e.g., "how many present days in May?"), LAZY would be faster, but the simpler default code is fine at our scale.
If the AttendanceStatus enum only has PRESENT and ABSENT, what about half-days, leaves, holidays?
The enum is deliberately minimal because my parents' workflow is binary. The frontend's friendlyStatus() helper has display labels for HALF_DAY, LEAVE, HOLIDAY, WEEKEND etc., so when the business decides to track them, adding new enum values is a one-line change. The DB column is already wide enough.
How does the calendar view in the worker dashboard query this table?
Typically: SELECT attendance_date, attendance_status FROM attendance WHERE employee_id = ? AND attendance_date BETWEEN start_of_month AND end_of_month. The frontend renders each date as a green dot (PRESENT) or red dot (ABSENT). Days with no row are rendered as neutral (not yet marked).
Can attendance for a future date be marked?
The entity itself doesn't prevent it — LocalDate happily accepts any date. Whether future dates are allowed is a business-rule decision made in the service layer. NOOL currently doesn't block future dates because there's no harm (you'd just be guessing). If you wanted to enforce "no future attendance", you'd add a check in AttendanceServiceImpl.
