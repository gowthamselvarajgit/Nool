pom.xml —
The Maven shopping list

Every Java project starts with one file that says "here's what libraries I need". In Maven projects, that file is called pom.xml. Before we can compile a single line of NOOL's backend, we have to understand what's in this file and why.

🤔 What is pom.xml?

"POM" stands for Project Object Model. It's an XML file that tells the Maven build tool three things:

  1. Who I am — project name, version, Java version
  2. What I need — list of external libraries (called "dependencies")
  3. How to build me — compile settings, plugins, packaging instructions
Think of it like a recipe card: if you wanted to cook a dish, you'd list the ingredients (dependencies), the kitchen tools (plugins), and the brand of flour (versions). pom.xml is that recipe card for the Java compiler.

⚡ What Happens When You Run mvn?

When my mom (just kidding, when I) type ./mvnw spring-boot:run in the backend folder:

  1. Maven reads pom.xml
  2. It looks at every <dependency> entry
  3. For each one, it downloads the JAR file from a central server (Maven Central) if you don't have it locally
  4. It puts all those JARs on the Java classpath
  5. It compiles your Java code together with those libraries
  6. It runs the Spring Boot app

The first time, this can take a few minutes (downloading hundreds of MB). Every time after that, it's instant because the JARs are cached in ~/.m2/repository/.

                        <phase>test-compile</phase>

              works — at compile time it scans for @Getter, @Setter etc.

              bytecode. The result: your source file stays clean, but the
              compiled class has all the getter/setter methods.
            -->

        </plugins>

🎯 What Do "Scopes" Mean?

A dependency's <scope> controls when the JAR is on the classpath. Maven uses four common scopes:

ScopeAvailable at compile?Available at runtime?Bundled in final JAR?
compile (default)YesYesYes
runtimeNoYesYes
testOnly in test sourcesOnly in testsNo
providedYesNo (assumed already there)No
Why mysql-connector is runtime: our code never imports com.mysql.cj.jdbc.Driver directly — Hibernate finds it dynamically by class name. So we don't need it on the compile classpath, only when the app actually runs. Slightly faster compiles, slightly cleaner API.

🎬 Real Scenario — What Happens at Build Time

First time someone clones NOOL and runs the backend
  1. I clone the repo to a new computer.
  2. I cd into backend/ and run ./mvnw spring-boot:run.
  3. Maven reads pom.xml. Sees parent = spring-boot-starter-parent:4.0.6.
  4. Maven downloads the parent POM from Maven Central (~2 KB).
  5. Parent POM lists ~50 inherited library versions.
  6. Maven then resolves each of our 10 dependencies, plus all their transitive dependencies (deps-of-deps). Total ends up ~250 JARs.
  7. Maven downloads any JAR not already in ~/.m2/repository/. First time: ~300 MB, ~3 minutes.
  8. Maven invokes the Java compiler with Lombok as an annotation processor.
  9. Compiler walks every .java file, generates Lombok methods, produces .class files into target/classes/.
  10. Spring Boot Maven plugin launches the app — embedded Tomcat starts on port 8083.
  11. Console shows the Spring banner, "Started BackendApplication in 4.2 seconds".
Second time (and every time after)
  1. All JARs are already cached locally.
  2. Only .java files that changed since last build get recompiled.
  3. App is up in 4-5 seconds total. Maven dependency phase is < 1 second.
Common pom.xml mistakes to avoid
Specifying versions for libraries that Spring Boot's parent already manages
If the parent POM defines a version, leave the <version> tag off your dependency entry. Otherwise you might pin an old version that's incompatible with the rest of the Spring ecosystem.
Forgetting <optional>true</optional> on Lombok
Without it, any project that depends on YOUR project would also have to add Lombok as a dependency — pointless friction. optional=true says "I use this internally; downstream doesn't need it."
Adding a dependency without checking if it's already transitively pulled in
Run ./mvnw dependency:tree to see what's already on the classpath. Adding duplicates causes version conflicts.
Putting test libraries in compile scope
Use <scope>test</scope>. Otherwise JUnit, Mockito etc. end up in your production JAR — wasted ~5MB and a slightly bigger attack surface.
Removing the Lombok annotationProcessorPaths block to "clean up"
If you remove it, Lombok stops working at compile time. Your code will reference auto-generated getter methods that don't exist, and the build will fail mysteriously. Keep both the dependency AND the processor path config.
Interview-style questions about pom.xml
Why does NOOL use the spring-boot-starter-parent as its parent POM?
The parent POM provides "managed dependencies" — it knows the right, mutually-compatible versions of ~50 common libraries (Hibernate, Jackson, Tomcat, etc.). I just declare which "starters" I need; Spring Boot handles the version harmony. Without it, I'd spend days manually picking versions and fixing conflicts.
What's the difference between spring-boot-starter-web and spring-boot-starter-webmvc?
They're essentially the same in current Spring Boot versions — spring-boot-starter-webmvc is the modern, more specific name (introduced in Spring Boot 4) that makes it clear we're using the servlet-based MVC stack, not the reactive WebFlux stack. Older code uses spring-boot-starter-web; both work.
Why are JWT versions pinned to 0.11.5 but other library versions aren't?
Because Spring Boot's parent POM doesn't manage JWT. I have to pick the version myself. 0.11.5 is a known-stable jjwt release. For the libraries Spring Boot DOES manage (JPA, Security, etc.), I omit the version and inherit the recommended one.
What is an "annotation processor" and why does Lombok need one?
Annotation processors are Java compiler plugins that read special annotations (@Getter, @Builder, etc.) and generate additional code before the regular compile step finishes. Lombok works this way: I write @Getter private String name;, the processor injects a public String getName() { return this.name; } method into the .class file. The original .java source stays clean.
What's a "fat JAR"?
A fat JAR (or "uber JAR") is a single .jar file containing your compiled code AND all its dependencies' .jar contents bundled in. Spring Boot's Maven plugin builds one when you run ./mvnw package. You can then ship just that one file to production and run it with java -jar backend-0.0.1-SNAPSHOT.jar — no need for a separate dependency-management setup on the server.
What does SNAPSHOT in the version mean?
It's Maven's convention for "in development, not a stable release". A SNAPSHOT version can change at any time without bumping the number — useful while developing. When you release, you drop the SNAPSHOT suffix (0.0.1-SNAPSHOT → 1.0.0) and that version becomes immutable in repositories.
