🧠 The Problem It Solves

After JwtAuthenticationFilter (Chapter 21) sets the security context with Claims, business code needs an easy way to ask "who's calling this method?" Without a helper, every service method would have to write something like:

Without the helper — verbose and error-proneJava
Object principal = SecurityContextHolder
        .getContext()
        .getAuthentication()
        .getPrincipal();
Claims claims = (Claims) principal;
Long ownerId = claims.get("ownerId", Long.class);

// Repeat this ~7-line pattern in 50 different service methods. Easy to typo
// "owner_id" or "ownerID", easy to forget the null check, easy to confuse
// principal with the user object.

With CurrentUserUtil, the same lookup becomes:

With the helper — one lineJava
Long ownerId = CurrentUserUtil.getOwnerId();
The whole point of this file: one place that knows how to read the JWT's Claims out of the security context. Every other class just calls these static methods. No copy-paste, no typos, no chance of forgetting to cast Object to Claims.

📜 The Full File — Every Line Explained

auth/security/CurrentUserUtil.java Java 
package com.nool.backend.auth.security;

import io.jsonwebtoken.Claims;
import org.springframework.security.core.context.SecurityContextHolder;

public class CurrentUserUtil {              // no @Component — purely static utility

    // ── PRIVATE HELPER — get the Claims out of the security context ─
    private static Claims getClaims(){
        Object principal = SecurityContextHolder
                .getContext()
                .getAuthentication()
                .getPrincipal();

        if (principal instanceof Claims claims) {   // Java 16+ pattern matching
            return claims;
        }
        throw new RuntimeException("Invalid security context");
    }

    // ── PUBLIC GETTERS — one per Claim we care about ─────────────

    public static Long getUserId(){
        return getClaims().get("userId", Long.class);
    }

    public static String getRole(){
        return getClaims().get("role", String.class);
    }

    public static Long getEmployeeId() {
        return getClaims().get("employeeId", Long.class);
    }

    public static Long getOwnerId() {
        return getClaims().get("ownerId", Long.class);
    }
}

⚡ Why static and Not a Spring Bean?

Notice this class is not annotated with @Component — it's purely static. Why?

Static utility (NOOL's choice)Spring bean alternative
Call from anywhere: CurrentUserUtil.getOwnerId()Must inject: @Autowired CurrentUserService cus;
No constructor changes needed in callersEvery service needs an extra constructor argument
Reads from ThreadLocal — already per-requestSame, but with injection overhead
Slightly harder to mock in unit testsEasier to mock (just inject a fake)

The trade-off: static methods are slightly harder to mock in unit tests. For NOOL, we accept that because (a) we test mostly with integration tests that have real security contexts, and (b) the call-site simplicity is worth a lot when used in dozens of services.

🆕 Java 16+ Pattern Matching — That instanceof Block

One line in the file deserves a closer look — it uses a modern Java feature:

Pattern matching for instanceof (Java 16+)Java
if (principal instanceof Claims claims) {
    return claims;
}

The old, pre-Java-16 way would be:

Pre-Java-16 — verboseJava
if (principal instanceof Claims) {
    Claims claims = (Claims) principal;   // explicit cast — redundant
    return claims;
}

The modern syntax combines the type check + cast + variable declaration into one line. It's safer (the variable is only in scope inside the if) and cleaner. Java 21 (NOOL's compile target) supports this fully.

💥 Why the RuntimeException?

If the principal somehow isn't a Claims object, the helper throws:

The safety throwJava
throw new RuntimeException("Invalid security context");

When could this happen? In NOOL, only if:

  • Someone calls a getter from outside an HTTP request thread (e.g., a startup runner before a request arrives).
  • A different authentication mechanism populated the principal differently (won't happen in NOOL).
  • The security context is empty (no authenticated user).

The exception is intentionally generic — it bubbles up to the GlobalExceptionHandler (Chapter 09), which logs it and returns 500. That's acceptable because reaching this exception path indicates a bug, not a normal flow.

A stricter version would throw a custom SecurityException and map it to 401 in the handler. For NOOL's current scale, the generic RuntimeException is enough — the situation is rare and unambiguously a bug.

public PaginationResponseDto<LedgerEntryResponseDto> getMyLedger(...) {

From AttendanceServiceImpl — checking permissionsJava
private void assertSelfOrAdmin(Long employeeId) {
    String role = CurrentUserUtil.getRole();
    Long  myId  = CurrentUserUtil.getEmployeeId();

    if ("ADMIN".equals(role)) return;          // admin sees anyone
    if (employeeId.equals(myId)) return;        // worker sees self

    throw new ResourceNotFoundException("Not allowed");
}

This pattern repeats across many services. Each "my data" endpoint and each cross-user access check goes through this helper.

🔗 The Full Identity Chain

Here's how a user's identity flows from login all the way to a database query:

From login to queryflow
1. Login
   POST /api/auth/login {mobile, password}
                ↓
2. AuthServiceImpl
   Verifies password.
   Builds claims map {userId, role, employeeId, ownerId}.
   JwtUtil.generateToken(claims) → JWT string.
                ↓
3. Frontend stores token in localStorage.

══════════ Next request ══════════

4. POST /api/inventory/my-ledger
   Header: "Authorization: Bearer eyJhbGc..."
                ↓
5. JwtAuthenticationFilter
   Verifies signature + expiry.
   Decodes Claims.
   SecurityContextHolder.setAuthentication(claims, ...).
                ↓
6. SecurityConfig rules
   "/inventory/my-ledger" → hasAnyAuthority(ADMIN, SAREE_OWNER).
   role is SAREE_OWNER → ALLOW.
                ↓
7. SareeInventoryController.getMyLedger()
                ↓
8. SareeInventoryServiceImpl.getMyLedger()
                ↓
9. ★ CurrentUserUtil.getOwnerId()
   Reads SecurityContextHolder → principal → Claims → "ownerId" → 1
                ↓
10. Query database scoped to ownerId = 1
    "SELECT * FROM saree_ledger WHERE owner_id = 1 ..."

The entire chain works because every step trusts the previous one. The JWT is verified once at the filter; downstream code can trust the Claims completely.

🎬 Real-World Scenarios

Scenario 1: Rahul requests his own inventory
  1. Frontend sends POST /api/inventory/my-ledger with Rahul's JWT.
  2. Filter sets Claims with ownerId=1 in the security context.
  3. Controller delegates to service.
  4. Service calls CurrentUserUtil.getOwnerId() → returns 1.
  5. Service queries: SELECT * FROM saree_ledger WHERE owner_id = 1 ....
  6. Rahul sees only HIS data — never anyone else's.
Scenario 2: Prema checks her own salary history
  1. Frontend sends POST /api/salary-payments/history with Prema's JWT.
  2. Filter sets Claims with employeeId=5, role=WORKER.
  3. Service calls CurrentUserUtil.getEmployeeId() → 5.
  4. Service queries: SELECT * FROM salary_payments WHERE employee_id = 5 ....
  5. Prema sees only her own salary history.
Scenario 3: Admin views any employee's salary history
  1. Dad (admin) sends POST /api/salary-payments/employee/5/history with the employee ID in the URL.
  2. Filter sets Claims with role=ADMIN, employeeId=null.
  3. Service: assertSelfOrAdmin(5). Role is ADMIN → return without check.
  4. Service queries for employee_id = 5 (from the URL path variable).
  5. Admin sees Prema's history even though they're not Prema.
Scenario 4: Ravi (worker) tries to view Prema's salary (should fail)
  1. Ravi crafts a request: POST /api/salary-payments/employee/5/history (Prema's ID).
  2. SecurityConfig allows the URL pattern for ADMIN or WORKER — passes.
  3. Service: assertSelfOrAdmin(5).
    • Role = WORKER. Not admin → continue checking.
    • CurrentUserUtil.getEmployeeId() → 7 (Ravi's ID).
    • 7.equals(5)? No → throw ResourceNotFoundException.
  4. Handler returns 404. Cross-worker access prevented at the service layer.
Common CurrentUserUtil mistakes
Calling CurrentUserUtil.getOwnerId() from a non-request context (e.g., a startup hook)
There's no security context outside of HTTP requests. Calling these methods elsewhere throws "Invalid security context". Pass user IDs explicitly to background jobs.
Trusting getEmployeeId() to never return null
Admins have employeeId = null. Always check for null before using it. Or use it only in WORKER-only endpoints.
Reading raw SecurityContextHolder in services directly
Always go through CurrentUserUtil. The utility is the single source of truth — if the JWT format changes, you only edit one file.
Comparing role strings inconsistently: role.equals("admin") vs "ADMIN".equals(role)
JWT stores roles as uppercase enum names ("ADMIN", "WORKER", "SAREE_OWNER"). Always compare against uppercase literals. Use "ADMIN".equals(role) to avoid null pointer exceptions.
Storing the current-user info in a regular instance field of a service
Spring services are singletons — sharing a field would cause concurrent requests to overwrite each other. Always fetch from the security context per-call (which is per-request via ThreadLocal).
Interview questions about CurrentUserUtil
Why is this class not a Spring bean?
It's purely static — no state, no dependencies, just reads from SecurityContextHolder which is itself a static facade backed by ThreadLocal. Making it a bean would mean every service that uses it has to inject one more dependency, adding noise without any benefit. Static methods are simpler for this case.
How does SecurityContextHolder know which user is "current"?
By default, it uses a ThreadLocal. The web servlet container (Tomcat in NOOL's case) handles each HTTP request on a dedicated thread, so the ThreadLocal naturally scopes to that request. The JwtAuthenticationFilter writes to it; everything downstream on the same thread reads from it.
What's the cost of calling getOwnerId() dozens of times in one request?
Effectively zero. Each call is: ThreadLocal lookup, get authentication, get principal, cast to Claims, Map.get. All in-memory operations totaling microseconds. No DB hit, no network. We can call these helpers freely without performance worry.
Would caching the result inside a request help performance?
Not meaningfully. The lookup is already fast. Caching would add complexity and the risk of stale data. For NOOL's scale, the straightforward repeated calls are fine.
How would you unit-test code that uses CurrentUserUtil?
Two common approaches. (1) For integration tests, use Spring Security's testing support: @WithMockUser or manually setting up the security context before each test. (2) For pure unit tests, refactor the service to accept the user ID as a parameter (one method overload takes the ID, another reads from CurrentUserUtil internally). The latter approach is what Spring patterns recommend.
What if NOOL ever supports impersonation ("admin logs in as a specific user to debug")?
The cleanest way: add an "impersonatedOwnerId" claim to the JWT during impersonation, and add corresponding getters to this util. Spring Security also has a built-in SwitchUserFilter that handles the role-swap automatically. Either way, this utility would be the natural place to expose the impersonated identity to services.
Phase 5 wrap-up — how do the four security files work together?
(1) JwtUtil mints and verifies JWTs. (2) JwtAuthenticationFilter intercepts every request and uses JwtUtil to put the user's Claims into the SecurityContext. (3) SecurityConfig declares URL-to-role rules that check those Claims before letting the request reach a controller. (4) CurrentUserUtil exposes the Claims to business code in a clean, one-line API. Together they form NOOL's complete stateless auth system.

🎉 Phase 5 (Security) Complete

You've now seen NOOL's full security architecture — JWT creation, request-level verification, URL-level rules, and how business code reads the caller's identity. Together they enforce that only Mom and Dad can add workers, only owners see their own data, and tokens cannot be forged. Defense in depth: every layer protects what the previous layer might miss.
