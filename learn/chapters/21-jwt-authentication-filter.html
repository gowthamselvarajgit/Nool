🧠 What's a Servlet Filter?

A filter is a piece of code that runs before every HTTP request reaches its controller. Spring Boot uses filters to handle cross-cutting concerns: logging, compression, security, CORS, etc. Filters form a chain — each filter does its job, then calls the next one.

The filter chain (simplified)flow
HTTP request arrives
       │
       ▼
[ CORS Filter ]              ← handles Cross-Origin Resource Sharing
       │
       ▼
[ JwtAuthenticationFilter ]  ★ ← THIS FILE — extracts the JWT
       │
       ▼
[ Spring Security Filters ]  ← checks role-based rules
       │
       ▼
[ DispatcherServlet ]        ← routes to the right controller
       │
       ▼
[ @RestController ]          ← business code finally runs
Why have this as a filter and not inside the controller? Because every single endpoint needs auth — putting the check in each controller would mean copy-pasting code 50 times. A filter runs once per request, centrally, before any controller is reached. One place to maintain, one place to debug.

📜 The Full File — Every Line Explained

auth/security/JwtAuthenticationFilter.java Java 
package com.nool.backend.auth.security;

import io.jsonwebtoken.Claims;
import jakarta.servlet.FilterChain;
import jakarta.servlet.ServletException;
import jakarta.servlet.http.HttpServletRequest;
import jakarta.servlet.http.HttpServletResponse;
import org.springframework.security.authentication.UsernamePasswordAuthenticationToken;
import org.springframework.security.core.authority.SimpleGrantedAuthority;
import org.springframework.security.core.context.SecurityContextHolder;
import org.springframework.stereotype.Component;
import org.springframework.web.filter.OncePerRequestFilter;

import java.io.IOException;
import java.util.List;

@Component                                          // Spring picks this up automatically
public class JwtAuthenticationFilter
        extends OncePerRequestFilter {              // ★ guarantees this runs exactly once per request

    private final JwtUtil jwtUtil;

    public JwtAuthenticationFilter(JwtUtil jwtUtil) {
        this.jwtUtil = jwtUtil;                       // constructor injection of JwtUtil
    }

    @Override
    protected void doFilterInternal(
            HttpServletRequest  request,
            HttpServletResponse response,
            FilterChain         filterChain
    ) throws ServletException, IOException {

        // ── STEP 1: Look for the Authorization header ────────────────
        String authHeader = request.getHeader("Authorization");
        // Standard HTTP header: "Authorization: Bearer eyJhbGc..."

        // ── STEP 2: If header exists and starts with "Bearer ", extract token ─
        if (authHeader != null && authHeader.startsWith("Bearer ")) {

            String token = authHeader.substring(7);     // strip "Bearer " (7 chars)

            // ── STEP 3: Check expiry & signature ─────────────────────
            if (!jwtUtil.isTokenExpired(token)) {     // also implicitly checks signature

                // ── STEP 4: Decode the payload ───────────────────────
                Claims claims = jwtUtil.extractAllClaims(token);
                String role   = claims.get("role", String.class);

                // ── STEP 5: Tell Spring Security who is calling ──────
                UsernamePasswordAuthenticationToken authentication =
                        new UsernamePasswordAuthenticationToken(
                                claims,                              // principal = the JWT claims
                                null,                                // credentials (we don't need them)
                                List.of(new SimpleGrantedAuthority(role))   // authorities = [role]
                        );

                SecurityContextHolder.getContext()
                        .setAuthentication(authentication);
                // ★ From here on, any code in this request can call
                //   SecurityContextHolder.getContext().getAuthentication()
                //   and find out "the user calling is role X".
            }
            // If isTokenExpired threw or returned true, we DON'T set authentication.
            // The request continues but as an "anonymous" user — Spring Security's
            // role checks will then reject it.
        }

        // ── STEP 6: Pass to the next filter (always) ─────────────────
        filterChain.doFilter(request, response);
        // Whether or not we authenticated, the chain continues.
        // Downstream filters will reject unauthenticated requests at the
        // SecurityConfig rules.
    }

    // ── SKIP PUBLIC URLs ──────────────────────────────────────────
    @Override
    protected boolean shouldNotFilter(HttpServletRequest request) {
        return request.getRequestURI().startsWith("/api/auth/");
    }
    // ★ Don't run this filter for /api/auth/* endpoints (login, register).
    //   Those endpoints exist precisely to GET a token, so requiring a
    //   token to call them would be a chicken-and-egg problem.
}

📋 The Two Methods Explained

MethodPurposeWhen called
doFilterInternal(req, res, chain) The core logic — extract token, verify, populate security context, continue the chain Once per HTTP request
shouldNotFilter(req) Decide whether to skip this filter for this request Spring asks before running doFilterInternal

🔄 Why OncePerRequestFilter?

Spring's OncePerRequestFilter base class guarantees the filter runs exactly once per request, even if the request goes through internal forwards or includes (e.g., when a controller does request.getRequestDispatcher(...)). Without it, the same filter could run multiple times per request, which would either waste CPU re-verifying the same token or — worse — overwrite the security context with stale data.

🚪 Why /api/auth/* Endpoints Are Skipped

The login endpoint (POST /api/auth/login) is the very mechanism by which users get a JWT in the first place. Requiring a JWT to call it would be a chicken-and-egg problem: "to log in, you need to already be logged in".

The shouldNotFilter override returns true for any URL starting with /api/auth/. So login, logout, and token validation endpoints skip this filter entirely. SecurityConfig (Chapter 22) separately marks these paths as permitAll().

🧰 What Is the SecurityContext?

SecurityContextHolder.getContext() is Spring Security's per-request "who is calling?" object. It's typically backed by a ThreadLocal, so it's automatically scoped to the current request thread.

When the filter does:

Setting authenticationJava
SecurityContextHolder.getContext().setAuthentication(authentication);

...it's saying: "For the rest of this HTTP request, treat the caller as having these credentials and authorities." Downstream filters and controllers can read this info via:

Reading authentication anywhere downstreamJava
Object principal = SecurityContextHolder.getContext()
                       .getAuthentication()
                       .getPrincipal();
// principal is the Claims object we set in the filter

That's exactly what CurrentUserUtil (Chapter 23) does to expose the caller's userId, role, employeeId, and ownerId to service-layer code.

🎯 Why Pass claims as the Principal?

Notice the unusual choice — the filter passes the JWT's Claims object as the principal:

The principal IS the ClaimsJava
new UsernamePasswordAuthenticationToken(
    claims,                            // ★ the entire JWT payload
    null,
    List.of(new SimpleGrantedAuthority(role))
);

Normally the principal would be a String username or a UserDetails object. Here it's the full Claims map. Why?

If we used a username StringUsing Claims (NOOL's choice)
Need DB lookup on every request to get role + IDsEverything's already in the Claims — no DB hit
Controllers need to take Username param, then load Employee/Owner separatelyCurrentUserUtil.getEmployeeId() reads it directly from the JWT
Slower under loadPure in-memory after the initial parse
The whole point of JWT is to AVOID per-request DB lookups for identity. By passing Claims as the principal, every layer of NOOL can read userId/role/employeeId straight from the request thread — no JOINs, no caching, no complexity.

🎬 Real-World Scenarios

Scenario 1: Rahul requests his inventory
  1. Frontend sends POST /api/inventory/my-ledger with Authorization: Bearer eyJhbGc....
  2. Filter chain reaches JwtAuthenticationFilter.
  3. shouldNotFilter sees path starts with /api/inventory/ — does NOT skip.
  4. doFilterInternal reads the Authorization header.
  5. Strips "Bearer ", verifies expiry (not expired), extracts Claims.
  6. Role from Claims: "SAREE_OWNER".
  7. Sets SecurityContextHolder with authentication = Claims + authority "SAREE_OWNER".
  8. Calls filterChain.doFilter(...).
  9. Next filters run. SecurityConfig rule for that path requires ADMIN or SAREE_OWNER → passes.
  10. Controller runs. It calls CurrentUserUtil.getOwnerId() to get Rahul's owner ID, fetches his ledger, returns JSON.
Scenario 2: Login request (filter skipped)
  1. Frontend sends POST /api/auth/login with mobile + password.
  2. Filter chain reaches JwtAuthenticationFilter.
  3. shouldNotFilter sees path starts with /api/auth/ → returns true → skip.
  4. The filter is completely bypassed. The request continues directly to subsequent filters.
  5. SecurityConfig marks /api/auth/** as permitAll() → passes.
  6. AuthController.login() runs, BCrypt-verifies password, mints a JWT, returns it.
Scenario 3: Expired token
  1. Request arrives with a 25-hour-old JWT.
  2. Filter extracts the token.
  3. jwtUtil.isTokenExpired(token) returns true.
  4. The if (!jwtUtil.isTokenExpired(token)) branch is skipped — no authentication is set.
  5. Filter still calls filterChain.doFilter(...).
  6. Downstream SecurityConfig rules require authentication — request is rejected with 401.
  7. Frontend's handleResponse wipes localStorage and redirects to /login.
Scenario 4: Tampered token
  1. Attacker modifies the payload to escalate from SAREE_OWNER to ADMIN.
  2. Filter extracts the token, calls isTokenExpired (which internally parses signature).
  3. Signature recomputation fails — SignatureException thrown.
  4. Exception propagates out of doFilterInternal.
  5. Spring's exception handling in the filter chain catches it.
  6. Final response: 401 Unauthorized. No business code ever runs.
Scenario 5: Missing Authorization header
  1. Request to a protected endpoint arrives with no Authorization header.
  2. Filter: authHeader == null → skip the whole "if" block.
  3. SecurityContext stays empty.
  4. Filter chain continues; SecurityConfig's authentication requirement rejects → 401.
Common filter mistakes
Returning early (not calling filterChain.doFilter)
Always call doFilter, even on failure. Otherwise the request just hangs — no response is ever sent. Skip authentication if you must, but always continue the chain.
Catching exceptions inside the filter and returning 200 OK
If parsing fails, let the exception propagate. Spring Security's exception translation will turn it into 401. Catching and swallowing means a tampered token might look "OK" downstream.
Forgetting to extend OncePerRequestFilter
Use OncePerRequestFilter, not the bare Filter interface. It deduplicates against internal Spring forwards/includes, avoiding subtle "filter ran twice" bugs.
Hard-coding "Bearer " checks inside service code
Centralize the Bearer-token extraction in this filter. Services should never look at the Authorization header — they should read CurrentUserUtil instead. Single source of truth.
Calling the filter for /api/auth/login too
That's exactly what shouldNotFilter prevents. Without it, login requests would fail because there's no JWT yet.
Reading Claims fields with .get(key) (returns Object) without casting
Use the typed overload: claims.get("role", String.class). Catches typos at compile time and returns the right type immediately.
Interview questions about JwtAuthenticationFilter
What's the difference between extending Filter and OncePerRequestFilter?
Filter is the raw Servlet API — Spring may run it multiple times per request if there are internal forwards or includes. OncePerRequestFilter is Spring's wrapper that uses request attributes to ensure the filter runs exactly once. For most use cases — including auth — OncePerRequestFilter is the safer choice.
Why does the filter pass Claims as the principal instead of a username?
Because the Claims already contain everything we need — userId, role, employeeId, ownerId — and they're cryptographically signed. Using Claims as the principal lets every downstream layer read user identity without a DB lookup. Compare to a username principal, where every service method would need to load the User from the DB to find out the role.
What happens to the security context after the request finishes?
Spring Security clears it. The SecurityContextHolder defaults to ThreadLocal mode, and the filter chain's outer wrapper resets it at the end of each request. So even if the same thread serves the next request, no leftover identity contaminates it.
How would you handle multiple authentication mechanisms (JWT + API keys)?
Add a second filter (e.g., ApiKeyAuthenticationFilter) that runs before or after this one. Each filter checks for its own credential type; whichever finds one populates the security context first. Both filters call filterChain.doFilter unconditionally so the chain continues regardless.
Why is shouldNotFilter used instead of just checking the path inside doFilterInternal?
shouldNotFilter is Spring's optimized hook — it's checked BEFORE the filter is invoked, saving the overhead of an inner method call. Checking inside doFilterInternal would also work but is slightly slower and more error-prone.
What's the order of filters in Spring Security?
SecurityConfig registers our JWT filter explicitly with .addFilterBefore(jwtAuthenticationFilter, UsernamePasswordAuthenticationFilter.class). That places it before Spring's built-in form-login filter. The full chain runs in a defined order; if you ever add custom filters, you control where they slot in.
If the JWT is valid but the user has since been deleted, what happens?
The filter happily authenticates them based on the JWT alone — it doesn't hit the DB. The deleted user can make requests until their token expires. If you need immediate revocation, you'd add a per-request DB check (e.g., "is this userId still active?"). NOOL accepts the up-to-24h window because the trade-off is statelessness.
