🤔 The Slightly-Unusual Path: /employee-daily-working

You'll have noticed: this controller's @RequestMapping is "/employee-daily-working" — with an "-ing" suffix. The other controllers use cleaner nouns like /employees and /attendance. Why the inconsistency?

Honest answer: historical drift. The path was named when the feature was being thought of as a verb ("daily working" = the act of working daily). A cleaner name following the rest of NOOL's conventions would be /daily-work or /employee-daily-work. Renaming it now would mean updating the frontend's api.js service AND the SecurityConfig rules — doable but not free. So the URL stayed. A worthwhile cleanup for a future major version.

package com.nool.backend.controller.employee;

import com.nool.backend.dto.common.DateRangeDto;
import com.nool.backend.dto.common.PaginationRequestDto;
import com.nool.backend.dto.common.PaginationResponseDto;
import com.nool.backend.dto.dailywork.EmployeeDailyWorkListDto;
import com.nool.backend.dto.dailywork.EmployeeDailyWorkRequestDto;
import com.nool.backend.dto.dailywork.EmployeeDailyWorkResponseDto;
import com.nool.backend.dto.dailywork.EmployeeWorkSummaryDto;
import com.nool.backend.service.employee.EmployeeDailyWorkService;
import jakarta.validation.Valid;
import lombok.RequiredArgsConstructor;
import org.springframework.web.bind.annotation.*;

@RestController
@RequestMapping("/employee-daily-working")   // all URLs: /api/employee-daily-working/*
@RequiredArgsConstructor
public class EmployeeDailyWorkController {

    private final EmployeeDailyWorkService employeeDailyWorkService;

    // ── 1. LOG DAILY WORK ──────────────────────────────────────
    @PostMapping
    public EmployeeDailyWorkResponseDto addDailyWork(
            @Valid @RequestBody EmployeeDailyWorkRequestDto requestDto){
        return employeeDailyWorkService.addDailyWork(requestDto);
    }

    // ── 2. ADMIN PAGINATED LIST ────────────────────────────────
    @PostMapping("/list")
    public PaginationResponseDto<EmployeeDailyWorkListDto> getDailyWorkList(
            @RequestBody PaginationRequestDto paginationRequestDto){
        return employeeDailyWorkService.getDailyWorkList(paginationRequestDto);
    }

    // ── 3. WORKER'S OWN PAGINATED LIST ─────────────────────────
    @PostMapping("/my-list")
    public PaginationResponseDto<EmployeeDailyWorkListDto> getMyDailyWorkList(
            @RequestBody PaginationRequestDto paginationRequestDto){
        return employeeDailyWorkService.getMyDailyWorkList(paginationRequestDto);
    }

    // ── 4. SUMMARY FOR A SPECIFIC EMPLOYEE ─────────────────────
    @PostMapping("/employee/{employeeId}/summary")
    public EmployeeWorkSummaryDto getEmployeeWorkSummary(
            @PathVariable Long employeeId,
            @Valid @RequestBody DateRangeDto dateRangeDto){
        return employeeDailyWorkService.getEmployeeWorkSummary(employeeId, dateRangeDto);
    }

    // ── 5. WORKER'S OWN SUMMARY ────────────────────────────────
    @PostMapping("/summary")
    public EmployeeWorkSummaryDto getMyWorkSummary(
            @Valid @RequestBody DateRangeDto dateRangeDto){
        return employeeDailyWorkService.getMyWorkSummary(dateRangeDto);
    }
}

📁 Notice — No GET, No PUT, No DELETE

This is the first NOOL controller we've seen with only POST endpoints.

Standard CRUD verbNOOL daily-work patternWhy
POST (Create)POST /Standard — add a new row
GET (Read by id)(none)No "get one row" endpoint — daily-work rows are seen via lists only
GET (List)POST /listPOST used because the body has many filter fields
PUT (Update)(none)Once entered, daily-work entries shouldn't be edited (history)
PATCH (Partial)(none)Same reasoning
DELETE (Remove)(none — only cascade)Daily-work history is never directly deleted; only removed when a worker is hard-deleted
Why no GET / PUT / DELETE? Once Dad logs Prema's work for May 15, that row is meant to be permanent history. If a mistake is made, the correct fix is to delete the row (admin-only DB action) and re-create it — not to PATCH the count. This "append-only-ish" pattern mirrors the ledger philosophy from the SareeLedgerEntry (Chapter 15). Keeping history immutable preserves audit integrity for salary calculations.

🛡️ Where Authorization Is Enforced

Same two-layer model as every other controller:

  1. SecurityConfig URL rules (Chapter 22). The Daily Work rules:
    • POST /employee-daily-working → ADMIN only
    • POST /employee-daily-working/list → ADMIN only
    • POST /employee-daily-working/my-list → ADMIN or WORKER
    • POST /employee-daily-working/employee/*/summary → ADMIN or WORKER
    • POST /employee-daily-working/summary → ADMIN or WORKER
  2. Service-layer self-or-admin check (Chapter 36) — narrows the /employee/{id}/summary endpoint so workers can only request their own ID.

🎬 Real-World Scenarios

Scenario 1: Dad logs Friday's polishing for all workers
  1. End of day. Dad asks each worker. He fills the form 5 times — once per worker.
  2. Each submission goes to POST /api/employee-daily-working.
  3. JWT filter (Chapter 21) verifies token, sets role=ADMIN.
  4. SecurityConfig matches the URL → ADMIN required → passes.
  5. @Valid on the DTO validates fields (all four @NotNull pass, both counts ≥ 0).
  6. Service inserts the row. UNIQUE constraint backstops against duplicates.
  7. Response includes todayRevenue per worker. Dad sees a running total.
Scenario 2: Prema checks her own week
  1. Saturday morning. Prema opens NOOL on her phone.
  2. Frontend POSTs /api/employee-daily-working/my-list with pagination.
  3. SecurityConfig: ADMIN or WORKER → passes.
  4. Service reads employeeId=5 from JWT, fetches paginated rows.
  5. Each row includes its computed todayEarning.
  6. Frontend renders a table. Prema sees: Mon ₹240, Tue ₹270, Wed ₹300, Thu (no row), Fri ₹240.
Scenario 3: Dad pulls workforce productivity for May
  1. Admin → Workforce Analytics → "May 1-15".
  2. For each worker, frontend POSTs /api/employee-daily-working/employee/{id}/summary with { "fromDate": "2026-05-01", "toDate": "2026-05-15" }.
  3. SecurityConfig: ADMIN or WORKER → passes.
  4. Service self-or-admin: role=ADMIN → no narrowing.
  5. Service computes totals: 13 workdays, 104 fresh, 26 repolish, ₹3,120 revenue.
  6. Dashboard renders a sortable table.
Scenario 4: Ravi tries to view Prema's summary
  1. Ravi (WORKER, employeeId=8) crafts POST /api/employee-daily-working/employee/5/summary.
  2. SecurityConfig: WORKER allowed at the URL level.
  3. Service: role=WORKER, caller(8) != requested(5) → throws AccessDeniedException.
  4. Spring returns 403. Defense in depth.
Scenario 5: Prema views her own summary via /summary
  1. Prema's app POSTs /api/employee-daily-working/summary with just the date range.
  2. No employeeId in the URL — service infers from JWT.
  3. Service delegates to getEmployeeWorkSummary(5, range).
  4. Permission check: trivially passes (caller is requesting own data).
  5. Returns summary DTO.
  6. The URL itself makes it impossible to request someone else's summary — extra safety on top of the in-service check.

Common daily-work-controller mistakes
Adding a PATCH endpoint to edit existing rows
Daily work entries are meant to be immutable history. If a mistake needs correction, the right path is DB-level admin action — not an API endpoint that workers could exploit to revise their counts.
Renaming the path to /daily-work without coordination
The frontend's api.js hardcodes /employee-daily-working. SecurityConfig has rules referencing this path. Renaming requires updating both. Plan it as a coordinated change.
Forgetting @Valid on the request body
Without it, @PositiveOrZero on counts doesn't run — negative numbers would silently be accepted, throwing off salary calculations.
Returning the entity directly from addDailyWork
The DTO contains the computed todayRevenue field. The entity doesn't have that — it's derived. Always go through the service's DTO builder.
Adding a "delete one row" endpoint
Conscious omission. Daily work history shouldn't be deleted casually — it's the basis of salary trust. If correction is needed, hard-delete a worker (which cascades) or fix at the DB level.
Interview questions
Why does this module have no GET-by-id endpoint?
Because the use cases don't need it. The UI shows either a paginated list (with all fields) or a summary (aggregated). There's never a reason to fetch one specific work row by its ID — you'd already have all its data from the list. Avoiding unnecessary endpoints keeps the API surface small.
Why is the base path /employee-daily-working and not just /daily-work?
Historical artifact. The verb-form name was chosen early and stuck. Renaming would require coordinated changes to the frontend api.js, SecurityConfig rules, and any external documentation. The cost-benefit didn't justify the cleanup. A future major-version rewrite would fix it.
What's the role pattern across all five endpoints?
Writes are ADMIN-only (only Dad can log work). Admin reads (/list, /employee/{id}/summary) are ADMIN-or-WORKER, with the service narrowing to "self only" for workers. Worker-scoped reads (/my-list, /summary) accept either role but always operate on the caller's own data — physically impossible to misuse.
If a worker accidentally types negative numbers, what happens?
The frontend prevents it (form validation). If somehow bypassed, the @PositiveOrZero on the DTO catches it. Validation kicks in BEFORE the method body runs; MethodArgumentNotValidException bubbles up to GlobalExceptionHandler (Chapter 09); user gets a clean 400 with a friendly message.
Could you add an /employee-daily-working/today endpoint?
Easily — but the existing /list with a sortBy=workDate DESC and size=N serves the same need. Adding more endpoints means more SecurityConfig rules, more test surface, more documentation. The NOOL philosophy: avoid endpoints that don't earn their keep.
