Chapter 04 · The Foundation

The Database Model —
Every table, every relationship

The database is the foundation. If the data model is wrong, no amount of clever code can fix it. In this chapter we'll see every table in NOOL, what each one stores, and how they connect.

🧠 Why I Designed the Database First

Before writing a single line of Java code, I drew the database tables on paper. Why? Because everything else in the app is downstream of the data model. The Java entities mirror the tables. The DTOs are reshapes of entity data. The REST endpoints expose entities. The React forms collect data that ends up in entities. Get the tables right, and everything else falls into place.

The rule I followed: Every table answers ONE clear question. users answers "who can log in?". saree_ledger answers "which sarees came in and went out, when, for whom?". If a table has unclear purpose, it probably needs to be split into two.

📚 Each Table, Explained

Now let's go through each table in detail. We'll deep-dive each one in its own chapter later, but here's the summary you need first.

1. users — who can log in?

The foundational table. Every person who logs into NOOL (admin, worker, or owner) has a row here.

ColumnTypePurpose
user_idBIGINT PKAuto-generated unique ID
mobile_numberVARCHAR(15) UNIQUE10-digit Indian mobile, used as login username
passwordVARCHAR(100)BCrypt hash of the password (never plain text!)
roleENUMADMIN / WORKER / SAREE_OWNER
active_statusBOOLEANIf false, login is blocked (soft-disable)
created_at, updated_atDATETIMEAudit timestamps

2. user_profile — the bridge to business entities

A User has login credentials only. To find out what business entity they represent (Employee? Owner?), we look up their UserProfile row. It points to either an Employee OR a SareeOwner — never both.

ColumnTypePurpose
profile_idBIGINT PKAuto-generated ID
user_idBIGINT FK UNIQUELinks to users(user_id)
employee_idBIGINT FK nullableIf this user is a worker, points to their employees row
owner_idBIGINT FK nullableIf this user is a saree owner, points to their saree_owner row
Admin users have NULL for both employee_id and owner_id, because they aren't a worker or an owner — they're the business owner (my mom and dad).

3. employees — workers / polishers

One row per worker at the workshop (Prema, Ravi, etc.). Holds business-specific info, not login data.

ColumnTypePurpose
employee_idBIGINT PKAuto-generated
employee_nameVARCHAR(255)"Prema", "Ravi"
mobile_numberVARCHAR UNIQUESame as their User row's mobile
joining_dateDATEFirst day on the job
statusENUMACTIVE / LEFT
polishing_rateDOUBLE₹ per saree this worker earns
user_idBIGINT FKTheir login row (can be NULL temporarily)

4. attendance — daily roll call

One row per (employee, date) pair, recording whether they showed up.

ColumnTypePurpose
attendance_idBIGINT PKAuto-generated
employee_idBIGINT FKWhich worker
attendance_dateDATEThe day
attendance_statusENUMPRESENT / ABSENT
remarksVARCHAROptional note
UNIQUE constraint: (employee_id, attendance_date) — you cannot mark Prema present twice for the same day. The database itself refuses. This prevents accidental double-counting.

5. employee_daily_work — daily polishing output

One row per (employee, date) — what each worker produced that day.

ColumnTypePurpose
work_idBIGINT PKAuto-generated
employee_idBIGINT FKWhich worker
work_dateDATEThe day
fresh_saree_countINTBrand-new sarees polished
repolish_saree_countINTTouch-up polishes
remarksVARCHAROptional

Total earnings for the day = (fresh + rePolish) × employee.polishing_rate. Also has UNIQUE on (employee_id, work_date).

6. salary_payment — money paid to workers

One row per payment event. NOT one per day worked — one per payment.

ColumnTypePurpose
payment_idBIGINT PKAuto-generated
employee_idBIGINT FKWhom we paid
from_date / to_dateDATEThe work period being paid for
amount_paidDECIMALHow much
payment_dateDATEWhen the payment was made
payment_modeENUMCASH / UPI / BANK_TRANSFER
remarksVARCHAROptional

Pending salary for a worker = Sum(earnings from daily_work in range) − Sum(payments in range).

7. saree_owner — clients

One row per client (Rahul, Sharma Madam, etc.).

ColumnTypePurpose
owner_idBIGINT PKAuto-generated
owner_nameVARCHAR"Rahul Textiles", "Sharma Sarees"
mobile_numberVARCHAR UNIQUESame as their User row
statusENUMACTIVE / INACTIVE
polish_rate_per_sareeDOUBLE₹ this owner pays per returned saree
user_idBIGINT FKTheir login row

8. saree_ledger ⭐ — the inventory ledger (THE most important table)

This is the heart of NOOL. Every saree that enters or leaves the workshop becomes one row in this table. Like a bank passbook.

ColumnTypePurpose
entry_idBIGINT PKAuto-generated row ID
owner_idBIGINT FKWhose sarees
entry_typeENUMRECEIPT (in) or RETURN (out)
entry_dateDATEThe business date
quantityINTHow many sarees (always positive)
remarksVARCHAROptional note ("Diwali batch")
created_atDATETIMEWhen the row was inserted (audit)

This table is append-only — we never UPDATE or DELETE rows. To correct a mistake, we insert a compensating row. The "in-hand" balance is always derived:

Sarees still with the workshop (for one owner)
inHand = SUM(RECEIPT.quantity) − SUM(RETURN.quantity)

9. owner_payment — money received from clients

One row per payment received. Same shape as salary_payment but the direction is reversed.

ColumnTypePurpose
payment_idBIGINT PKAuto-generated
owner_idBIGINT FKWho paid
amount_paidDECIMALHow much
payment_dateDATEWhen
payment_modeENUMCASH / UPI / BANK_TRANSFER
remarksVARCHAROptional

🎨 Schema Patterns Used Throughout NOOL

Pattern 1 — Append-only ledger

Used for saree_ledger. Never UPDATE, never DELETE. Each row is one immutable event. Balance is always derived from SUM. Why? Complete audit trail + no risk of drift.

Pattern 2 — Soft delete via status flag

Used for users.active_status, employees.status, saree_owner.status. To "delete" a worker, flip status to LEFT or INACTIVE. The row stays in the DB so foreign-key relationships (attendance, daily work, salary) stay intact. Hard-delete would orphan or cascade-destroy years of history.

Pattern 3 — Unique constraint to prevent duplicates

Used on (employee_id, attendance_date) and (employee_id, work_date). The database itself refuses double-entries. Catches bugs the application code might miss.

Pattern 4 — Auto-timestamps via @PrePersist / @PreUpdate

Every table has created_at (set once on insert) and most have updated_at (updated on every edit). These are filled automatically by JPA lifecycle callbacks — I never have to remember to set them.

Pattern 5 — Enums stored as STRING

Every enum (Role, EmployeeStatus, etc.) is stored as its NAME in the DB ("ADMIN"), not as a number (0). Why? Readability in SQL queries, and resilience — if I reorder the enum in Java, the DB doesn't silently break.

Pattern 6 — Per-entity rates (no global price)

Each owner has their own polish_rate_per_saree. Each worker has their own polishing_rate. Rates are NOT global settings, because real businesses negotiate per-person. This unlocks loyalty discounts, performance raises, etc.

🪄 How These Tables Get Created Automatically

I never typed out any of those CREATE TABLE statements manually. JPA + Hibernate did it for me. In application.properties we set:

backend/src/main/resources/application.properties properties 
spring.jpa.hibernate.ddl-auto=update

This setting tells Hibernate: "When the app starts, look at all my @Entity classes, compare them to the actual MySQL schema, and ALTER the schema to match." So when I add a new field to the Employee Java class, Hibernate runs an ALTER TABLE automatically on next startup. No manual migrations needed.

For production this would change! ddl-auto=update is great for development but risky in production (Hibernate can drop columns or make destructive changes if your entities are wrong). For production, you'd switch to validate (or none) and use a migration tool like Flyway or Liquibase. NOOL hasn't reached production scale yet, so update is fine.

🎯 Key Things to Remember

  1. Users are separate from Employees and Owners. Login info ≠ business info.
  2. The ledger is append-only. Balance is always SUM(RECEIPT) − SUM(RETURN).
  3. Soft delete only. Disable via status flag, never DROP rows.
  4. Unique constraints on (employee, date) for attendance and daily work prevent duplicates.
  5. Rates live on individual entities — not global settings.
  6. Hibernate creates the schema for you from @Entity classes (in dev mode).
  7. Every table has audit timestamps (created_at, updated_at) filled automatically.
