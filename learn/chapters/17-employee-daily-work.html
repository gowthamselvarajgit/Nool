🧠 What This Entity Tracks

Attendance answers "did Prema show up today?". EmployeeDailyWork answers "what did she produce while she was here?" — broken into two numbers:

  • freshCount — brand-new sarees polished for the first time
  • rePolishCount — touch-ups on sarees that came back from a client

Together, those two numbers × the worker's polish rate = their earnings for that day. Multiply by all days of the week = weekly earnings. The whole salary system is built on summing these rows.

Why split fresh vs re-polish? Because they're operationally different work. Sometimes my parents want to know "how many of yesterday's polishes were re-work?" as a quality signal. Storing the two counts separately lets that analysis happen later without changing the schema. The salary formula treats them equally (both earn the same rate), but the data tells two stories.

package com.nool.backend.entity.employee;

import jakarta.persistence.*;
import lombok.*;
import java.time.LocalDate;
import java.time.LocalDateTime;

@Entity
@Table(
    name = "employee_daily_work",
    uniqueConstraints = {                                  // ★ same pattern as Attendance
        @UniqueConstraint(columnNames = {"employee_id", "work_date"})
    }
)
@Getter @Setter
@NoArgsConstructor
@AllArgsConstructor
@Builder
public class EmployeeDailyWork {

    // ── PRIMARY KEY ───────────────────────────────────────────────
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    @Column(name = "work_id")
    private Long id;

    // ── THE DATE ──────────────────────────────────────────────────
    @Column(name = "work_date", nullable = false)
    private LocalDate workDate;
    // Day-precision — same reasoning as Attendance.attendanceDate.

    // ── FRESH SAREE COUNT ────────────────────────────────────────
    @Column(name = "fresh_saree_count", nullable = false)
    private Integer freshCount;
    // Number of brand-new sarees polished today. Always >= 0.

    // ── RE-POLISH COUNT ──────────────────────────────────────────
    @Column(name = "repolish_saree_count", nullable = false)
    private Integer rePolishCount;
    // Number of touch-up polishes today. Always >= 0.

    // ── REMARKS (optional) ────────────────────────────────────────
    @Column(name = "remarks")
    private String remarks;
    // e.g., "Heavy work day", "Two damaged sarees rejected".

    // ── EMPLOYEE LINK ─────────────────────────────────────────────
    @ManyToOne
    @JoinColumn(name = "employee_id", nullable = false)
    private Employee employee;
    // Many daily-work rows belong to one employee — across days.

    // ── AUDIT TIMESTAMPS ──────────────────────────────────────────
    @Column(name = "created_at", nullable = false, updatable = false)
    private LocalDateTime createdAt;

    @Column(name = "updated_at")
    private LocalDateTime updatedAt;

    // ── LIFECYCLE HOOKS ───────────────────────────────────────────
    @PrePersist
    protected void onCreate(){
        LocalDateTime now = LocalDateTime.now();
        this.createdAt = now;
        this.updatedAt = now;
    }

    @PreUpdate
    protected void onUpdate(){
        this.updatedAt = LocalDateTime.now();
    }
}

💰 The Earnings Formula

This entity exists primarily to drive ONE calculation:

A worker's earnings on a single day
dailyEarnings = (freshCount + rePolishCount) × employee.polishRate

With Prema's rate of ₹30/saree and her work row above (8 fresh + 2 re-polish):

dailyEarnings = (8 + 2) × ₹30 = ₹300 for May 15

To get weekly or monthly earnings, the salary service sums across many rows:

Total earnings for a date range
totalEarnings = SUM((freshCount + rePolishCount) × polishRate) over rows in [from, to]

Then subtract the SUM of salary_payment.amount_paid for the same range to get the pending salary. We'll see that exact code in Chapter 39 (SalaryServiceImpl).

🗄️ What This Becomes in MySQL

CREATE TABLE employee_daily_work (
    work_id              BIGINT      NOT NULL AUTO_INCREMENT,
    employee_id          BIGINT      NOT NULL,
    work_date            DATE        NOT NULL,
    fresh_saree_count    INT         NOT NULL,
    repolish_saree_count INT         NOT NULL,
    remarks              VARCHAR(255),
    created_at           DATETIME(6) NOT NULL,
    updated_at           DATETIME(6),

    PRIMARY KEY (work_id),
    UNIQUE KEY uk_dailywork_emp_date (employee_id, work_date),   -- ★ the guard
    FOREIGN KEY (employee_id) REFERENCES employees(employee_id)
);

Notice that EmployeeDailyWork is structurally identical to Attendance:

Why two tables instead of one? They answer different business questions and have different audit requirements. Attendance can be inferred quickly ("is this worker around today?"); daily work needs deliberate end-of-day entry. Mixing them would mean rows with attendance but no counts, or counts but no attendance — confusing. Keep them separate, indexed identically, and let the salary service JOIN them when needed.

🔢 Why Integer (Wrapper) Not int (Primitive)?

Notice freshCount and rePolishCount are Integer, not the primitive int. There's a practical reason for entities:

Integer (wrapper)int (primitive)
Can be nullCannot be null (always defaults to 0)
Hibernate distinguishes "not set" from "0"Hibernate sees 0 — ambiguous
Bean Validation (@NotNull, @Min) works@NotNull can't apply to primitives
Slightly more memoryTiny bit less memory

For entity fields, Integer is the right call — it lets the schema and code distinguish between "the value is zero" and "the value was never provided", and plays well with validation annotations.

🎬 Real-World Scenarios

Scenario 1: Dad logs Prema's day-end output
  1. 5:30 PM. Dad logs into NOOL → Admin → Daily Work → Add.
  2. Picks Prema, date today (2026-05-15), fresh count 8, re-polish 2.
  3. Frontend POSTs to /api/employee-daily-working.
  4. Service checks: row exists for (5, today)? No.
  5. Inserts: work_id=89, employee_id=5, work_date=2026-05-15, fresh=8, repolish=2.
  6. Prema's earnings for the day are now derivable: (8+2) × ₹30 = ₹300.
Scenario 2: Dad realizes he made a typo
  1. After saving, Dad realizes he typed fresh=8 but it should have been 9.
  2. He opens the existing entry, changes fresh from 8 to 9, saves.
  3. Backend issues UPDATE employee_daily_work SET fresh_saree_count=9, updated_at=NOW() WHERE work_id=89.
  4. @PreUpdate stamps the new updated_at. Original created_at stays.
  5. New earnings: (9+2) × ₹30 = ₹330. All downstream calculations reflect the correction immediately.
Scenario 3: Two NOOL tabs open, both try to save Prema for today
  1. Mom is using NOOL on her phone; Dad on the laptop. Both navigate to Prema's daily work page.
  2. Both type fresh/repolish values, both click Save almost simultaneously.
  3. One INSERT succeeds first. The second hits the UNIQUE constraint and fails.
  4. The failing client gets a 409 with message: "A daily work entry already exists for this employee on this date."
  5. The race condition is impossible to corrupt the data — at most one row exists for that (employee, date).
Common mistakes around EmployeeDailyWork
Using int instead of Integer for counts
Use Integer for entity fields. It lets you distinguish "not set" from 0, allows validation annotations like @NotNull, and avoids subtle Hibernate quirks.
Combining fresh and re-polish into one number
Keep them separate — they tell different stories. Even though the salary formula adds them, future analytics may want to know the breakdown ("rework ratio" = re-polish ÷ total).
Allowing negative counts (typo: -8 instead of 8)
Add a validation rule like @Min(0) on the corresponding DTO field. The entity itself doesn't enforce non-negative, but the DTO + GlobalExceptionHandler combo catches it before insert.
Storing the earned amount on this row (e.g., amountEarned = ₹300)
Don't store derived values. Compute (fresh + rePolish) × polishRate on demand. If the rate ever changes, the stored value would become wrong. The ledger pattern again: store events, derive numbers.
Forgetting that a day with zero work still needs no row
Days a worker did no polishing simply have no row in this table. Don't insert "0/0" rows — they clutter queries. The salary service treats "no row" as 0 earnings for the day automatically (since SUM() ignores missing rows).
Interview questions about EmployeeDailyWork
Why split fresh and re-polish into two counts?
They're operationally different work. Fresh polish is on brand-new sarees; re-polish is touch-ups on sarees that came back from a client. The salary formula treats them equally (both earn the same rate), but the breakdown is useful for analytics: "what's our rework ratio?" Storing them separately costs nothing and unlocks future questions without a schema migration.
Why is this table different from attendance when they look so similar?
They answer different questions. Attendance is "did the worker show up?"; DailyWork is "what did they produce?". A worker can be PRESENT and have done zero work (training day), or be ABSENT with zero work. Mixing them into one table would create awkward semantics — "are these counts ABSENT-and-still-did-3-sarees?" Keeping them separate is clearer.
If a worker did 0/0 on a day, do we insert a row?
No — we just don't insert anything. The salary calculation uses SUM, which ignores missing rows. Having an explicit "0/0" row would be noise: it wouldn't change any totals and would clutter dashboard queries. The presence of a row means "we logged work that day". The absence means "nothing to log".
Why isn't the earned amount stored on each row?
Because it's derived — and storing derived values invites drift. If we ever change Prema's rate from ₹30 to ₹32, old rows with the old computed amount become wrong. Storing only the counts + employee FK lets us always recompute at query time using the current rate. Same philosophy as the inventory ledger.
What's the UNIQUE constraint protecting against?
Double-logging the same worker for the same day — accidentally inserting two rows (8/2 and 9/3) for (Prema, May 15). Without it, the salary calculation would double-count. The constraint at the database level makes the corruption impossible regardless of any bug in the application layer.
Could you compute fresh-vs-rePolish ratios using only this table?
Yes — SELECT employee_id, SUM(fresh_saree_count) AS fresh, SUM(repolish_saree_count) AS rep FROM employee_daily_work WHERE work_date BETWEEN ? AND ? GROUP BY employee_id. Each worker's rework ratio = rep ÷ (fresh + rep). That's exactly the kind of analysis the split design enables.
