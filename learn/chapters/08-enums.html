The 6 Enums —
The fixed list of choices

Before we build entities, services, or controllers, we need to define the fixed vocabulary the system uses: what counts as a role? What's a valid attendance status? What payment methods do we accept? In Java, the answer to "fixed list of choices" is an enum. NOOL has six of them.

🤔 What is an Enum?

An enum (short for "enumeration") is a Java type that has a fixed, finite set of allowed values. Once you declare:

A minimal enum exampleJava
public enum Role {
    ADMIN,
    WORKER,
    SAREE_OWNER
}

...then anywhere in the code that expects a Role, only those three values are legal. The compiler refuses to let you write Role.PRESIDENT — it doesn't exist. You can't typo "admin" vs "ADMIN" because there are no strings involved. It's compile-time-enforced spelling.

Why we use enums everywhere: NOOL has a lot of "this can only be one of a few things" situations — what's the role? What's the attendance status? What's the payment mode? Each of these is an enum. Three benefits: (1) spelling is enforced, (2) the list of valid choices lives in ONE place, (3) IDEs auto-complete the options.

All six live in the same folder: backend/src/main/java/com/nool/backend/enums/. Each file is tiny — usually 5-7 lines including the package declaration. Let's walk through them one by one.

1️⃣ Role — Three Kinds of Users

enums/Role.java Java 
package com.nool.backend.enums;

public enum Role {
    ADMIN,           // my mom and dad — full control
    WORKER,          // polishers like Prema
    SAREE_OWNER      // clients like Rahul
}

What each value means

  • ADMIN — can do everything: add/edit workers and owners, mark attendance, record inventory, pay salaries, see analytics. There are 1-2 admins (my mom and dad).
  • WORKER — a polisher. Sees only their own attendance, work, and salary. Cannot edit anyone else's data.
  • SAREE_OWNER — a client. Sees only their own inventory and payments. Cannot edit anything.

Where it's used

  • Stored in the users.role column for every login row
  • Embedded in every JWT token as a claim — every API request carries the user's role
  • Drives the per-role sidebar in Layout.jsx (different menus per role)
  • Drives SecurityConfig's URL rules — .hasAuthority("ADMIN") etc.

2️⃣ EmployeeStatus — Is This Worker Still With Us?

enums/EmployeeStatus.java Java 
package com.nool.backend.enums;

public enum EmployeeStatus {
    ACTIVE,    // currently employed
    LEFT       // no longer with the workshop (soft-deleted)
}

What each value means

  • ACTIVE — the worker is still employed. Default for new hires. Their row counts in payroll lists, attendance forms, etc.
  • LEFT — they've stopped working at the workshop. Their row stays in the DB (for salary history, audit, etc.), but they're hidden from the "current employees" list and can't log in.
Why "LEFT" and not "DELETE"? Because soft delete is safer. If we hard-deleted a worker, all their daily-work, attendance, and salary-payment records would either get orphaned or cascade-deleted (losing history). Flipping the status to LEFT preserves everything.

Real-world scenario

A worker leaves the workshop
  1. One of the polishers, Suresh, decides to move to another city.
  2. Dad opens NOOL → Employees → Suresh's row → clicks "Mark as Left".
  3. Backend runs: UPDATE employees SET status='LEFT' WHERE employee_id = 7.
  4. Suresh's user account also gets active = false so he can't log in anymore.
  5. All of Suresh's historical daily-work and salary rows remain in the DB — Dad can still pull up "what did we pay Suresh last March?" any time.

3️⃣ OwnerStatus — Is This Client Still Active?

enums/OwnerStatus.java Java 
package com.nool.backend.enums;

public enum OwnerStatus {
    ACTIVE,      // currently a client
    INACTIVE     // no longer doing business with the workshop
}

What each value means

  • ACTIVE — currently doing business with the workshop. Appears in dropdowns, can be selected for new receipts/returns.
  • INACTIVE — no longer a client. Historical ledger rows preserved. Hidden from selection in new transactions.
Why "INACTIVE" for owners but "LEFT" for employees? They mean essentially the same thing, but the words match how my parents naturally describe each role. A worker "left the job"; a client "isn't buying anymore". Domain language > technical uniformity.

4️⃣ AttendanceStatus — Did They Show Up Today?

enums/AttendanceStatus.java Java 
package com.nool.backend.enums;

public enum AttendanceStatus {
    PRESENT,    // worker came to the workshop today
    ABSENT      // worker did not come
}

Two values — kept deliberately simple

My parents wanted the simplest possible attendance system. "Did Prema come today or not?" — that's it. They mark PRESENT or ABSENT, and that's the whole story.

Could we add HALF_DAY, LEAVE, HOLIDAY? Yes, and the codebase has hooks ready for it — the React UI (formatters.js) can already render labels for "Half Day", "On Leave", "Holiday" etc. But the actual backend enum stops at PRESENT/ABSENT to match my parents' real-world workflow. Adding more values later is a one-line change: add to the enum, the DB column already accepts any string up to 255 chars.

Where it's used

  • Stored in attendance.attendance_status
  • The Attendance calendar (worker dashboard) colors days green (PRESENT) or red (ABSENT)
  • The salary summary uses it to compute "worked days vs. absent days" trends

5️⃣ LedgerEntryType — Sarees IN or OUT?

enums/LedgerEntryType.java Java 
package com.nool.backend.enums;

public enum LedgerEntryType {
    RECEIPT,    // sarees coming INTO the workshop from an owner
    RETURN      // sarees going OUT, returned to the owner after polishing
}

The cornerstone of the inventory system

Remember from Chapter 04 — NOOL's inventory is an append-only ledger. Every transaction becomes a row in saree_ledger, and this enum decides what kind of transaction it is. The in-hand balance formula uses this enum directly:

In-hand balance for one owner
inHand = SUM(quantity WHERE type=RECEIPT) − SUM(quantity WHERE type=RETURN)

Real-world scenario

Walking through Rahul's week

Monday: Rahul drops off 100 sarees. NOOL inserts:

INSERT INTO saree_ledger
  (owner_id, entry_type, entry_date, quantity)
VALUES (1, 'RECEIPT', '2026-05-11', 100);

Friday: Dad returns 50 polished. NOOL inserts:

INSERT INTO saree_ledger
  (owner_id, entry_type, entry_date, quantity)
VALUES (1, 'RETURN', '2026-05-15', 50);

Rahul's current in-hand: 100 − 50 = 50 still being polished.

6️⃣ PaymentMode — How the Money Moved

enums/PaymentMode.java Java 
package com.nool.backend.enums;

public enum PaymentMode {
    CASH,            // physical rupee notes — most common in the workshop
    UPI,             // PhonePe / Google Pay / etc.
    BANK_TRANSFER    // NEFT / IMPS / cheque deposits
}

Why all three matter

  • CASH — still the default in small Indian businesses. My parents pay most workers in cash at the end of the week.
  • UPI — phone-based instant payments (PhonePe, GPay). Increasingly common, especially for owner payments.
  • BANK_TRANSFER — for larger amounts. Cheque deposits, NEFT.

Recording the mode is useful for two reasons: tax/audit ("show me all CASH payments this month for the books") and reconciliation (cross-checking with the bank statement).

Where it's used

  • salary_payment.payment_mode — how a worker was paid
  • owner_payment.payment_mode — how a client paid us
  • Frontend payment forms have a dropdown with these three options

🗄️ How Enums Get Stored in MySQL

Every entity that uses an enum field has it marked with @Enumerated(EnumType.STRING):

Example from User.javaJava
@Enumerated(EnumType.STRING)
@Column(name = "role", nullable = false)
private Role role;

EnumType.STRING tells Hibernate: store the enum's NAME as text, not as a number. So the users.role column ends up with values like 'ADMIN', 'WORKER', 'SAREE_OWNER' — readable in any SQL client. 

-- A sample query showing how enums look in the DB:
SELECT user_id, mobile_number, role, active_status
FROM   users;

user_id | mobile_number | role         | active_status
   1    | 9876543210    | ADMIN        | true
   2    | 9000000001    | SAREE_OWNER  | true
   3    | 9000000002    | WORKER       | true

-- Same idea on saree_ledger:
entry_id | owner_id | entry_type | quantity
   1     | 1        | RECEIPT    | 100
   2     | 1        | RETURN     | 50
The other option is EnumType.ORDINAL — store the value as an integer position (ADMIN=0, WORKER=1, SAREE_OWNER=2). It's slightly smaller, but dangerous: if you ever reorder the enum (move SAREE_OWNER to position 1), all the historical data silently becomes wrong. STRING is safer. Always use STRING for production enums.
Common enum mistakes
Using EnumType.ORDINAL (storing as a number)
Always EnumType.STRING. Reordering the enum in Java code shouldn't silently break the database. Stringy is forwards-compatible; ordinal is fragile.
Forgetting @Enumerated(EnumType.STRING) annotation
Without it, Hibernate defaults to ORDINAL — opening the trap above. Always declare it explicitly on every enum field.
Using a String field instead of an enum ("just store the role as text")
Strings invite typos ("admin" vs "ADMIN" vs "Admin"). Enums force exact match at compile time. The slight extra ceremony of declaring an enum is worth it many times over.
Renaming an enum value casually (e.g., LEFT → RESIGNED)
If you rename, you must also UPDATE the database column to match — otherwise old rows with the old value can't be deserialized. Be deliberate.
Adding new values at the START of the enum
If you use ORDINAL storage, that shifts all positions. Even with STRING storage, prefer adding new values at the END to keep code reviews focused. (Though with STRING storage, the order doesn't actually matter at runtime.)
Interview questions about enums
Why use enums instead of just strings or integers?
Three reasons: compile-time spell-checking (you can't typo Role.ADMINN — it doesn't compile), exhaustive switch statements (compiler warns if you miss a case), and self-documenting code (everyone reading Role.ADMIN knows exactly the value is). Strings give you none of those.
Difference between EnumType.STRING and EnumType.ORDINAL?
STRING stores the enum's name as text ("ADMIN", "WORKER"). ORDINAL stores its position as a number (0, 1). STRING is human-readable, survives reordering, and is safer in long-lived databases. ORDINAL is more compact but brittle. Always use STRING for production enums — the size savings are not worth the risk.
How would you add a new value to an existing enum?
Three steps. (1) Add the new constant to the Java enum (preferably at the end). (2) If using STRING storage and the column is sized large enough, MySQL needs no schema change — it just accepts the new string. (3) Update any switch statements that previously assumed only the old values. With EnumType.STRING + a wide enough VARCHAR column, this is a one-line, backwards-compatible addition.
Can a Java enum have methods?
Yes. Enums in Java are full-fledged classes — they can have fields, constructors, and methods. For example, you could add a displayName() method to Role that returns "Administrator" instead of "ADMIN". NOOL's enums don't currently need methods — they're pure value lists — but the option exists if needed.
Why does NOOL's AttendanceStatus only have PRESENT and ABSENT? Real attendance has half-days, leaves, holidays...
Deliberate simplicity. My parents' real workflow is binary — a worker either showed up that day or didn't. Half-days are rare and handled informally (paid or unpaid by mutual agreement). The frontend's formatters.js already has display labels for HALF_DAY, LEAVE, HOLIDAY etc., so when the business needs it, adding the values is a tiny change to the enum and the DB column.
How do you serialize a Java enum to JSON for the React frontend?
By default, Jackson serializes enums by their .name() — so Role.ADMIN becomes "ADMIN" in JSON. The frontend receives a string and displays it directly. NOOL doesn't customize this (no @JsonValue on enums) — the default works fine.
