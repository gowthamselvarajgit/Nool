🧠 What This Service Does

When Dad clicks "Add Daily Work" for Prema, this is the file that runs. It validates the worker exists, inserts the row, computes today's earnings, and sends back the response. It also powers the worker's "my history" page and the per-worker summary reports. Five public methods and one private helper.

package com.nool.backend.service.impl.employee;

// imports omitted for brevity

@Service
@RequiredArgsConstructor
public class EmployeeDailyWorkServiceImpl implements EmployeeDailyWorkService {

    private final EmployeeDailyWorkRepository employeeDailyWorkRepository;
    private final EmployeeRepository          employeeRepository;

    private EmployeeDailyWorkListDto mapToListDto(EmployeeDailyWork employeeDailyWork) {
        return EmployeeDailyWorkListDto.builder()
                .workId(employeeDailyWork.getId())
                .employeeId(employeeDailyWork.getEmployee().getId())
                .employeeName(employeeDailyWork.getEmployee().getName())
                .workDate(employeeDailyWork.getWorkDate())
                .freshCount(employeeDailyWork.getFreshCount())
                .rePolishCount(employeeDailyWork.getRePolishCount())
                .todayEarning(
                    employeeDailyWork.getFreshCount() *
                    employeeDailyWork.getEmployee().getPolishRate()
                )
                .build();
    }
}

The repository for the employees table. Needed because adding a daily-work entry requires looking up the worker first (to verify they exist and to read their polishRate for earnings computation).
6 private EmployeeDailyWorkListDto mapToListDto(EmployeeDailyWork employeeDailyWork) {
A private helper method. Takes one JPA entity, returns the list-row DTO. Marked private because nothing outside this class should call it. Used in two places (getDailyWorkList and getMyDailyWorkList); extracting it avoids copy-pasting the same builder twice.
7 return EmployeeDailyWorkListDto.builder()
Begin building a DTO using Lombok's generated builder. The builder pattern lets us chain .field(value) calls in any order and end with .build() — much cleaner than a positional constructor with 7 args.
8 .workId(employeeDailyWork.getId())
Copy the entity's primary key (the auto-generated work_id) into the DTO's workId field. .getId() is Lombok's auto-generated getter on the entity.
9 .employeeId(employeeDailyWork.getEmployee().getId())
Flatten the relationship. The entity has a full Employee object via its @ManyToOne; we extract just the ID and put it as a plain Long on the DTO. The frontend doesn't need the whole Employee object — only the ID for internal linking.
10 .employeeName(employeeDailyWork.getEmployee().getName())
Flatten the name too. This is a deliberate denormalization — the frontend renders "Prema · 8 + 2 sarees" in the list without a second API call to translate IDs to names. One request, all the display data ready to go.
11 .workDate(employeeDailyWork.getWorkDate())
Copy the business date as-is. LocalDate serializes cleanly to a "YYYY-MM-DD" string in the response JSON.
12-13 .freshCount(...).rePolishCount(...)
Copy the two counts straight through. Both are Integer on entity and DTO; no transformation needed.

The crucial computed field. Multiplies fresh count by the worker's per-saree rate to compute today's earnings. Notice the formula uses ONLY freshCount, not (fresh + rePolish). 3️⃣ addDailyWork() — Logging Today's Work

EmployeeDailyWorkServiceImpl.addDailyWork()Java
@Override
public EmployeeDailyWorkResponseDto addDailyWork(EmployeeDailyWorkRequestDto requestDto) {
    Employee employee = employeeRepository.findById(requestDto.getEmployeeId())
            .orElseThrow(() -> new ResourceNotFoundException("Employee not found"));

    EmployeeDailyWork employeeDailyWork = EmployeeDailyWork.builder()
            .employee(employee)
            .workDate(requestDto.getWorkDate())
            .freshCount(requestDto.getFreshCount())
            .rePolishCount(requestDto.getRePolishCount())
            .build();

    EmployeeDailyWork saved = employeeDailyWorkRepository.save(employeeDailyWork);
    double  todayEarning = saved.getFreshCount() * employee.getPolishRate();

    return EmployeeDailyWorkResponseDto.builder()
            .workId(saved.getId())
            .employeeId(employee.getId())
            .employeeName(employee.getName())
            .workDate(saved.getWorkDate())
            .freshCount(saved.getFreshCount())
            .rePolishCount(saved.getRePolishCount())
            .todayRevenue(todayEarning)
            .build();
}

Line-by-line walkthrough

1 @Override
Tells the compiler this method overrides one declared in the interface. If we ever rename the method or change its signature without updating the interface, compilation fails — a safety net against typos. Strictly optional but always recommended.
2 public EmployeeDailyWorkResponseDto addDailyWork(EmployeeDailyWorkRequestDto requestDto) {
The method signature. By the time this runs, Spring has already deserialized the JSON body into requestDto, and the controller's @Valid has verified all four required fields. So we can trust the DTO is structurally valid.
3-4 Employee employee = employeeRepository.findById(requestDto.getEmployeeId())
   .orElseThrow(() -> new ResourceNotFoundException("Employee not found"));
Step 1: verify the employee exists. findById returns Optional<Employee> — present if found, empty if not. .orElseThrow(...) means: if empty, throw the custom exception. The exception is caught by GlobalExceptionHandler and returned as HTTP 404. Without this guard, line 6 below would NPE on a missing worker.
5 EmployeeDailyWork employeeDailyWork = EmployeeDailyWork.builder()
Step 2: build the entity. Start the Lombok builder for the entity. This is just an in-memory object so far — no DB write yet.
6 .employee(employee)
Set the @ManyToOne relationship. Pass the full Employee object we just loaded. Hibernate will use its ID to populate the employee_id foreign key column during INSERT.
7 .workDate(requestDto.getWorkDate())
Copy the business date from the request. This is when the work was DONE — not when the row is being inserted. Dad might log Friday's work on Saturday morning; workDate stores Friday's date so reports remain accurate.
8-9 .freshCount(requestDto.getFreshCount()).rePolishCount(requestDto.getRePolishCount())
Copy the two integer counts. Validation has already ensured both are non-negative (the DTO uses @PositiveOrZero). Zero is allowed because a worker might have done only one kind of polish on a given day.
10 .build();
Finalise. We now hold a fully-constructed but unsaved entity. Its id is still null — Hibernate assigns it on insert.
11 EmployeeDailyWork saved = employeeDailyWorkRepository.save(employeeDailyWork);
Step 3: persist the row. Spring Data delegates to Hibernate, which builds INSERT INTO employee_daily_work (...) VALUES (...) and executes it. This is where the UNIQUE constraint on (employee_id, work_date) is tested: a duplicate triggers a DataIntegrityViolationException, translated by GlobalExceptionHandler into "A daily work entry already exists" (HTTP 409). The @PrePersist hook on the entity also stamps created_at and updated_at. The returned saved carries the DB-assigned id.
12 double todayEarning = saved.getFreshCount() * employee.getPolishRate();
Step 4: compute today's earnings. Multiply fresh count by the worker's per-saree rate. employee.getPolishRate() reads from the already-loaded Employee — no extra DB query. Same quirk as the mapper: only freshCount is in the product, not fresh + rePolish. The local variable is named todayEarning, but the response DTO field is named todayRevenue — a mild naming inconsistency between this method and the list mapper.
13-21 return EmployeeDailyWorkResponseDto.builder().workId(...)....build();
Step 5: build and return the response. Eight chained builder calls. workId from the saved row's ID, employeeId and employeeName from the loaded employee (flattened denormalization), workDate/freshCount/rePolishCount from the saved row, and finally todayRevenue = the value we just computed. Spring serializes the result to JSON.

1 Long employeeId = CurrentUserUtil.getEmployeeId();
Resolve the caller's identity. CurrentUserUtil (Chapter 23) reads the JWT claims from Spring's SecurityContextHolder and returns the employeeId claim — a Long for workers, null for admins (admins don't have an employee record). The key guarantee: the caller cannot lie about this — it comes from the cryptographically signed JWT, not from a request parameter.

Guard against non-worker callers. SecurityConfig allows admins to call this endpoint too, but admins have no employee record so their JWT has employeeId = null. Throwing here surfaces as a 500 via GlobalExceptionHandler — a stricter implementation would throw Spring Security's AccessDeniedException for a cleaner 403, but the current code uses a plain RuntimeException.

Same PageRequest construction as the admin method. Zero-based page, size, direction parsed from the DTO's string field, and sort column name.

The key difference from getDailyWorkList. Instead of findAll, we call findByEmployeeId with the JWT-derived ID. Same entity → DTO mapping pattern as the admin list. The helper computes todayEarning per row using the worker's polish rate.

Same response-wrapping pattern. Notably, totalElements is now the worker's total row count across all pages, NOT the workshop-wide total — because the WHERE clause was applied before the COUNT.

6️⃣ getEmployeeWorkSummary() — The Aggregation

EmployeeDailyWorkServiceImpl.getEmployeeWorkSummary()Java
@Override
public EmployeeWorkSummaryDto getEmployeeWorkSummary(Long employeeId, DateRangeDto dateRangeDto) {
    String role = CurrentUserUtil.getRole();
    if (!"ADMIN".equals(role)) {
        Long caller = CurrentUserUtil.getEmployeeId();
        if (caller == null || !caller.equals(employeeId)) {
            throw new org.springframework.security.access.AccessDeniedException(
                    "You can only view your own work records");
        }
    }
    Employee employee = employeeRepository.findById(employeeId)
            .orElseThrow(() -> new ResourceNotFoundException("Employee not found"));

    List<EmployeeDailyWork> works = employeeDailyWorkRepository
            .findByEmployeeIdAndWorkDateBetween(employeeId,
                    dateRangeDto.getFromDate(), dateRangeDto.getToDate());

    long totalWorkDays = works.size();
    long totalFresh = works.stream()
            .mapToLong(EmployeeDailyWork::getFreshCount)
            .sum();

    long totalRePolish = works.stream()
            .mapToLong(EmployeeDailyWork::getRePolishCount)
            .sum();

    double totalRevenue = totalFresh * employee.getPolishRate();
    return EmployeeWorkSummaryDto.builder()
            .employeeId(employeeId)
            .employeeName(employee.getName())
            .totalWorkDays(totalWorkDays)
            .totalFreshCount(totalFresh)
            .totalRePolishCount(totalRePolish)
            .totalRevenue(totalRevenue)
            .build();
}

Line-by-line walkthrough

1 String role = CurrentUserUtil.getRole();
Step 1: read the caller's role from the JWT. Returns "ADMIN", "WORKER", or "SAREE_OWNER". We need this to decide whether to enforce the self-only check below.
2 if (!"ADMIN".equals(role)) {
Skip the self-check for admins. Admins can request any worker's summary. Note the safe ordering: "ADMIN".equals(role), not role.equals("ADMIN") — calling .equals on the literal string means it won't NPE even if role is somehow null.
3 Long caller = CurrentUserUtil.getEmployeeId();
For non-admin callers, read the caller's own employee ID from the JWT. This is the worker's own row in the employees table.
4-7 if (caller == null || !caller.equals(employeeId)) { throw new AccessDeniedException(...); }
The self-or-admin check. Reject if (a) the JWT has no employeeId (defensive — shouldn't happen for a WORKER) OR (b) the requested employeeId doesn't match the caller's own. AccessDeniedException is Spring Security's standard exception — its filter chain converts it to HTTP 403 Forbidden. SecurityConfig allows WORKERs to call this URL; the in-service check narrows access to "self only".
8-9 Employee employee = employeeRepository.findById(employeeId).orElseThrow(...);
Step 2: verify the employee exists. Even after passing the permission check, we still confirm the row exists. Defensive against an admin requesting a deleted or nonexistent ID. Throws ResourceNotFoundException → 404.
10-12 List<EmployeeDailyWork> works = repo.findByEmployeeIdAndWorkDateBetween(employeeId, fromDate, toDate);
Step 3: fetch all rows in the range. One SQL: SELECT * FROM employee_daily_work WHERE employee_id = ? AND work_date BETWEEN ? AND ?. For realistic ranges (a week or a month), this returns at most ~30 rows — safe to process in memory.
13 long totalWorkDays = works.size();
Step 4a: count the working days. Each row represents one work day (the UNIQUE constraint on the entity guarantees no duplicates per (employee, date)), so .size() is the number of distinct days. Cast to long to match the DTO field type.
14-16 long totalFresh = works.stream().mapToLong(EmployeeDailyWork::getFreshCount).sum();
Step 4b: sum the fresh counts. .stream() opens a stream of entities. .mapToLong(...) extracts the integer count from each and returns a primitive LongStream — no autoboxing overhead. The method reference EmployeeDailyWork::getFreshCount is shorthand for w -> w.getFreshCount(). Finally .sum() is a built-in terminal operation returning a primitive long.
17-19 long totalRePolish = works.stream().mapToLong(EmployeeDailyWork::getRePolishCount).sum();
Same pattern, summing the other count field. Could theoretically be combined with the fresh-sum into one stream pass for efficiency, but with realistic small lists (~30 rows) the duplication is harmless and the code reads more clearly.
20 double totalRevenue = totalFresh * employee.getPolishRate();
Step 5: compute the total revenue. totalFresh is a long; polishRate is a Double. Java auto-widens the long to double so the multiplication produces a double. The same quirk yet again: only fresh count is in the product. If your business says re-polish should also earn, this is the third location to update for consistency.
21-28 return EmployeeWorkSummaryDto.builder()...build();
Step 6: build and return the summary DTO. Six fields: employeeId (the requested ID), employeeName (flattened from the loaded Employee), the three computed totals (totalWorkDays, totalFreshCount, totalRePolishCount), and totalRevenue. We use employee.getName() directly — no separate name query needed because we already have the Employee in memory.

🎬 End-to-End Scenario — Dad logs Prema's Friday work

The full trace through the service
  1. Frontend POSTs { "employeeId": 5, "workDate": "2026-05-15", "freshCount": 10, "rePolishCount": 2 } to /api/employee-daily-working.
  2. Spring deserializes the JSON into a EmployeeDailyWorkRequestDto. @Valid passes (all fields present, both counts ≥ 0).
  3. Controller calls service.addDailyWork(requestDto).
  4. Step 1: findById(5) returns Prema's Employee row. Her polishRate = 30.0.
  5. Step 2: Build an EmployeeDailyWork entity with employee=Prema, workDate=2026-05-15, freshCount=10, rePolishCount=2.
  6. Step 3: save(entity). Hibernate runs INSERT INTO employee_daily_work (employee_id, work_date, fresh_saree_count, repolish_saree_count, created_at, updated_at) VALUES (5, '2026-05-15', 10, 2, NOW(), NOW()). The new work_id is, say, 89.
  7. Step 4: todayEarning = 10 * 30.0 = 300.0.
  8. Step 5: Build the response DTO: { workId: 89, employeeId: 5, employeeName: "Prema", workDate: "2026-05-15", freshCount: 10, rePolishCount: 2, todayRevenue: 300.0 }.
  9. Spring serializes to JSON. Controller returns HTTP 200. Frontend updates: "Prema today: 10 + 2 sarees · ₹300".
Common mistakes
Forgetting to load the Employee before saving
The entity's @ManyToOne field needs a managed Employee instance, not just an ID. findById ensures Hibernate has the right reference and that the worker actually exists.
Computing earnings as (freshCount + rePolishCount) × rate in only one place
If you change the formula, update all three locations together: mapToListDto (line 14-17 of the helper), addDailyWork (line 12), and getEmployeeWorkSummary (line 20). They must agree or the responses will disagree with the summaries.
Throwing RuntimeException("Access denied") for null employeeId
A cleaner version throws Spring's AccessDeniedException so the filter chain maps it to HTTP 403, not a 500. The current NOOL code uses RuntimeException — slightly less precise but functional.
Skipping @Valid on the controller parameter
Without it, the DTO's @PositiveOrZero validation doesn't fire, and negative counts could slip in — distorting earnings.
Interview questions
Why doesn't addDailyWork have @Transactional?
Spring Data wraps individual save() calls in their own transaction automatically. @Transactional is needed only when one method coordinates MULTIPLE writes that must commit or roll back together — like EmployeeServiceImpl.createEmployee (Chapter 30), which inserts an Employee, a User, and a UserProfile all together.
What if two requests log Prema for the same day simultaneously?
Both reach save(). The first INSERT succeeds. The second hits the UNIQUE constraint on (employee_id, work_date) and throws DataIntegrityViolationException. GlobalExceptionHandler translates it into the friendly "A daily work entry already exists" message. No data corruption.
Why mapToLong instead of map + reduce?
mapToLong returns a primitive LongStream with .sum() built in — no autoboxing of every value, faster, cleaner. The map + reduce alternative would need .reduce(0L, Long::sum) and box every element.
Why is the formula only freshCount × rate?
Deliberate (or possibly inherited) business rule. The interpretation: re-polish is rework on already-paid sarees, so the workshop doesn't earn fresh revenue from them. Workers may still be paid for re-polish through some other mechanism. If this interpretation is wrong, update all three multiplication sites consistently.
Why does getMyWorkSummary delegate to getEmployeeWorkSummary?
DRY. The calculation lives in one place. The "my" variant just resolves the caller's ID from the JWT and forwards. The permission check inside getEmployeeWorkSummary trivially passes when calling for self.
If a worker has zero rows in the date range, what does the summary return?
findByEmployeeIdAndWorkDateBetween returns an empty list. works.size() is 0. Both stream sums return 0. totalRevenue = 0 × rate = 0.0. The DTO is built with all zeros — no nulls, no exceptions. A clean degenerate case.
