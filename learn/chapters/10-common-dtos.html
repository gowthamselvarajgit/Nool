Common DTOs —
Pagination, dates, generic wrappers

Across NOOL's backend, many endpoints share the same input/output shapes — "give me page 2 of 20 employees", "summarize my work between these two dates", etc. Rather than copy-paste the same fields into every controller, we define them once in five small DTOs and reuse them everywhere.

🤔 Quick Refresher — What's a DTO?

DTO stands for Data Transfer Object. It's a plain class whose only job is to carry data between layers (controller ↔ service, frontend ↔ backend), with optional validation annotations on the fields.

EntityDTO
What it representsA database tableThe shape of data sent over HTTP
Annotations@Entity, @Column, @ManyToOne@NotNull, @Min, @Pattern
Has DB relationships?Yes (lazy/eager loading, FKs)No — just plain fields
Used byRepository + Service layersController layer + view/frontend
Why we separate them: if the frontend posts JSON directly into an entity, it can accidentally set fields it shouldn't (like id or createdAt). DTOs are a safety boundary — they expose only what the API meant to expose.

1️⃣ ApiResponseDto<T> — The Generic Wrapper

dto/common/ApiResponseDto.javaJava
package com.nool.backend.dto.common;

import lombok.AllArgsConstructor;
import lombok.Builder;
import lombok.Data;
import lombok.NoArgsConstructor;

@Data
@Builder
@NoArgsConstructor
@AllArgsConstructor
public class ApiResponseDto<T> {       // ★ generic — T is the actual payload type
    private boolean success;             // true = OK, false = error
    private String  message;             // human-readable status text
    private T       data;                // the actual response payload
}

How the generic <T> works

The <T> is a type parameter. You decide what T means when you use the class. Examples:

Using ApiResponseDto with different T typesJava
// T = Long — return a single ID
ApiResponseDto<Long> response1 = ApiResponseDto.<Long>builder()
    .success(true)
    .message("Employee created")
    .data(42L)
    .build();

// T = EmployeeResponseDto — return one employee object
ApiResponseDto<EmployeeResponseDto> response2 = ApiResponseDto.<EmployeeResponseDto>builder()
    .success(true)
    .message("Found")
    .data(employeeDto)
    .build();

// T = List<String> — return a list of strings
ApiResponseDto<List<String>> response3 = ApiResponseDto.<List<String>>builder()
    .success(true)
    .message("OK")
    .data(List.of("alpha", "beta"))
    .build();

JSON shape

What the frontend seesJSON
{
  "success": true,
  "message": "Employee created",
  "data":    42
}

Used by: mainly the auth and admin user services where the response is a simple confirmation. Other endpoints (like inventory, attendance) return the data object directly without the wrapper — Spring serializes whatever the controller returns. Both styles coexist in NOOL.

2️⃣ PaginationRequestDto — "Give me page N of M"

dto/common/PaginationRequestDto.javaJava
package com.nool.backend.dto.common;

import jakarta.validation.constraints.Min;
import jakarta.validation.constraints.NotBlank;
import lombok.Data;

@Data     // Lombok: getters, setters, toString, equals, hashCode
public class PaginationRequestDto {

    @Min(value = 0, message = "Page number must be zero or greater")
    private int page = 0;             // page index, ZERO-based (page 0 = first page)

    @Min(value = 1, message = "Page size must be at least 1")
    private int size = 10;            // how many rows per page (default 10)

    private String sortBy = "createdAt";       // which column to sort by
    private String sortingDirection = "DESC";  // "ASC" or "DESC"
    private String searchKeyword = "";        // optional search text
}

How the frontend uses it

Almost every list-fetching endpoint in NOOL accepts this DTO. For example, the employeeService.getList() in api.js sends a body like:

Sample POST /api/employees/list request bodyJSON
{
  "page": 2,
  "size": 10,
  "sortBy": "createdAt",
  "sortingDirection": "DESC",
  "searchKeyword": "Prema"
}

The backend uses these values to build a Spring Data Pageable: PageRequest.of(2, 10, Sort.by(DESC, "createdAt")). Spring Data then generates SQL like: 

SELECT * FROM employees
WHERE  employee_name LIKE '%Prema%'
ORDER BY created_at DESC
LIMIT 10 OFFSET 20;  -- page 2 × size 10 = skip 20

Why zero-indexed pages?

Spring Data's convention: page 0 = first page. So "show me the first page" → page=0, "second page" → page=1. The React UI shows them as "Page 1", "Page 2" to users, but converts to 0-based when posting. A common +1/-1 trip.

3️⃣ PaginationResponseDto<T> — "Here's page N of M"

dto/common/PaginationResponseDto.javaJava
package com.nool.backend.dto.common;

import lombok.*;
import java.util.List;

@Data
@Builder
@NoArgsConstructor
@AllArgsConstructor
public class PaginationResponseDto<T> {     // ★ generic again — T is the row type
    private List<T> content;                  // the actual rows of this page
    private int     page;                     // which page this is (0-based)
    private int     size;                     // page size used
    private long    totalElements;            // total rows across ALL pages
    private int     totalPages;               // total page count
    private boolean last;                     // is this the last page?
}

JSON shape

Sample response: 47 employees total, 10 per page, showing page 2JSON
{
  "content": [
    { "employeeId": 21, "employeeName": "Prema", ... },
    { "employeeId": 22, "employeeName": "Ravi", ... },
    // ... 8 more rows
  ],
  "page":          2,
  "size":          10,
  "totalElements": 47,
  "totalPages":    5,
  "last":          false
}

The frontend uses these numbers to draw the "Page 3 of 5 · 47 results" footer and to enable/disable the Next/Previous buttons (e.g., disable Next if last == true).

Why a custom DTO instead of returning Spring's Page<T> directly?

Spring Data has its own Page<T> class that serializes with many more fields (pageable, sort, numberOfElements, etc.). Returning it directly leaks Spring-specific structure to the frontend and clutters the response. NOOL's DTO exposes only the 6 fields the frontend actually needs — cleaner and contract-stable.

4️⃣ DateRangeDto — "Between these two dates"

dto/common/DateRangeDto.javaJava
package com.nool.backend.dto.common;

import jakarta.validation.constraints.NotNull;
import lombok.Data;
import java.time.LocalDate;

@Data
public class DateRangeDto {

    @NotNull(message = "From date is required")
    private LocalDate fromDate;

    @NotNull(message = "To date is required")
    private LocalDate toDate;
}

Where it's used

Many summary/report endpoints take a date range. For example:

  • POST /api/attendance/employee/{id}/summary — attendance summary for one worker in a range
  • POST /api/salary-payments/summary — what salary was earned vs. paid in a range
  • POST /api/admin/dashboard/revenue — workshop revenue in a range
  • POST /api/inventory/owner/{id}/summary — sarees received/returned in a range

Sample request body:

A date-range queryJSON
{
  "fromDate": "2026-05-01",
  "toDate":   "2026-05-15"
}

Why LocalDate and not LocalDateTime?

Business dates in NOOL are day-precision, not minute-precision. "Worked on May 5" means the whole day, no hour-of-day. LocalDate matches that exactly and avoids timezone complications.

5️⃣ MonthYearRequestDto — "For this month"

dto/common/MonthYearRequestDto.javaJava
package com.nool.backend.dto.common;

import jakarta.validation.constraints.Max;
import jakarta.validation.constraints.Min;
import jakarta.validation.constraints.NotNull;
import lombok.Data;

@Data
public class MonthYearRequestDto {

    @NotNull(message = "Month is required")
    @Min(1)
    @Max(12)
    private Integer month;        // 1 = Jan, 12 = Dec

    @NotNull(message = "Year is required")
    private Integer year;         // e.g., 2026
}

Where it's used

For endpoints that report on a specific calendar month. The main one is POST /api/admin/dashboard/revenue/month — used by the admin dashboard to show "May 2026 revenue".

Sample request:

Asking for May 2026JSON
{
  "month": 5,
  "year":  2026
}

Why not just send a date string?

Sending "2026-05" would require us to write our own parser (since LocalDate needs a day). Two integers are simpler, easier to validate, and easier to reason about (1 ≤ month ≤ 12 is a clean rule). The frontend already has the values as numbers from the date picker — no string conversion needed.

🎬 A Real Pagination Round-Trip

Mom views page 3 of the workers list with search "Pre"
  1. Mom is on Admin Dashboard → Employees. She types "Pre" in the search box and clicks page 3.
  2. Frontend's employeeService.getList(2, 10, "Pre") posts to /api/employees/list:
Request bodyJSON
{
  "page": 2,                    // 0-based — actually the third page
  "size": 10,
  "sortBy": "createdAt",
  "sortingDirection": "DESC",
  "searchKeyword": "Pre"
}
  1. Spring auto-binds the JSON into a PaginationRequestDto instance.
  2. @Valid kicks in: page ≥ 0? size ≥ 1? Both pass.
  3. Service builds a PageRequest, issues a SQL query against employees with LIKE + ORDER BY + LIMIT + OFFSET.
  4. Service maps each Employee entity to an EmployeeResponseDto.
  5. Service builds a PaginationResponseDto<EmployeeResponseDto> with content + totals + last flag.
  6. Response JSON goes back. Frontend draws the table + the pagination footer.
Common DTO mistakes
Sending a JPA entity as the response (e.g., returning Employee directly)
Always map to a DTO first. Entities leak lazy-loaded fields, internal timestamps, and FK relationships that frontend has no business knowing about.
Using 1-based pagination on the backend
Spring Data is 0-based. Match that on the API to avoid converting in two places. The UI can show 1-based labels to users; the API stays 0-based.
Reusing PaginationRequestDto as a path variable / query string
It's designed for JSON POST bodies (@RequestBody). For GET-with-query-params, use individual @RequestParams or a custom DTO.
Forgetting @Valid on the controller method parameter
Without @Valid, the @Min / @NotNull annotations on the DTO don't run. Negative page numbers slip through, sorting crashes. Always annotate @Valid @RequestBody PaginationRequestDto dto.
Defaulting sortBy to a non-existent column
The default is "createdAt" — which every entity in NOOL has. If a specific entity doesn't, the controller must override the default before calling the service, or the SQL will throw at runtime.
Interview questions about common DTOs
Why use generics on ApiResponseDto<T> and PaginationResponseDto<T>?
Generics let one class serve many response shapes safely. Without them, I'd either write ApiResponseEmployee, ApiResponseOwner, etc. (boilerplate explosion), or use Object data (losing type safety). Generics give us both reuse and compile-time checks.
Why is page zero-indexed and not one-indexed?
Spring Data's Pageable is zero-indexed — page 0 is the first page. The backend just follows that convention. The UI converts to 1-indexed when displaying ("Page 1 of 5"), but the wire format stays 0-based to avoid double-conversion bugs.
What does @Data from Lombok generate?
It's a meta-annotation that combines @Getter, @Setter, @ToString, @EqualsAndHashCode, and @RequiredArgsConstructor. For a plain data DTO with no special needs, @Data is the shortest way to get all the standard behaviors.
Could you replace PaginationResponseDto with Spring's Page<T>?
You could — Spring Data's Page serializes to JSON automatically. But it includes extra fields (pageable, sort, empty, first, etc.) that we don't need, and it ties the API contract to Spring's internal class structure. The custom DTO is cleaner and more stable.
How does @NotNull on fromDate get enforced?
Two pieces. (1) The Spring Boot validation starter (in pom.xml) brings in Hibernate Validator, which knows how to evaluate @NotNull. (2) The controller method must annotate the parameter with @Valid. When the frontend posts a request without fromDate, validation fails, the controller method never runs, and MethodArgumentNotValidException is thrown — caught by the GlobalExceptionHandler (Chapter 09) and returned as a 400.
Why have these DTOs in a dto/common/ sub-folder?
To signal that they're cross-cutting — not tied to any one domain. Compare with dto/employee/ (only employee-related DTOs) or dto/payment/ (only payment-related). Putting common DTOs in their own folder makes the import patterns clean and signals intent: "any module can use these".
