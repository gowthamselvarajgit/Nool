application.properties —
The settings panel

Every Spring Boot app has one file where you put all the "knobs and dials" — port number, database connection, JWT secret, timezone. Change a value here, and the whole app behaves differently. Let's understand every line.

⚙️

application.properties

backend/src/main/resources/application.properties
Spring Boot configDB connectionJWTProperties

🤔 What This File Does

application.properties is Spring Boot's configuration file. Spring Boot looks for this file in src/main/resources/ automatically when the app starts. Every property in here is read once at startup and made available to every class in the project.

The format is dead simple: key=value, one per line. Lines starting with # are comments.

Why this file matters: moving sensitive values (database passwords, JWT secrets, ports) out of code and into config means I can run NOOL in different environments (my laptop vs. my parents' workshop computer vs. a cloud server) just by changing this one file. The code stays untouched.

📜 The Full File — Every Line Explained

backend/src/main/resources/application.properties properties 
# ════════════════════════════════════════════════════════════════
# APPLICATION IDENTITY
# ════════════════════════════════════════════════════════════════

spring.application.name=backend
# Internal name for this app. Shows up in logs, metrics, etc.
# Not critical, just helpful when running multiple Spring apps side-by-side.


# ════════════════════════════════════════════════════════════════
# HTTP SERVER
# ════════════════════════════════════════════════════════════════

server.port = 8083
# The port Tomcat listens on. So http://localhost:8083 is where the API lives.
# I picked 8083 instead of the default 8080 because port 8080 is too commonly
# used (Jenkins, Tomcat, dev tools — collisions everywhere). 8083 is rare enough
# to avoid clashes on my parents' computer.

server.servlet.context-path=/api
# Prefix every URL with /api. So:
#   - Login   = POST http://localhost:8083/api/auth/login
#   - Inventory = POST http://localhost:8083/api/inventory/receipt
# Why? When the backend and a static frontend share the same domain in production,
# /api/* gets routed to the backend and everything else to the frontend.


# ════════════════════════════════════════════════════════════════
# JWT (JSON Web Token) SETTINGS
# ════════════════════════════════════════════════════════════════

jwt.secret=NOOL_BACKEND_AUTH_SECRET_KEY_FOR_LOCAL_DEVELOPMENT_USE_ONLY_2026
# ★ The cryptographic secret used to sign every JWT.
#
# IMPORTANT THINGS ABOUT THIS:
#   1. Must be at least 256 bits (32 chars) for HS256 algorithm — it is.
#   2. If anyone gets this string, they can forge valid tokens. Treat it like
#      a password.
#   3. For local development this hard-coded value is fine. For production,
#      override via environment variable or a secrets manager (NEVER commit
#      a production secret to git).
#   4. If you change this string, EVERY existing JWT becomes invalid instantly.
#      All logged-in users get kicked out — they have to log in again.
#      That's actually a useful emergency lever if you ever suspect a leak.

jwt.expiration=86400000
# Token lifetime in milliseconds.
#   86,400,000 ms = 86,400 seconds = 1,440 minutes = 24 hours
#
# So once you log in, you stay logged in for 24 hours. After that, you have
# to log in again. This is a trade-off:
#   - Shorter (e.g. 15 min)  → more secure, more annoying
#   - Longer  (e.g. 30 days) → less secure if a token leaks, less annoying
# 24 hours is a comfortable middle ground for an internal business app.


# ════════════════════════════════════════════════════════════════
# SEED ADMIN — auto-created on first startup
# ════════════════════════════════════════════════════════════════

admin.mobile=9876543210
admin.password=Admin@123
# When NOOL boots for the very first time and the database is empty,
# AdminBootstrapRunner.java reads these two values and inserts an ADMIN
# user with this mobile + password. Otherwise nobody could log in.
#
# IMPORTANT: change the password in production! The seed admin password
# is intentionally simple and well-known so I can quickly log in during
# development. A real deployment must override admin.password before
# the first run.


# ════════════════════════════════════════════════════════════════
# BUSINESS DEFAULTS
# ════════════════════════════════════════════════════════════════

# Rate charged to saree owners per returned saree (in INR).
# Used by owner payment summary to compute total payable amount.
nool.rate-per-saree=70.0
# ★ FALLBACK rate per saree. Used only when an owner has NO custom rate set.
#
# Most owners (like Rahul) have their own negotiated rate stored in the
# saree_owner table. For new owners or test data without a rate, this ₹70
# acts as the default.
#
# It's also a knob to centrally bump prices across all "rate-less" owners
# without touching code.


# ════════════════════════════════════════════════════════════════
# DATABASE CONNECTION
# ════════════════════════════════════════════════════════════════

spring.datasource.url=jdbc:mysql://localhost:3306/nool_db?createDatabaseIfNotExist=true&useSSL=false&allowPublicKeyRetrieval=true&serverTimezone=Asia/Kolkata
# ★ The MySQL connection string. Let's break it down piece by piece:
#
#   jdbc:mysql://          → use the MySQL JDBC driver
#   localhost:3306         → MySQL is running on this same machine, port 3306
#   /nool_db               → connect to a database named "nool_db"
#
# Query parameters:
#   createDatabaseIfNotExist=true → if nool_db doesn't exist yet, MySQL creates
#                                    it automatically. Means I don't have to
#                                    pre-create the schema on a new computer.
#   useSSL=false                  → for local dev only — no encrypted connection.
#                                    For production, this MUST be true with proper
#                                    cert setup.
#   allowPublicKeyRetrieval=true  → needed by MySQL 8's default auth plugin to
#                                    let JDBC fetch the server's public key.
#   serverTimezone=Asia/Kolkata   → makes MySQL interpret DATETIME values in IST.
#                                    Critical to avoid the "off-by-one-day" bug
#                                    you'll see again later (Chapter 61).

spring.datasource.username=root
spring.datasource.password=root
# MySQL credentials. "root/root" is the universal local-dev convention.
# In production these become a dedicated user with limited privileges and
# a strong password — definitely not "root".

spring.datasource.driver-class-name=com.mysql.cj.jdbc.Driver
# Tells Spring which JDBC driver class to load. com.mysql.cj.* is MySQL 8's
# new driver namespace (vs. com.mysql.jdbc.* for MySQL 5.x).
# Strictly speaking, Spring Boot can auto-detect this from the URL, but
# specifying it explicitly avoids confusion if multiple drivers are on
# the classpath.


# ════════════════════════════════════════════════════════════════
# JPA / HIBERNATE BEHAVIOR
# ════════════════════════════════════════════════════════════════

spring.jpa.hibernate.ddl-auto=update
# ★ CRITICAL SETTING. Controls how Hibernate manages the database schema.
#
# Possible values:
#   create     → DROP all tables and CREATE fresh ones every startup. Loses data.
#   create-drop → CREATE on startup, DROP on shutdown. For tests only.
#   update     → Compare entities to current schema, ALTER TABLE to match. Keeps data.
#   validate   → Just check entities match schema, don't change anything.
#   none       → Hibernate doesn't touch the schema at all.
#
# We use "update" because:
#   - On a fresh DB: Hibernate creates all tables for us — no manual CREATE TABLE
#   - When I add a new field to an entity: Hibernate ALTERs the table on next startup
#   - Existing data is preserved
#
# WARNING: "update" is convenient for development but RISKY in production.
# It can sometimes generate destructive ALTERs (drop a column you renamed).
# For production, switch to "validate" or "none" and manage schema changes
# explicitly with Flyway or Liquibase.

spring.jpa.show-sql=true
# Print every SQL statement Hibernate generates to the console.
# Insanely useful for debugging: you can see EXACTLY what SQL runs for each
# repository call. In production, turn this off — it floods the logs.

spring.jpa.properties.hibernate.format_sql=true
# Pretty-print those SQL statements with line breaks and indentation.
# Without this, you'd see one huge unreadable single-line query.

spring.jpa.properties.hibernate.dialect=org.hibernate.dialect.MySQLDialect
# Tell Hibernate which "flavor" of SQL to generate. MySQLDialect knows
# MySQL-specific quirks (LIMIT syntax, AUTO_INCREMENT, etc.). Hibernate
# can usually auto-detect from the URL, but pinning it is safer.


# ════════════════════════════════════════════════════════════════
# TIMEZONE — Indian Standard Time, EVERYWHERE
# ════════════════════════════════════════════════════════════════

spring.jpa.properties.hibernate.jdbc.time_zone=Asia/Kolkata
# When Hibernate writes/reads DATETIME values to/from MySQL, treat them
# as Asia/Kolkata (IST, UTC+5:30) — not the server's default timezone.
# Without this, if the server happens to run in UTC, your "May 1 at midnight
# IST" gets stored as "April 30 at 18:30 UTC" — off-by-one-day chaos.

spring.jackson.time-zone=Asia/Kolkata
# When Jackson (the JSON library) serializes LocalDateTime to JSON for the
# frontend, format it in IST too. Together with the previous line, this means
# IST is the ONE timezone throughout the system — no conversions ever.


# ════════════════════════════════════════════════════════════════
# LOGGING
# ════════════════════════════════════════════════════════════════

logging.level.org.hibernate.SQL=debug
# Log every SQL statement at DEBUG level (in addition to show-sql above).
# These two settings overlap slightly; together they give the cleanest output.

logging.level.org.hibernate.type.descriptor.sql.BasicBinder=trace
# Log the actual VALUES being bound to SQL parameters. With this on, you
# see not just "SELECT * FROM users WHERE mobile = ?" but also "binding
# parameter [1] as [VARCHAR] - [9876543210]". Hugely useful for debugging.


# ════════════════════════════════════════════════════════════════
# 404 HANDLING
# ════════════════════════════════════════════════════════════════

spring.mvc.throw-exception-if-no-handler-found=true
spring.web.resources.add-mappings=false
# Together these two settings make Spring Boot throw a NoHandlerFoundException
# when someone requests a URL that doesn't map to any controller. Why this matters:
#
# By default, Spring Boot's catch-all would return a generic HTML error page —
# but our API is JSON-only. We want unknown paths to return JSON like
# { "message": "Not Found", "path": "/api/foo" }, which our GlobalExceptionHandler
# does by catching that NoHandlerFoundException.
#
# add-mappings=false tells Spring Boot to NOT auto-serve static files from
# the classpath. We don't have any (the frontend is in a separate folder).

🔌 How These Settings Reach the Code

Java classes don't read this file directly. Spring Boot reads it at startup and exposes values through two mechanisms:

Mechanism 1: Spring Boot auto-configures itself

All spring.* properties are read by Spring Boot's auto-configuration. They affect how Spring builds the app:

  • server.port → tells Tomcat what port to bind
  • spring.datasource.url/username/password → tells Spring how to make a connection pool
  • spring.jpa.* → configures Hibernate

We don't write code to consume these — they "just work".

Mechanism 2: @Value for our custom properties

For NOOL-specific properties (jwt.secret, admin.mobile, nool.rate-per-saree), we explicitly inject them into Java classes using the @Value annotation:

Inside JwtUtil.java — reading jwt.secret and jwt.expiration Java 
@Component
public class JwtUtil {

    @Value("${jwt.secret}")         // reads from application.properties
    private String secret;

    @Value("${jwt.expiration}")
    private long expiration;

    // ... uses `secret` and `expiration` to sign / verify tokens
}

At startup, Spring sees @Value("${jwt.secret}") and substitutes in the actual string from the properties file. By the time any controller method runs, those fields are already populated. No file I/O in business logic.

Scenario — what console output looks like with these settings

With show-sql=true and BasicBinder=trace, when Rahul logs in, the backend console looks like this:

🏭 What Changes for Production?

The values in this file are tuned for local development. For a production deployment, you'd override several of them. Standard practice in Spring Boot is to use environment variables that take precedence over the file:

SettingLocal-dev valueProduction override
jwt.secretHard-coded string$JWT_SECRET env var, set via secrets manager. Long random string.
admin.passwordAdmin@123$ADMIN_PASSWORD — strong unique password
spring.datasource.urllocalhost:3306Managed MySQL host (RDS, etc.) over SSL
spring.datasource.passwordrootGenerated strong password in secrets vault
spring.jpa.hibernate.ddl-autoupdatevalidate or none + Flyway/Liquibase migrations
spring.jpa.show-sqltruefalse (saves logs from drowning)
logging.level.*debug/traceinfo (or warn) for less noise

You don't need to maintain two files. Spring Boot supports environment-specific overrides via application-prod.properties alongside the base file, activated with --spring.profiles.active=prod. NOOL doesn't use profiles yet because there's only one deployment (my parents' computer), but it's a natural next step.

Common application.properties mistakes
Committing a production JWT secret to git
Use a placeholder in the file and override via environment variable in production. NEVER let a real secret end up in git history — even if you "remove it later", it's still in the commits.
Leaving ddl-auto=update in production
A small entity change (renaming a field) can cause Hibernate to DROP + CREATE the column, wiping data. Production must use validate with explicit migrations.
Forgetting serverTimezone=Asia/Kolkata in the JDBC URL
If the MySQL server's default timezone differs from your app's, dates drift by hours. Always pin both the JDBC URL's serverTimezone AND the JPA property hibernate.jdbc.time_zone.
Putting comments inline on the same line as a value
jwt.expiration=86400000 # 24 hours doesn't work — the comment becomes part of the value, causing parse errors. Use a separate line starting with #.
Hard-coding values that should be configurable (like the saree rate) directly in service code
If a value might change between environments, put it in application.properties and inject with @Value. Code changes need redeploys; properties changes just need a restart.
Interview questions about Spring Boot configuration
Where does Spring Boot look for application.properties?
By default, in src/main/resources/application.properties (inside the JAR at runtime). It also checks the current directory, ./config/, and a few other classpath locations. Profile-specific files like application-prod.properties are layered on top when the matching profile is active.
How do you override these values without editing the file?
Three common ways: (1) environment variables — Spring Boot maps JWT_SECRET env var to jwt.secret property automatically; (2) command-line args — java -jar app.jar --server.port=9090; (3) external application.properties placed next to the JAR. Precedence order: command-line > env vars > external file > bundled file.
What's the difference between application.properties and application.yml?
Same purpose, different syntax. YAML is more compact for nested settings (spring: jpa: show-sql: true instead of three dotted lines). Both are supported. NOOL uses .properties because it's simpler and the project is small.
Why is ddl-auto=update dangerous in production?
Hibernate compares your entities against the live schema and tries to ALTER it to match. If you rename a field in Java, Hibernate doesn't know it's a rename — it just sees "old column gone, new column appeared". Depending on version, it might DROP the old column (and its data) before adding the new one. Schema changes should be deliberate; use a real migration tool (Flyway/Liquibase) instead.
Could the JWT secret be loaded from a file at runtime instead?
Yes. Spring Boot supports spring.config.import=file:/etc/nool/secrets.properties, which loads a second properties file from disk and merges it. Or use a secrets-manager client (AWS Secrets Manager, HashiCorp Vault) that exposes values as environment variables. The simplest production pattern is env vars set by the orchestrator (k8s, systemd, docker-compose).
What does spring.jpa.show-sql=true actually do at the JVM level?
It enables a Hibernate log appender that prints generated SQL to stdout. It's a convenience wrapper around configuring SLF4J. The slight subtlety is that show-sql uses a simpler formatter than logging.level.org.hibernate.SQL=debug — using both gives you nicely formatted output PLUS structured log entries. For parameter values, you separately need BasicBinder=trace.
