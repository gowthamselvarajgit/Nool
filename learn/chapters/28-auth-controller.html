Notice how thin this controller is. The login method body is one line: return authService.login(requestDto);. Controllers in NOOL are deliberately thin — receive HTTP, call service, return result. All the brains live in services.

📜 The Full File — Every Line Explained

auth/controller/AuthController.java Java 
package com.nool.backend.auth.controller;

import com.nool.backend.auth.dto.AuthLoginResponseDto;
import com.nool.backend.auth.dto.AuthLoginRequestDto;
import com.nool.backend.auth.service.AuthService;
import jakarta.validation.Valid;
import lombok.RequiredArgsConstructor;
import org.springframework.web.bind.annotation.*;

@RestController                       // every method returns JSON, not a view name
@RequestMapping("/auth")            // every URL prefixed: /api/auth/*
@RequiredArgsConstructor             // constructor injection of `authService`
public class AuthController {

    private final AuthService authService;

    // ── POST /api/auth/login ──────────────────────────────────────
    @PostMapping("/login")
    public AuthLoginResponseDto login(
            @Valid @RequestBody AuthLoginRequestDto requestDto) {

        return authService.login(requestDto);
        // One-liner — all the logic is in AuthServiceImpl (Chapter 25).
    }

    // ── POST /api/auth/logout ─────────────────────────────────────
    @PostMapping("/logout")
    public void logout() {
        // JWT logout handled on client side.
        // The frontend deletes the token from localStorage.
        // Nothing for the server to do.
    }

    // ── GET /api/auth/validate ────────────────────────────────────
    @GetMapping("/validate")
    public void validateToken() {
        // JWT already validated by JwtAuthenticationFilter.
        // If the request reached this method, the token is valid.
        // (Well — but this endpoint is under /auth/** which the filter SKIPS!)
        // See note below.
    }
}

⚙️ The Three Critical Annotations

AnnotationWhat it does
@RestController Combo of @Controller + @ResponseBody. Every method's return value is serialized to JSON via Jackson (no view templates).
@RequestMapping("/auth") Adds /auth in front of every endpoint URL in this class. Combined with the global context-path /api, full URLs become /api/auth/login, etc.
@RequiredArgsConstructor Lombok generates a constructor taking all final fields. Spring uses it for dependency injection — no @Autowired needed.

🔍 The Login Method Annotations

Three more annotations stacked on the login method parameter:

AnnotationEffect
@PostMapping("/login") Method handles only POST requests at /api/auth/login
@RequestBody Tells Spring to deserialize the HTTP request body (JSON) into AuthLoginRequestDto
@Valid Triggers Bean Validation on the DTO — checks @NotBlank, @Size, etc.

⚠️ An Important Subtlety — Why /validate Is Tricky

The /validate endpoint as written has an interesting quirk worth flagging. Looking at JwtAuthenticationFilter (Chapter 21):

From JwtAuthenticationFilterJava
@Override
protected boolean shouldNotFilter(HttpServletRequest request) {
    return request.getRequestURI().startsWith("/api/auth/");
}

The filter skips all /api/auth/* URLs — including /validate! That means the comment "JWT already validated by filter" isn't quite accurate for this endpoint. The filter doesn't run for it.

In practice, the /validate endpoint isn't strictly useful as written — it accepts any request without checking the token. A stricter implementation would either (a) carve /validate out of the shouldNotFilter skip list, or (b) make the controller method actively parse the token. NOOL accepts this because the frontend rarely calls /validate — most validation happens implicitly when any protected endpoint succeeds or fails.

🚪 Why /api/auth/* Is Public

From SecurityConfig (Chapter 22):

SecurityConfig ruleJava
.requestMatchers("/auth/**").permitAll()

Login and logout HAVE to be reachable without a token — otherwise no one could ever log in (chicken and egg). So all /auth/* endpoints are explicitly marked as public. The shouldNotFilter in the JWT filter avoids wasting CPU trying to parse non-existent tokens for these requests.

🎬 Real-World Scenarios

Scenario 1: Rahul logs in
  1. Frontend sends POST http://localhost:8083/api/auth/login with JSON body.
  2. Tomcat receives the request. Spring routes it through the filter chain.
  3. JwtAuthenticationFilter.shouldNotFilter returns true for /api/auth/ → SKIP.
  4. SecurityConfig matches /auth/** → permitAll().
  5. Spring deserializes JSON body into AuthLoginRequestDto.
  6. @Valid validates: mobile present, length 10-15, password present.
  7. Login method runs: return authService.login(requestDto);
  8. Service does its work (Chapter 25), returns AuthLoginResponseDto.
  9. Spring serializes the DTO to JSON, returns HTTP 200.
  10. Frontend receives the JSON, stores the token, navigates to dashboard.
Scenario 2: Empty password — validation rejects
  1. Frontend posts { "mobileNumber": "9876543210", "password": "" }.
  2. Body deserialized into DTO. @Valid kicks in.
  3. @NotBlank on password fails.
  4. Spring throws MethodArgumentNotValidException before login() body executes.
  5. GlobalExceptionHandler.handleValidation catches it, returns HTTP 400 with details.
  6. Service never runs — no wasted DB query.
Scenario 3: Logout
  1. User clicks "Logout" in the sidebar.
  2. Frontend deletes nool_token from localStorage (this is the actual logout).
  3. Optionally, frontend also sends POST /api/auth/logout — a courtesy ping.
  4. Controller's logout() method runs. Body is empty. No-op.
  5. Returns HTTP 200 with no body.
  6. Frontend redirects to /login.
Scenario 4: Wrong password
  1. Frontend posts mobile + wrong password.
  2. Validation passes (both fields are present).
  3. Service is called.
  4. Service throws BadRequestException("Invalid mobile number or password").
  5. GlobalExceptionHandler converts to HTTP 400.
  6. Frontend reads error message, shows it in red below the password field.
Common AuthController mistakes
Putting business logic inside the controller
Controllers should be thin. Move logic to the service. NOOL's controller is one line — perfect.
Forgetting @Valid on the request body
Without it, @NotBlank / @Size on the DTO don't run. Bad input slips through.
Returning the entity (e.g., User) directly
Return a DTO. Entities leak fields like the BCrypt password hash and internal timestamps.
Wrapping the return value in ResponseEntity.ok(...) unnecessarily
Spring auto-converts the DTO return value to 200 OK with JSON body. No need for explicit ResponseEntity unless you need a custom status code.
Adding try/catch blocks inside the controller
Let exceptions propagate. GlobalExceptionHandler catches them centrally and returns clean JSON. Try/catch in controllers leads to inconsistent error responses.
Interview questions
Why three endpoints when only login does anything?
For frontend convenience. /logout exists so the frontend can send a POST after deleting the token (some monitoring tools track logout events). /validate gives the frontend a "check my token" pinger if needed. Both are placeholders that could grow real logic later — e.g., a server-side token blacklist would put the revocation in /logout.
Why is the controller's login method just one line?
Layered architecture. Controllers handle HTTP concerns (parsing requests, returning responses). Services handle business logic. By keeping the controller thin, the same logic could be reused for, say, a future API key auth or a CLI tool — anything that calls authService.login(...) would get the same behavior.
What's the difference between @Controller and @RestController?
@Controller is the traditional MVC controller — returns view names for server-side rendering. @RestController = @Controller + @ResponseBody on every method — returns objects that Jackson serializes to JSON. For REST APIs always use @RestController.
Why does @RequestBody need @Valid?
@RequestBody just tells Spring to deserialize JSON into the parameter. @Valid additionally triggers Bean Validation on that parameter — running all the @NotBlank, @Size, etc. annotations defined on the DTO. Without @Valid, the DTO is populated but validation rules don't run.
What's the full URL for the login endpoint?
http://localhost:8083/api/auth/login. Breaking it down: 8083 is the configured server.port; /api is the configured server.servlet.context-path; /auth from @RequestMapping("/auth") on this class; /login from @PostMapping("/login") on the method.
