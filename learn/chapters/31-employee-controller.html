📜 The Full File — Every Line Explained

controller/employee/EmployeeController.javaJava
package com.nool.backend.controller.employee;

import com.nool.backend.dto.common.PaginationRequestDto;
import com.nool.backend.dto.common.PaginationResponseDto;
import com.nool.backend.dto.employee.*;
import com.nool.backend.service.employee.EmployeeService;
import jakarta.validation.Valid;
import lombok.RequiredArgsConstructor;
import org.springframework.web.bind.annotation.*;

@RestController                       // Spring: JSON-only controller
@RequestMapping("/employees")       // every path: /api/employees/*
@RequiredArgsConstructor             // constructor injection of `employeeService`
public class EmployeeController {

    private final EmployeeService employeeService;

    // ── 1. CREATE WORKER ────────────────────────────────────────
    @PostMapping
    public EmployeeResponseDto createEmployee(
            @Valid @RequestBody CreateEmployeeRequestDto requestDto){
        return employeeService.createEmployee(requestDto);
    }

    // ── 2. UPDATE WORKER ────────────────────────────────────────
    @PutMapping
    public void updateEmployee(
            @Valid @RequestBody UpdateEmployeeRequestDto requestDto){
        employeeService.updateEmployee(requestDto);
    }

    // ── 3. STATUS UPDATE (mark as LEFT) ─────────────────────────
    @PatchMapping("/status")
    public void updateEmployeeStatus(
            @Valid @RequestBody EmployeeStatusUpdateDto requestDto){
        employeeService.updateEmployeeStatus(requestDto);
    }

    // ── 4. GET ONE BY ID ────────────────────────────────────────
    @GetMapping("/{employeeId}")
    public EmployeeResponseDto getEmployeeById(
            @PathVariable Long employeeId){
        return employeeService.getEmployeeById(employeeId);
    }

    // ── 5. PAGINATED LIST ───────────────────────────────────────
    @PostMapping("/list")
    public PaginationResponseDto<EmployeeListResponse> getEmployeeList(
            @RequestBody PaginationRequestDto paginationRequestDto){
        return employeeService.getEmployeeList(paginationRequestDto);
    }

    // ── 6. WORKER'S OWN PROFILE ─────────────────────────────────
    @GetMapping("/me")
    public EmployeeResponseDto getMyProfile(){
        return employeeService.getMyProfile();
    }

    // ── 7. HARD DELETE (rare) ───────────────────────────────────
    @DeleteMapping("/{employeeId}")
    public void deleteEmployee(@PathVariable Long employeeId){
        employeeService.deleteEmployee(employeeId);
    }
}

🤔 Why HTTP Methods Matter — POST vs PUT vs PATCH vs DELETE

NOOL uses different HTTP verbs deliberately:

VerbUsed forNOOL example
POST Create something new, or invoke a non-idempotent action Add new employee. Also used for /list because the search body is too large for query params.
GET Read data — safe, idempotent, no side effects Fetch one by ID, fetch "me"
PUT Replace the whole resource — idempotent Update employee name + rate + mobile (all fields together)
PATCH Update a SUBSET of fields Change only status (LEFT or ACTIVE) without touching anything else
DELETE Remove the resource Hard-delete an employee row
Why POST /list instead of GET? Two reasons. (1) The pagination DTO has many fields (page, size, sortBy, sortingDirection, searchKeyword) — putting them all in query params is ugly. (2) Search keywords might contain special characters that need escaping. A POST body sidesteps both problems. It's a NOOL convention used everywhere: /api/X/list for paginated reads.

⛓️ Why Each Method Is One Line

Look at the bodies:

  • return employeeService.createEmployee(requestDto);
  • employeeService.updateEmployee(requestDto);
  • ...and so on for all seven.

This is intentional. Controllers in NOOL are thin. Their only jobs:

  1. Receive the HTTP request
  2. Parse it into typed DTOs (Spring + @Valid do this)
  3. Delegate to the service
  4. Return the result (Spring serializes to JSON)

No business logic, no validation beyond what's on the DTO, no DB calls. Anything more interesting belongs in the service.

Benefits of thin controllers:
  • Reusability — the same service can be called by a CLI, a scheduled job, or another service. The HTTP layer is just one consumer.
  • Testability — services are easier to unit-test than controllers. If business logic lived in controllers, you'd need MockMvc for every test.
  • Maintainability — bugs are usually in business logic. Knowing they're all in one place (the service) makes them faster to find.

🔍 The Full Request Flow

Let's trace what happens when Dad submits "Add Employee":

Request flow for POST /api/employeesflow
1. Browser → HTTP request:
   POST http://localhost:8083/api/employees
   Authorization: Bearer eyJhbGc...
   Content-Type: application/json
   Body: { "employeeName": "Prema", "mobileNumber": "9000000002", ... }

2. Tomcat receives → routes through filters.

3. JwtAuthenticationFilter:
   - Reads Authorization header.
   - Verifies signature + expiry via JwtUtil.
   - Sets SecurityContext with role=ADMIN, userId=1.

4. SecurityConfig rules:
   - Matches POST /employees → hasAuthority("ADMIN"). Pass.

5. Spring dispatcher routes to EmployeeController.createEmployee:
   - @RequestBody deserializes JSON into CreateEmployeeRequestDto.
   - @Valid runs validation:
     • @NotBlank on employeeName     ✓
     • @NotNull on joiningDate       ✓
     • @Positive on polishingRate    ✓
     • @Pattern on mobileNumber      ✓
     • @NotBlank on password         ✓
   - All pass.

6. Method body: return employeeService.createEmployee(requestDto);

7. Service does its full createEmployee flow (Chapter 30):
   - Dup check, orphan cleanup, insert Employee, insert User+Profile via AdminUserService.

8. Service returns EmployeeResponseDto.

9. Spring auto-serializes the DTO to JSON.

10. HTTP 200 OK with JSON body sent back to browser.

11. Frontend updates the employee list table.

🎬 Real-World Scenarios

Scenario 1: Successful Add Employee
  1. Dad fills the form → submit.
  2. Frontend posts JSON to POST /api/employees.
  3. Filter passes, SecurityConfig passes, validation passes.
  4. Service runs (Chapter 30).
  5. Response: { employeeId: 5, employeeName: "Prema", status: "ACTIVE", ... }.
  6. Frontend appends Prema to the table — instant feedback.
Scenario 2: Validation rejects invalid mobile
  1. Dad accidentally types "12345" as the mobile.
  2. Frontend posts.
  3. @Valid runs @Pattern on mobileNumber → "12345" doesn't match Indian regex.
  4. Spring throws MethodArgumentNotValidException BEFORE the controller method body runs.
  5. GlobalExceptionHandler returns HTTP 400: "mobileNumber: Mobile number must be a valid 10-digit Indian mobile number".
  6. Frontend highlights the field in red. Service is never called — no wasted DB hit.
Scenario 3: Workers can call /me but not /list
  1. Prema (WORKER) calls GET /api/employees/me → SecurityConfig allows for ADMIN or WORKER → returns her profile.
  2. Prema calls POST /api/employees/list → SecurityConfig rule POST /employees/list requires ADMIN → 403.
  3. So workers see their own data via /me but can't browse the full directory.
Scenario 4: Prema tries to GET another worker
  1. Prema calls GET /api/employees/8 (Ravi's ID) — crafted in browser dev tools.
  2. SecurityConfig: GET /employees/** allows ADMIN or WORKER → passes.
  3. Controller invokes getEmployeeById(8).
  4. Service's self-or-admin check (Chapter 30) detects Prema (employeeId=5) doesn't match the requested 8 → AccessDeniedException → 403.
  5. Defense in depth — even though SecurityConfig allowed the URL, the service blocks the misuse.
Scenario 5: Paginated list with search
  1. Admin UI: Dad types "ra" in the search box.
  2. Frontend posts to /api/employees/list with { page: 0, size: 10, sortBy: "createdAt", sortingDirection: "DESC", searchKeyword: "ra" }.
  3. Controller delegates to service.
  4. Service builds PageRequest, calls repo.searchEmployees("ra", pageable).
  5. Repository runs the LOWER(name) LIKE or mobile LIKE query.
  6. Returns Page with Ravi (matches name) and any others matching the pattern.
  7. Service wraps in PaginationResponseDto, returns.
  8. Frontend renders the filtered table.

Common controller mistakes
Adding business logic inside the controller method
Keep controllers thin. Even a single if for "if admin then X else Y" belongs in the service. The controller's job is HTTP routing, not business decisions.
Forgetting @Valid on @RequestBody parameters
Without it, the validation annotations on the DTO don't run. Bad inputs pass through silently. @Valid is what turns the DTO from a "shape" into a "validated shape".
Wrapping the return in ResponseEntity.ok(...) unnecessarily
Spring automatically wraps the return as 200 OK with the body as JSON. Only use ResponseEntity if you need a different status code (e.g., 201 Created) or custom headers.
Try/catch inside controller methods
Let exceptions propagate. GlobalExceptionHandler catches them centrally. Local try/catch leads to inconsistent error responses across endpoints.
Using @RequestParam for complex search bodies
Pagination + search params can be many fields. Use @RequestBody with a DTO (POST /list pattern) — cleaner URLs, easier escaping.
Using GET with a request body
HTTP technically allows it, but most clients (cURL, fetch) discourage it. Use POST when a body is needed.
Interview questions
Why are NOOL's controllers so thin?
To separate concerns. Controllers handle HTTP plumbing (parsing, status codes, serialization). Services handle business logic. This makes the service reusable (other entry points: CLI, scheduler), the controller easier to scan, and bugs easier to locate.
Why POST /list for read endpoints? Isn't GET more "RESTful"?
Strict REST says GET for reads. But NOOL's pagination/search bodies have many fields and special characters. Putting all that in query parameters is messy. POST with a body is a pragmatic NOOL convention. The endpoint is still read-only — it just uses the wrong verb for "purer" REST taste. We accept the trade-off for cleaner URLs.
Why does updateEmployee use PUT but updateStatus uses PATCH?
PUT semantically means "replace the whole resource" — the Update DTO has all the editable fields together. PATCH means "modify a subset" — the Status DTO has just status. Following these conventions makes the API self-documenting: clients reading the spec know what each endpoint expects.
What happens if a worker calls POST /api/employees?
SecurityConfig (Chapter 22) rule POST /employees → hasAuthority("ADMIN") rejects it with 403 Forbidden. The controller method never runs. Even if a worker tried to add themselves through that endpoint, they can't.
Why is getMyProfile a separate endpoint from getEmployeeById?
UX clarity and security. The worker's frontend doesn't need to know its own employeeId — it just calls /me and the backend infers identity from the JWT. This avoids leaking IDs in URLs and prevents requests that could be confused with admin lookups. Cleaner code, less attack surface.
Could you reuse getEmployeeById(currentUserId) for getMyProfile?
You could, but having a separate method makes the intent explicit. getMyProfile() reads better than getEmployeeById(currentUserId). Slightly more code, much clearer purpose.
Why doesn't this controller have any @PreAuthorize annotations?
NOOL centralizes authorization in SecurityConfig's URL rules. Adding method-level @PreAuthorize would duplicate the policy in two places. One source of truth (SecurityConfig) is easier to audit. Method-level annotations would only make sense if individual methods needed dynamically-computed rules.
