AdminUserService & Impl

When Dad adds a new worker or client, NOOL needs to create their login account too. This service is what does that work — hashing the password, inserting the User row, creating the UserProfile bridge, all in one atomic transaction.

@ServiceAccount creation@TransactionalBCrypt

🧠 What This Service Does

Two methods. Both follow the same pattern, just with different roles:

  1. createEmployeeUser(mobile, password, employeeId) — creates a User row with role WORKER, linked via UserProfile to an Employee.
  2. createOwnerUser(mobile, password, ownerId) — creates a User row with role SAREE_OWNER, linked via UserProfile to a SareeOwner.
Important: This service is called by other services — never directly by a controller. When Dad clicks "Add Employee" in the UI, the EmployeeServiceImpl first creates the Employee row, then calls adminUserService.createEmployeeUser(...) to provision the login. Same for adding owners. This service is reusable building block.

1️⃣ The Interface — AdminUserService.java

auth/service/AdminUserService.javaJava
package com.nool.backend.auth.service;

import com.nool.backend.auth.entity.User;

public interface AdminUserService {
    User createEmployeeUser(String mobileNumber, String rawPassword, Long employeeId);
    User createOwnerUser(String mobileNumber, String rawPassword, Long ownerId);
}

Two methods, both returning the saved User. The caller can use the returned User if they need its generated ID for follow-up linking.

2️⃣ The Implementation

auth/service/impl/AdminUserServiceImpl.javaJava
package com.nool.backend.auth.service.impl;

// imports omitted

@Service
@RequiredArgsConstructor
public class AdminUserServiceImpl implements AdminUserService {

    private final UserRepository        userRepository;
    private final UserProfileRepository userProfileRepository;
    private final PasswordEncoder       passwordEncoder;       // BCrypt

    // ── CREATE LOGIN FOR A WORKER ─────────────────────────────────
    @Override
    @Transactional
    public User createEmployeeUser(String mobileNumber, String rawPassword, Long employeeId) {

        // Step 1: prevent duplicate
        if (userRepository.findByMobileNumber(mobileNumber).isPresent()) {
            throw new DuplicateResourceException(
                "A login account with this mobile number already exists");
        }

        // Step 2: create the User row with BCrypt-hashed password
        User user = User.builder()
                .mobileNumber(mobileNumber)
                .password(passwordEncoder.encode(rawPassword))   // ★ HASH HERE
                .role(Role.WORKER)                               // fixed for this method
                .active(true)
                .build();
        User savedUser = userRepository.save(user);              // INSERT, get back the ID

        // Step 3: create the UserProfile bridging to the Employee
        UserProfile profile = UserProfile.builder()
                .user(savedUser)
                .employeeId(employeeId)                          // link to Employee row
                .build();
        userProfileRepository.save(profile);

        return savedUser;
    }

    // ── CREATE LOGIN FOR A SAREE OWNER ────────────────────────────
    @Override
    @Transactional
    public User createOwnerUser(String mobileNumber, String rawPassword, Long ownerId) {

        // Step 1: prevent duplicate
        if (userRepository.findByMobileNumber(mobileNumber).isPresent()) {
            throw new DuplicateResourceException(
                "A login account with this mobile number already exists");
        }

        // Step 2: create the User row
        User user = User.builder()
                .mobileNumber(mobileNumber)
                .password(passwordEncoder.encode(rawPassword))
                .role(Role.SAREE_OWNER)                          // fixed for this method
                .active(true)
                .build();
        User savedUser = userRepository.save(user);

        // Step 3: create the UserProfile bridging to the SareeOwner
        UserProfile userProfile = UserProfile.builder()
                .user(savedUser)
                .ownerId(ownerId)                                // link to SareeOwner row
                .build();
        userProfileRepository.save(userProfile);

        return savedUser;
    }
}

📋 Method-by-Method Walkthrough

StepBoth methods do
Step 1: dup checkIf mobile already exists → throw DuplicateResourceException
Step 2: hash + save UserBCrypt the password, build User with appropriate Role, save → returns row with generated id
Step 3: link UserProfileBuild UserProfile with the saved User + employeeId/ownerId, save
@Transactional commitsBoth INSERTs commit together. If step 3 fails, step 2's User is rolled back.

⚙️ The Critical Role of @Transactional

Notice both methods are annotated with @Transactional. This is essential: the two INSERTs (User + UserProfile) must commit together or fail together.

Without @Transactional, imagine:
  1. Step 2 inserts the User row successfully (user_id = 5).
  2. Step 3 fails (e.g., DB connection drops, or employeeId is invalid).
  3. Without transaction rollback, the User row stays but has no profile.
  4. That worker can now log in but the JWT will have employeeId = null — broken state.
With @Transactional: if step 3 fails, Spring rolls back step 2 too. Either both rows exist or neither does. Atomic.

🎬 Real-World Scenarios

Scenario 1: Dad adds Prema as a worker
  1. Dad fills "Add Employee" form: name "Prema", mobile 9000000002, password prema123, rate ₹30.
  2. EmployeeServiceImpl.create(...) runs:
    • Inserts the Employee row → gets employee_id=5.
    • Calls adminUserService.createEmployeeUser("9000000002", "prema123", 5).
  3. AdminUserService:
    • Dup check passes (no existing user with that mobile).
    • BCrypt-hashes "prema123" → stores 60-char hash.
    • Inserts User row → user_id=3.
    • Inserts UserProfile linking user_id=3 to employee_id=5.
    • Commit.
  4. Prema can now log in with mobile + password.
Scenario 2: Dad tries to add a worker with a duplicate mobile
  1. Dad accidentally types Rahul's mobile (9000000001) when creating a new worker.
  2. AdminUserService step 1 finds the existing User row → throws DuplicateResourceException.
  3. The transaction has just started — nothing has been inserted yet, so nothing to roll back.
  4. GlobalExceptionHandler (Chapter 09) converts to HTTP 409 with message: "A login account with this mobile number already exists".
  5. Frontend shows the error. Dad realises the typo, fixes the mobile, tries again.
Scenario 3: Adding Rahul as a SAREE_OWNER
  1. Dad opens "Add Owner": name "Rahul Textiles", mobile 9000000001, password rahul123, rate ₹65.
  2. SareeOwnerServiceImpl.create(...):
    • Inserts SareeOwner row → owner_id=1.
    • Calls adminUserService.createOwnerUser("9000000001", "rahul123", 1).
  3. Same flow as Prema's case but: role=SAREE_OWNER, profile.ownerId=1 (employeeId=null).
Scenario 4: Mid-transaction failure rollback
  1. Dad clicks Add Employee for Prema.
  2. Step 1 dup-check passes.
  3. Step 2 User insert succeeds (user_id=3).
  4. Step 3: imagine UserProfile insert fails because the database connection dropped.
  5. @Transactional catches the failure, rolls back the User insert too.
  6. DB returns to its pre-call state. No orphan User row remains.
Common mistakes
Forgetting @Transactional
Without it, partial saves can leave orphan rows. ALWAYS annotate methods that do multiple writes.
Calling passwordEncoder.encode() on an already-hashed password
BCrypt-hashing an already-hashed string produces a hash-of-a-hash that can never be verified by a typed password. Make sure callers always pass plain text.
Hard-coding the role inside this service instead of in dedicated methods
Two methods (createEmployeeUser / createOwnerUser) with fixed roles prevents callers from accidentally creating an ADMIN. The role is implied by the method name — safer.
Letting the controller call this service directly
This service is meant to be called by other services (EmployeeServiceImpl, SareeOwnerServiceImpl), AFTER they've created the business entity. Controllers shouldn't bypass that orchestration.
Interview questions
Why two separate methods instead of one parameterized one?
Type safety. createEmployeeUser(mobile, password, employeeId) can only create WORKERs and only links to Employee. A single createUser(mobile, password, role, businessId) would let callers pass mismatched combinations (e.g., role=WORKER with an ownerId). Separate methods make impossible combinations literally uncallable.
Why does @Transactional matter here?
Because we do TWO inserts (User + UserProfile) and they must be atomic. If step 2 succeeds but step 3 fails, we'd have a User without a profile — broken state. The transaction ensures both succeed or both fail together.
Could you make this an admin-only endpoint instead of a service?
Conceptually yes, but in NOOL there's no direct admin endpoint to "create a User in isolation". User accounts are always created together with an Employee or SareeOwner. So this service is more useful as a composable building block called by other services.
What if BCrypt encoding itself throws?
BCrypt's encode() is deterministic and only fails on out-of-memory or invalid input (very rare). If it threw, it would be a RuntimeException — caught by @Transactional, the User insert (if any) would be rolled back, and GlobalExceptionHandler would return 500. The user sees a clean error and can retry.
Why is active = true hard-coded?
Newly-created accounts should always start active — they were just added by an admin. If you want to add a disabled-on-creation flow later (e.g., "pending approval"), you'd add a method param. For now, true is the right default.
