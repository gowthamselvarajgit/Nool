AuthService & AuthServiceImpl

When the login form is submitted, this is the file that decides: "yes, you are who you say you are." It checks the mobile number, verifies the BCrypt-hashed password, builds the JWT, and returns it. The brain of NOOL's login flow.

🔑

🧠 Interface + Implementation Pattern

NOOL follows the classic Spring pattern of interface for contract, class for implementation:

FilePurpose
AuthService.java (interface)The contract — what methods this service offers
AuthServiceImpl.java (class)The actual implementation — the body of those methods
Why split into two files? So tests can mock the interface without caring about the implementation, so we can swap implementations (e.g., a fake auth for integration tests), and so the interface acts as documentation — you can see at a glance what the service offers without reading the body.

1️⃣ The Interface — AuthService.java

auth/service/AuthService.javaJava
package com.nool.backend.auth.service;

import com.nool.backend.auth.dto.AuthLoginRequestDto;
import com.nool.backend.auth.dto.AuthLoginResponseDto;

public interface AuthService {
    AuthLoginResponseDto login(AuthLoginRequestDto requestDto);
}

Six lines. One method. login() takes the request DTO and returns the response DTO. That's the contract.

2️⃣ The Implementation — AuthServiceImpl.java

auth/service/impl/AuthServiceImpl.javaJava
package com.nool.backend.auth.service.impl;

// imports omitted for brevity

@Service                                          // Spring auto-detects this
@RequiredArgsConstructor                         // Lombok: constructor for final fields
public class AuthServiceImpl implements AuthService {

    private final UserRepository  userRepository;       // to query users table
    private final PasswordEncoder passwordEncoder;      // BCrypt
    private final JwtUtil         jwtUtil;              // to mint the JWT

    @Override
    public AuthLoginResponseDto login(AuthLoginRequestDto requestDto) {

        // ── STEP 1: Look up the user by mobile number ────────────
        User user = userRepository
                .findByMobileNumber(requestDto.getMobileNumber())
                .orElseThrow(() -> new BadRequestException(
                        "Invalid mobile number or password"));
        // ★ Deliberately vague message — doesn't reveal whether the user exists.

        // ── STEP 2: Block disabled accounts ──────────────────────
        if (!user.isActive()) {
            throw new BadRequestException("User account is inactive");
        }

        // ── STEP 3: Verify password ──────────────────────────────
        if (!passwordEncoder.matches(requestDto.getPassword(), user.getPassword())) {
            throw new BadRequestException("Invalid mobile number or password");
        }
        // ★ matches() re-hashes the typed password and compares to the stored hash.
        //   The stored value is a BCrypt hash, never plain text.

        // ── STEP 4: Build JWT claims ─────────────────────────────
        UserProfile profile = user.getUserProfile();   // LAZY-loaded once here

        Map<String, Object> claims = new HashMap<>();
        claims.put("userId",     user.getId());
        claims.put("role",       user.getRole().name());
        claims.put("employeeId", profile != null ? profile.getEmployeeId() : null);
        claims.put("ownerId",    profile != null ? profile.getOwnerId()    : null);

        // ── STEP 5: Mint the JWT ─────────────────────────────────
        String token = jwtUtil.generateToken(claims, user.getMobileNumber());

        // ── STEP 6: Build the response DTO ───────────────────────
        return AuthLoginResponseDto.builder()
                .token(token)
                .role(user.getRole())
                .employeeId(profile != null ? profile.getEmployeeId() : null)
                .ownerId(profile != null ? profile.getOwnerId()    : null)
                .build();
    }
}

🔐 The Vague Error Message

Notice that step 1 and step 3 both throw the same message: "Invalid mobile number or password".

Why not say "no such user" vs "wrong password"? Because differentiating the messages reveals which mobile numbers are registered. An attacker could probe with hundreds of mobile numbers and learn which ones have accounts (then target those for phishing or brute-force attacks). A single vague message gives away nothing.

The "User account is inactive" message at step 2 is slightly more revealing — it tells an attacker the account exists but is disabled. A stricter security posture would use the same vague message there too. NOOL accepts this tiny information leak for clearer user feedback.

🎬 Real-World Scenarios

Scenario 1: Dad (admin) logs in successfully
  1. Frontend posts mobile 9876543210, password Admin@123.
  2. Service finds user_id=1, role=ADMIN, active=true.
  3. BCrypt verifies the password → matches.
  4. UserProfile loaded — employeeId=null, ownerId=null (admins have neither).
  5. Claims: { userId: 1, role: "ADMIN", employeeId: null, ownerId: null }.
  6. JwtUtil signs the token with 24h expiry.
  7. Response sent back. Frontend stores token in localStorage and navigates to /admin/dashboard.
Scenario 2: Rahul (owner) logs in
  1. Frontend posts mobile 9000000001, password rahul123.
  2. Service finds user_id=2, role=SAREE_OWNER, active=true.
  3. BCrypt verifies the password → matches.
  4. UserProfile loaded — ownerId=1 (links to his SareeOwner row).
  5. Claims: { userId: 2, role: "SAREE_OWNER", employeeId: null, ownerId: 1 }.
  6. JwtUtil mints the token. Frontend navigates to /owner/dashboard.
Scenario 3: Prema types the wrong password
  1. Frontend posts 9000000002, prema12 (forgot the last digit).
  2. Service finds user_id=3.
  3. BCrypt re-hashes "prema12", compares to the stored hash → no match.
  4. Throws BadRequestException("Invalid mobile number or password").
  5. GlobalExceptionHandler converts to HTTP 400.
  6. Frontend shows the error in red. Prema retypes and gets in on the second try.
Scenario 4: A disabled worker tries to log in
  1. Suresh left the workshop 6 months ago; his User.active was flipped to false.
  2. He tries to log in with his old (still correct) credentials.
  3. Service finds his User row.
  4. Step 2: !user.isActive() → true → throw "User account is inactive".
  5. Returns HTTP 400 with clear message. Suresh knows his account was disabled.
Common login-service mistakes
Saying "no such user" vs "wrong password" in different cases
Use the SAME vague message for both. Attackers can otherwise enumerate valid mobile numbers by reading error responses.
Comparing passwords with typed.equals(stored)
The stored value is a BCrypt hash. matches() re-hashes the input and compares. Never plain-text compare.
Logging the password (even hashed) anywhere
Never. Logs leak. The password should only travel from the HTTPS request body into matches() and then be forgotten.
Putting the password in the JWT claims
JWT payloads are readable by anyone. Claims should only contain non-sensitive identifiers (userId, role, FK IDs). NEVER passwords.
Forgetting profile != null guards
Admins have no UserProfile. Direct calls to profile.getOwnerId() would NullPointerException. The null check is essential.
Interview questions
Why is the login error message intentionally vague?
To prevent username enumeration. If the system said "no such user" vs "wrong password", attackers could probe with mobile numbers and learn which ones exist — useful for targeted phishing or focused brute-force attempts. A vague message reveals nothing.
Why does matches() work if BCrypt is one-way?
BCrypt is one-way for recovery — you can't decrypt a hash back. But verification works differently: matches() takes the typed password, re-hashes it with the SAME salt embedded in the stored hash, and byte-compares the two hashes. If the typed password is correct, the two hashes match exactly.
Why both employeeId and ownerId in the JWT?
For workers, employeeId is set so "my data" endpoints can filter without a DB lookup. For owners, ownerId. For admins, both are null (admins specify the target ID in URL paths). Embedding these IDs in the token avoids hitting the DB just to find out "who is this person in the business sense?".
Why is there no logout() method?
Because logout is stateless on the server. JWTs aren't tracked server-side, so "logout" just means the frontend deletes the token from localStorage. The POST /api/auth/logout endpoint exists for API symmetry but its handler method body is empty.
Why @RequiredArgsConstructor over field @Autowired?
Constructor injection (which @RequiredArgsConstructor generates) is recommended by Spring: makes dependencies explicit, fields can be final, easier to unit-test with mocks. Field @Autowired is older and harder to test.
