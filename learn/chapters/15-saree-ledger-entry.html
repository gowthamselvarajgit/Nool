🧠 The Big Idea — Ledger, Not Stock Counter

Before we look at the code, here's the core design decision: NOOL does NOT keep a single "current stock" number per owner. Instead, it records every event (saree coming in OR going out) as its own row, and computes the current balance on the fly.

The two possible designs side-by-side:

Stock counter (NOT what we use)Ledger (what NOOL uses)
StorageOne number per owner (currentStock)One row per transaction event
ReceiptAdd to the numberInsert a RECEIPT row
ReturnSubtract from the numberInsert a RETURN row
HistoryLost foreverComplete, every transaction
"How many on May 13?"Cannot answerEasy — SUM events up to that date
Audit trailNoneEvery event with date + remarks
Risk of driftHigh — bugs can silently corruptNone — always recomputable
This is exactly the same pattern banks use for passbooks. Your account balance isn't stored as a number — it's derived by summing every credit and debit on your statement. NOOL's inventory works the same way.

entity/owner/SareeLedgerEntry.java Java 
package com.nool.backend.entity.owner;

import com.nool.backend.enums.LedgerEntryType;       // RECEIPT or RETURN
import jakarta.persistence.*;
import lombok.*;
import java.time.LocalDate;
import java.time.LocalDateTime;

@Entity
@Table(name = "saree_ledger")            // SQL table — think "passbook"
@Getter @Setter
@NoArgsConstructor
@AllArgsConstructor
@Builder
public class SareeLedgerEntry {

    // ── PRIMARY KEY ───────────────────────────────────────────────
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    @Column(name = "entry_id")
    private Long id;                                // row number, auto-increment

    // ── OWNER LINK ────────────────────────────────────────────────
    @ManyToOne(fetch = FetchType.LAZY)            // many entries → one owner
    @JoinColumn(name = "owner_id", nullable = false)
    private SareeOwner sareeOwner;
    // ★ Every entry MUST belong to an owner (FK is not nullable).
    //   FetchType.LAZY = don't load the full SareeOwner object unless asked.
    //   Most ledger operations only need the IDs, so this avoids an extra
    //   JOIN on every read.

    // ── ENTRY TYPE (RECEIPT or RETURN) ────────────────────────────
    @Enumerated(EnumType.STRING)               // store as "RECEIPT" or "RETURN" text
    @Column(name = "entry_type", nullable = false, length = 16)
    private LedgerEntryType entryType;
    // ★ The direction. RECEIPT = sarees coming IN, RETURN = going OUT.
    //   The "in-hand" formula depends entirely on this.

    // ── ENTRY DATE (business date, not insert time) ───────────────
    @Column(name = "entry_date", nullable = false)
    private LocalDate entryDate;
    // ★ The date the transaction actually happened — NOT when typed
    //   into NOOL. Dad might record Friday's receipt on Monday morning.
    //   This stores Friday's date so reports stay accurate.

    // ── QUANTITY (always positive) ────────────────────────────────
    @Column(name = "quantity", nullable = false)
    private Integer quantity;
    // ★ Always positive! Direction is encoded in entryType, not in sign.
    //   "100 RECEIPT" reads more naturally than "+100", "30 RETURN"
    //   reads more naturally than "-30".

    // ── REMARKS (optional note) ───────────────────────────────────
    @Column(name = "remarks")
    private String remarks;
    // Free-form. "Diwali batch", "Damaged on arrival",
    // "Second batch — Saturday morning". Optional.

    // ── AUDIT TIMESTAMP (only created_at, NO updated_at!) ─────────
    @Column(name = "created_at", nullable = false, updatable = false)
    private LocalDateTime createdAt;
    // When the row was typed into NOOL — for audit log only.

    @PrePersist
    protected void onCreate() {
        this.createdAt = LocalDateTime.now();
    }

    // ★★★ NOTICE: No @PreUpdate. No `updatedAt` field. No setters that
    //   should ever be called after save. Ledger rows are IMMUTABLE.
    //   To "correct" a wrong row, insert a compensating row — never edit
    //   history. This is the same principle banks follow.
}

🧮 The Famous In-Hand Formula

The whole point of this entity is to enable this one calculation:

Sarees still with the workshop (for one owner)
inHand = SUM(quantity WHERE type=RECEIPT) − SUM(quantity WHERE type=RETURN)

And it's just as simple in SQL:

-- How many of Rahul's sarees are still in the workshop?
SELECT
    COALESCE(SUM(CASE WHEN entry_type = 'RECEIPT' THEN quantity ELSE 0 END), 0)
  - COALESCE(SUM(CASE WHEN entry_type = 'RETURN'  THEN quantity ELSE 0 END), 0)
AS in_hand
FROM saree_ledger
WHERE owner_id = 1;

The actual NOOL code (in SareeInventoryServiceImpl, Chapter 45) splits this into two simpler queries — one for RECEIPT sum, one for RETURN sum — and subtracts in Java. Either way, the result is always computed fresh; never stored.

-- A typical week with Rahul (owner_id = 1):
INSERT INTO saree_ledger
    (owner_id, entry_type, entry_date, quantity, remarks, created_at)
VALUES
    (1, 'RECEIPT', '2026-05-11', 100, 'Diwali batch',       '2026-05-11 11:00:00'),
    (1, 'RETURN',  '2026-05-13', 40,  'First batch ready', '2026-05-13 16:30:00'),
    (1, 'RETURN',  '2026-05-15', 30,  'Second batch',      '2026-05-15 17:00:00');

-- Compute Rahul's current in-hand:
SELECT
  (SELECT COALESCE(SUM(quantity), 0) FROM saree_ledger
     WHERE owner_id = 1 AND entry_type = 'RECEIPT')
- (SELECT COALESCE(SUM(quantity), 0) FROM saree_ledger
     WHERE owner_id = 1 AND entry_type = 'RETURN')
  AS in_hand;

-- Result: 100 - (40 + 30) = 30 sarees still being polished

🔒 Why This Entity Is Deliberately Immutable

You'll have noticed: there's no @PreUpdate, no updatedAt field, no expectation of editing a row after creation. That's not laziness — it's a deliberate append-only design.

What "append-only" means

  • Once a ledger row is written, it stays exactly as-is forever.
  • No UPDATE statements ever run against this table from business code.
  • No DELETE either, except in rare admin correction scenarios (Chapter 46).
  • If a row was wrong, the correct fix is to insert a compensating row — not edit the wrong one.

Why this matters

Audit reliability. If a tax inspector ever asks "what sarees did you handle for Rahul in March?", my parents can show every single transaction with the exact date and remark, and prove that no data was edited after the fact. Editing history would destroy that trust.

Bug containment. Without UPDATE/DELETE, the worst a buggy service can do is insert a wrong row. That's recoverable. With UPDATE, a bug could silently corrupt years of history.

"But what if a row really IS wrong?"

Two clean correction patterns:

MistakeHow to fix it
Recorded RECEIPT of 100 but should have been 80 Insert a compensating RETURN of 20 with a remark like "Correction for entry #147".
Recorded RETURN of 50 that never actually happened Insert a compensating RECEIPT of 50 with a remark explaining.
Critical error — wrong owner, wrong date Use admin-only deleteEntry() endpoint (Chapter 46) to actually remove the bad row. This is the last-resort tool.

🎬 Real-World Scenarios

Scenario 1: Rahul brings 100 sarees on Monday
  1. Dad logs into NOOL → Inventory → Add Receipt.
  2. Picks "Rahul Textiles", quantity 100, date 2026-05-11, remarks "Diwali batch".
  3. Frontend posts to POST /api/inventory/receipt.
  4. SareeInventoryServiceImpl.addReceipt() validates and inserts.
  5. New ledger row: entry_id=1, owner_id=1, type=RECEIPT, qty=100, date=2026-05-11, remarks="Diwali batch".
  6. @PrePersist stamps created_at = NOW().
  7. Rahul's new in-hand: 100 received − 0 returned = 100 in hand.
Scenario 2: Wednesday — Dad returns 40 polished sarees
  1. Dad clicks Inventory → Add Return. Rahul Textiles, 40 sarees, today.
  2. Service computes current in-hand = 100. Requested 40 ≤ 100? Yes — approved.
  3. Inserts: entry_id=2, owner_id=1, type=RETURN, qty=40, date=2026-05-13.
  4. Rahul's new in-hand: 100 − 40 = 60.
Scenario 3: Dad accidentally tries to return 80 sarees (only 60 in hand)
  1. Dad fat-fingers 80 instead of 8.
  2. Service recomputes in-hand: 100 received − 40 returned = 60.
  3. Check: 80 > 60 → REJECT.
  4. Throws BadRequestException("Cannot return 80 sarees for Rahul Textiles. Only 60 remaining in hand.").
  5. GlobalExceptionHandler converts to HTTP 400 with friendly message.
  6. Frontend shows the error in red. No ledger row is inserted.
  7. Dad realizes his typo, retypes 8, and tries again. This time it succeeds.

This safety net is something a paper notebook fundamentally cannot provide.

Scenario 4: Dad recorded the wrong RECEIPT quantity — how to fix it?

Suppose Dad recorded a receipt of 100, but actually Rahul brought 80 (Dad mis-counted on Monday). Today he realizes the mistake.

Wrong approach: Edit entry #1 directly. ❌ Breaks the audit trail.

Right approach: Insert a compensating row:

INSERT INTO saree_ledger
    (owner_id, entry_type, entry_date, quantity, remarks, created_at)
VALUES
    (1, 'RETURN', '2026-05-15', 20,
     'Correction: entry #1 overstated by 20',
     '2026-05-15 09:00:00');

Now Rahul's in-hand drops by 20 to reflect the true count, and the original mistake + correction are both visible in the history forever.

Common mistakes around SareeLedgerEntry
Adding a currentStock field on SareeOwner and updating it on each transaction
Loses history; risks drift. The ledger pattern (one row per event) is the proven safe approach. Current stock is ALWAYS derived as a SUM.
Allowing negative quantities to mean RETURN
Sign-based directions are confusing. Use the entryType enum. Quantities are always positive — easier to read, debug, and aggregate.
Adding @PreUpdate to let admins edit past entries
Ledger rows are immutable. If a row is wrong, insert a compensating row. Editing history breaks audit trust and risks data corruption.
Storing the rate on each ledger row to "lock in" historical pricing
NOOL doesn't do this currently — it always uses the owner's current rate. If you ever extend to lock historical rates, the natural place is a rate_at_time column on this entity. But don't add it speculatively.
Removing the FK to saree_owner for "simplicity"
The FK prevents orphan rows — you can't insert a ledger entry pointing to a non-existent owner. DB-level referential integrity catches real bugs.
Computing in-hand WITHOUT a row lock on the owner
Two simultaneous returns can both read the same "before" total and both pass validation when only one should. The service layer (Chapter 45) uses pessimistic locking to serialize this.
Interview questions about SareeLedgerEntry
Why use an append-only ledger instead of a single stock count?
Three reasons. (1) Complete audit history — every transaction with dates and remarks is preserved. (2) No drift — a stock counter can get out of sync with reality; a ledger is always recomputable from raw events. (3) It mirrors how my parents already worked in paper notebooks (receipts in one column, returns in another), so the mental model is familiar.
Why is quantity always a positive Integer? Why not use sign for direction?
Clarity. "RETURN 40" reads more naturally than "-40". Queries like SUM(quantity) WHERE entry_type = 'RETURN' are simpler than worrying about signs. And a negative number that should have been positive could silently corrupt the in-hand calculation — using the enum + positive integers makes errors obvious.
Why doesn't this entity have @PreUpdate or updatedAt?
Because ledger entries should never be updated. They represent historical events — editing them would rewrite history and destroy audit trust. The absence of @PreUpdate isn't an oversight; it's a design statement: "this row is immutable".
What's the difference between entryDate and createdAt?
entryDate is the business date — when the sarees actually changed hands. createdAt is the technical date — when the row was typed into NOOL. They're often different: Dad might enter Friday's receipts on Monday morning. Reports use entryDate; audit logs use createdAt.
How would you compute the in-hand balance for a specific past date?
Sum all RECEIPTs minus all RETURNs up to that date: SUM(quantity WHERE type=RECEIPT AND entry_date <= ?) − SUM(quantity WHERE type=RETURN AND entry_date <= ?). This is the magic of the ledger pattern — you can rewind history to any point. A stock counter could never do this.
Could two concurrent return requests both succeed and overdraw the balance?
Without protection, yes — both could read the same "before" total, both pass validation, and both INSERT. To prevent this, the service layer (Chapter 45) uses a pessimistic lock on the owner row: SELECT ... FOR UPDATE via findByIdForUpdate. This serializes simultaneous transactions; the second one reads the updated balance and correctly fails the check.
Why @ManyToOne here instead of just a Long owner_id field?
Two benefits. (1) The FK constraint enforces database-level integrity — you can't insert a row pointing at a non-existent owner. (2) Convenient navigation in code: ledger.getSareeOwner().getOwnerName(). With FetchType.LAZY, we only pay for the JOIN when the navigation is actually used.
What does the table name saree_ledger (not saree_ledgers) suggest?
A small naming inconsistency — most tables in NOOL are plural (users, employees), but this one is singular. It's an accepted historical choice — the word "ledger" naturally feels singular even when there are many entries. A naming-consistency cleanup would rename to saree_ledger_entries or saree_ledgers.
