🧠 Why Employee Is Separate From User

A reminder from Chapter 11: a User holds login data, an Employee holds business data. Splitting them keeps both classes focused. Here's what each one is responsible for:

User (the login row)Employee (the worker row)
Mobile number, password, roleName, joining date, polish rate, status
Used at every API call (auth check)Used in payroll, attendance, daily work
Sensitive (BCrypt hash)Routine business facts
One row per person who logs inOne row per worker, even if no login yet
The flexible bit: the user_id link on Employee is nullable. That means Dad can add a new worker to the system before setting up their phone login. The Employee row exists; the User row gets created later. We'll see exactly how that helps below.

📜 The Full File — Every Line Explained

entity/employee/Employee.java Java 
package com.nool.backend.entity.employee;
// ↑ Lives in entity/employee/, a sub-package grouping all worker-related entities

import com.nool.backend.auth.entity.User;        // for the OneToOne login link
import com.nool.backend.enums.EmployeeStatus;     // ACTIVE / LEFT
import jakarta.persistence.*;
import lombok.*;
import java.time.LocalDate;                       // for joiningDate (day-precision)
import java.time.LocalDateTime;                   // for created/updated timestamps

@Entity
@Table(name = "employees")                    // SQL table = `employees`
@Getter @Setter
@NoArgsConstructor
@AllArgsConstructor
@Builder
public class Employee {

    // ── PRIMARY KEY ───────────────────────────────────────────────
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    @Column(name = "employee_id")            // MySQL auto-increments: 1, 2, 3...
    private Long id;

    // ── EMPLOYEE NAME ─────────────────────────────────────────────
    @Column(name = "employee_name", nullable = false)
    private String name;                       // "Prema", "Ravi", etc.

    // ── MOBILE NUMBER ─────────────────────────────────────────────
    @Column(name = "mobile_number",
            nullable = false, unique = true)   // duplicated from User.mobileNumber
    private String mobileNumber;
    // ★ NOTE: This is a deliberate denormalization. The same number lives
    //   on the linked User row too. Why duplicate? So we can look up a
    //   worker by phone without joining the users table — useful for
    //   admin lists and quick contact display.

    // ── JOINING DATE ──────────────────────────────────────────────
    @Column(name = "joining_date", nullable = false)
    private LocalDate joiningDate;             // for tenure calculations
    // LocalDate (not LocalDateTime) — only the day matters, not the hour.

    // ── STATUS ────────────────────────────────────────────────────
    @Enumerated(EnumType.STRING)              // store as "ACTIVE" or "LEFT" text
    @Column(name = "status", nullable = false)
    private EmployeeStatus status;
    // Two values only: ACTIVE (currently employed) or LEFT (no longer with us).
    // Soft-delete pattern — see Chapter 08.

    // ── POLISH RATE ★ ─────────────────────────────────────────────
    @Column(name = "polishing_rate", nullable = false)
    private Double polishRate;                 // rupees per saree polished
    // ★ THE MOST IMPORTANT BUSINESS FIELD on this entity.
    //   Daily earnings = (freshCount + rePolishCount) × this rate.
    //   Each worker can have their OWN rate — Prema might be ₹30/saree,
    //   Ravi might be ₹25/saree (he's newer).

    // ── LOGIN LINK ────────────────────────────────────────────────
    // (link to the User row that holds login credentials)
    @OneToOne
    @JoinColumn(name = "user_id",
                nullable = true,              // CAN be NULL — see below
                unique   = true)              // each User links to at most one Employee
    private User user;
    // ★ nullable=true lets Dad add an Employee BEFORE creating their phone
    //   login. Useful when a new worker shows up Monday but their phone
    //   account is set up Tuesday.

    // ── AUDIT TIMESTAMPS ──────────────────────────────────────────
    @Column(name = "created_at", nullable = false, updatable = false)
    private LocalDateTime createdAt;

    @Column(name = "updated_at")
    private LocalDateTime updatedAt;

    // ── LIFECYCLE HOOKS ───────────────────────────────────────────
    @PrePersist
    protected void onCreate(){
        LocalDateTime now = LocalDateTime.now();
        this.createdAt = now;
        this.updatedAt = now;
    }

    @PreUpdate
    protected void onUpdate(){
        this.updatedAt = LocalDateTime.now();
    }
}

🗄️ What This Becomes in MySQL

CREATE TABLE employees (
    employee_id     BIGINT       NOT NULL AUTO_INCREMENT,
    employee_name   VARCHAR(255)  NOT NULL,
    mobile_number   VARCHAR(255)  NOT NULL UNIQUE,
    joining_date    DATE          NOT NULL,
    status          VARCHAR(20)   NOT NULL,    -- enum stored as text
    polishing_rate  DOUBLE        NOT NULL,
    user_id         BIGINT        UNIQUE,        -- nullable but unique
    created_at      DATETIME(6)   NOT NULL,
    updated_at      DATETIME(6),
    PRIMARY KEY (employee_id),
    FOREIGN KEY (user_id) REFERENCES users(user_id)
);

💰 The Polish Rate — How a Worker's Pay Is Calculated

polishingRate (column polishing_rate) is the most important business field on this entity. It directly drives every salary calculation in NOOL.

Worker's earnings for one day
earnings = (freshCount + rePolishCount) × employee.polishingRate

So if Prema's rate is ₹30 and on May 15 she polished 8 fresh + 2 re-polish sarees:

earnings = (8 + 2) × ₹30 = ₹300 for that day

The summary screen (covered in Chapter 39) totals this across a date range to get total earnings, then subtracts SUM(salary_payment.amount_paid) in the same range to compute the pending salary.

Why rate is per-worker (not global)

Different workers earn different rates based on skill and experience:
  • Prema (experienced) → ₹30/saree
  • Ravi (newer, still learning) → ₹25/saree
Storing the rate per-employee gives my parents the flexibility to reward better workers, grant raises individually, and adjust without affecting anyone else. A global rate would be too rigid for a real workshop.

🪢 The Nullable user Link — Why It Matters

Notice the relationship to User is nullable = true:

The crucial lineJava
@OneToOne
@JoinColumn(name = "user_id", nullable = true, unique = true)
private User user;

That single nullable=true enables three useful flows:

ScenarioWhat it allows
Workshop hires a new worker mid-week Dad can add the Employee row immediately (so they can mark attendance, log daily work) and create the phone login later when the worker brings their phone.
Quick HR experiments Try someone for a week without bothering to set up app login. Track their work; decide whether to onboard fully.
Bulk seed data Initial bulk import can create all employees first, then later attach login accounts.

The unique = true constraint still applies whenever a User is set: no User can be linked to more than one Employee. So even with nullable, we can never end up with "two employees sharing one phone login".

🎬 Real-World Scenarios

Scenario 1: Dad hires Ravi as a new polisher
  1. Dad clicks "Add Employee" and fills the form:
    • Name: Ravi
    • Mobile: 9000000003
    • Joining date: 2026-03-01
    • Polish rate: ₹25/saree (lower than Prema's ₹30 — he's new)
    • Password: ravi123
  2. The employee-service code (covered in Chapter 30):
    • Creates a User row with role WORKER and BCrypt-hashed password.
    • Creates an Employee row with status ACTIVE, polishRate 25.0, linked to that User.
    • Creates a UserProfile row bridging them.
  3. The Employee's @PrePersist stamps createdAt and updatedAt.
  4. Hibernate runs INSERT INTO employees (...) VALUES (...).
  5. Ravi can now log in on his phone and see his own dashboard.
Scenario 2: Calculating Prema's weekly earnings

It's Saturday. Dad wants to know what Prema earned this week (May 11-15).

  1. Service queries: SELECT * FROM employee_daily_work WHERE employee_id = 5 AND work_date BETWEEN '2026-05-11' AND '2026-05-15'.
  2. For each day, computes (fresh + rePolish) × Prema.polishRate.
  3. Mon: 8+2=10 × ₹30 = ₹300
  4. Tue: 9+1=10 × ₹30 = ₹300
  5. Wed: 7+3=10 × ₹30 = ₹300
  6. Thu: she was absent (no row)
  7. Fri: 10+0=10 × ₹30 = ₹300
  8. Total earnings: ₹1,200
  9. Already paid in the same range: ₹0 (it's the end of the week, no payment yet)
  10. Pending salary: ₹1,200
Scenario 3: A worker leaves

Suresh (employee_id=7) tells Dad he's moving to another city. Two updates happen:

-- Mark him as LEFT
UPDATE employees SET status = 'LEFT', updated_at = '2026-05-15 17:00:00' WHERE employee_id = 7;

-- Disable his login
UPDATE users SET active_status = false, updated_at = '2026-05-15 17:00:00' WHERE user_id = 8;

All of Suresh's history (daily_work, attendance, salary_payment rows) stays in the database forever. The "Employees" list filter shows only ACTIVE workers by default, so he naturally disappears from operational screens. But Dad can still pull up Suresh's salary history any time for audit.

Common mistakes around Employee.java
Hard-deleting an employee when they leave
Use status = LEFT instead. Hard delete would orphan all attendance/daily-work/salary rows (or worse, cascade-delete them, losing all history).
Forgetting to set unique = true on the user_id link
Without it, two Employee rows could share the same User — meaning one phone login becomes "two workers" with split history. The unique constraint blocks this at the DB level.
Using BigDecimal wishfully — or int destructively — for the polish rate
NOOL uses Double, which is fine for whole-rupee rates (₹25, ₹30, ₹50). int would prevent half-rupee rates. BigDecimal would be ideal for money but adds boilerplate. Accept the small floating-point risk for simplicity.
Changing polishingRate and expecting old salary calculations to update
Rate changes apply going forward only. Old paid salaries don't get recalculated retroactively — they were already paid at the old rate. Don't retroactively update old salary_payment rows; insert a new payment for any difference if needed.
Duplicating mobile number between User and Employee and letting them drift
When a worker's phone changes, update BOTH the User row AND the Employee row. Service methods should update both together inside one transaction. Don't change one and forget the other.
Interview questions about Employee.java
Why is user_id nullable on Employee?
So Dad can add a new worker BEFORE creating their phone login. For example, Ravi joins Monday but his phone account is set up Tuesday. Between those two days, the Employee row exists with user_id = NULL; on Tuesday, the User row is created and linked. Without nullable, the admin would be forced to create both rows simultaneously, which doesn't match real workshop workflow.
What's the difference between Employee's mobileNumber and User's mobileNumber?
They store the SAME number, intentionally. The User row uses it for login lookup (SELECT * FROM users WHERE mobile_number = ?). The Employee row uses it as a contact reference — so my parents can call Prema without joining the users table from a worker list. It's a deliberate denormalization for convenience.
Why is polishingRate on each Employee instead of a global rate?
Real workshops pay different workers different rates based on skill, experience, and loyalty. Prema is fast and skilled → ₹30/saree. Ravi just joined → ₹25. Storing the rate per-employee gives my parents the flexibility to negotiate individually, reward better workers, and change one person's rate without touching anyone else.
What happens if I update Prema's polish rate? Do her old salaries change?
No — old paid salaries stay at the old rate (they were already disbursed). Only future calculations use the new rate. If Prema's rate changes from ₹30 to ₹32 on May 1, her April salary calculation is still ₹30 (already paid), but anything she polishes from May 1 onwards earns ₹32. This is by design — we never retroactively recompute completed payments.
Why use LocalDate for joiningDate instead of LocalDateTime?
Because joining is a day-precision event — we care that Prema joined on Aug 12, 2025, not what minute. LocalDate stores only the date (no hours/minutes), is smaller, and avoids any timezone shenanigans. LocalDateTime would store extra precision we don't need.
How would you add support for hourly pay alongside per-saree pay?
Two clean options. (1) Add a payMode enum (PER_SAREE / HOURLY) and an hourlyRate field — the salary service picks the formula based on mode. (2) Make the rate field a structured object: PayConfig {mode, rate, ...} stored as JSON in MySQL. Option 1 is simpler and probably right for NOOL's scale.
Could two employees have the same mobile number?
No — the column has unique = true, meaning MySQL itself rejects duplicates. If you try to insert a second Employee with an existing mobile, you get a DataIntegrityViolationException, which the GlobalExceptionHandler (Chapter 09) converts into "This mobile number is already registered".
