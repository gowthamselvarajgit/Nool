Recording every payout

EmployeeDailyWork tracks what a worker earned. SalaryPayment tracks what they were actually paid. Each row represents one disbursement — covering one specific date range — with date, amount, payment mode, and an optional remark.

🧠 Three Dates, One Payout

The most unusual thing about this entity is that it has three date fields. Most entities have one or two. Let's understand why each is needed:

FieldWhat it means
fromDateStart of the work period being paid for
toDateEnd of that work period
paymentDateWhen the money was actually handed over

Example: on May 15, Dad pays Prema ₹1,200 cash for her work between May 11 and May 14. That row stores: fromDate=2026-05-11, toDate=2026-05-14, paymentDate=2026-05-15.

Why three dates? Because pay days and work days are not the same. Workers might be paid weekly, monthly, or on-demand — and the payment date can be days after the work was done. Recording both lets us answer different questions: "how much did we pay Prema in May?" (uses paymentDate) and "how much was she paid for her work in week 19?" (uses fromDate/toDate).

entity/employee/SalaryPayment.java Java 
package com.nool.backend.entity.employee;

import com.nool.backend.enums.PaymentMode;     // CASH / UPI / BANK_TRANSFER
import jakarta.persistence.*;
import lombok.*;
import java.time.LocalDate;
import java.time.LocalDateTime;

@Entity
@Table(name = "salary_payments")            // SQL table = salary_payments
@Getter @Setter
@NoArgsConstructor
@AllArgsConstructor
@Builder
public class SalaryPayment {

    // ── PRIMARY KEY ───────────────────────────────────────────────
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    @Column(name = "salary_payment_id")
    private Long id;

    // ── WORK PERIOD START ─────────────────────────────────────────
    @Column(name = "from_date", nullable = false)
    private LocalDate fromDate;                    // e.g., 2026-05-11

    // ── WORK PERIOD END ───────────────────────────────────────────
    @Column(name = "to_date", nullable = false)
    private LocalDate toDate;                      // e.g., 2026-05-14

    // ── AMOUNT PAID ───────────────────────────────────────────────
    @Column(name = "amount_paid", nullable = false)
    private Double amountPaid;                    // in rupees, e.g., 1200.00

    // ── ACTUAL PAYMENT DATE ───────────────────────────────────────
    @Column(name = "payment_date", nullable = false)
    private LocalDate paymentDate;                 // when the money handed over

    // ── HOW WAS IT PAID ───────────────────────────────────────────
    @Enumerated(EnumType.STRING)
    @Column(name = "payment_mode", nullable = false)
    private PaymentMode paymentMode;               // CASH / UPI / BANK_TRANSFER

    // ── REMARKS (optional) ────────────────────────────────────────
    @Column(name = "remarks")
    private String remarks;
    // "Weekly advance", "Festival bonus included", etc.

    // ── EMPLOYEE LINK ─────────────────────────────────────────────
    @ManyToOne
    @JoinColumn(name = "employee_id", nullable = false)
    private Employee employee;

    // ── AUDIT TIMESTAMPS ──────────────────────────────────────────
    @Column(name = "created_at", nullable = false, updatable = false)
    private LocalDateTime createdAt;

    @Column(name = "updated_at")
    private LocalDateTime updatedAt;

    // ── LIFECYCLE HOOKS ───────────────────────────────────────────
    @PrePersist
    protected void onCreate(){
        LocalDateTime now = LocalDateTime.now();
        this.createdAt = now;
        this.updatedAt = now;
    }

    @PreUpdate
    protected void onUpdate(){
        this.updatedAt = LocalDateTime.now();
    }
}

🗄️ What This Becomes in MySQL

CREATE TABLE salary_payments (
    salary_payment_id BIGINT      NOT NULL AUTO_INCREMENT,
    employee_id       BIGINT      NOT NULL,
    from_date         DATE        NOT NULL,
    to_date           DATE        NOT NULL,
    amount_paid       DOUBLE      NOT NULL,
    payment_date      DATE        NOT NULL,
    payment_mode      VARCHAR(20) NOT NULL,        -- CASH / UPI / BANK_TRANSFER
    remarks           VARCHAR(255),
    created_at        DATETIME(6) NOT NULL,
    updated_at        DATETIME(6),

    PRIMARY KEY (salary_payment_id),
    FOREIGN KEY (employee_id) REFERENCES employees(employee_id)
);

-- Dad pays Prema ₹1,200 cash on May 15 for her May 11-14 work:
INSERT INTO salary_payments
    (employee_id, from_date, to_date, amount_paid, payment_date,
     payment_mode, remarks, created_at, updated_at)
VALUES
    (5, '2026-05-11', '2026-05-14', 1200.0, '2026-05-15',
     'CASH', 'Weekly advance',
     '2026-05-15 16:00:00', '2026-05-15 16:00:00');

-- How much have we paid Prema this month (by payment date)?
SELECT SUM(amount_paid)
FROM   salary_payments
WHERE  employee_id = 5
  AND  payment_date BETWEEN '2026-05-01' AND '2026-05-31';

-- How much have we paid Prema FOR her work in a specific period?
SELECT SUM(amount_paid)
FROM   salary_payments
WHERE  employee_id = 5
  AND  from_date >= '2026-05-11'
  AND  to_date   <= '2026-05-14';

🔄 How EmployeeDailyWork + SalaryPayment Work Together

Together, these two tables answer the pending-salary question:

Pending salary for one worker in a period
pending = SUM(daily earnings) − SUM(amount_paid in same period)

In SQL:

-- The full pending-salary calc for Prema (employee_id=5) for May 1-15:

-- Step 1: total earned
SELECT SUM((fresh_saree_count + repolish_saree_count) * e.polishing_rate) AS earned
FROM   employee_daily_work w
JOIN   employees e ON e.employee_id = w.employee_id
WHERE  w.employee_id = 5
  AND  w.work_date BETWEEN '2026-05-01' AND '2026-05-15';
-- Result: 1500.0  (50 sarees × ₹30)

-- Step 2: total already paid in the same period (by from_date/to_date range)
SELECT COALESCE(SUM(amount_paid), 0) AS paid
FROM   salary_payments
WHERE  employee_id = 5
  AND  from_date >= '2026-05-01'
  AND  to_date   <= '2026-05-15';
-- Result: 1200.0

-- Pending = earned - paid = 1500 - 1200 = 300

The actual code in SalaryServiceImpl (Chapter 39) does these two queries separately and computes the difference in Java. Simpler to read and debug than one monster SQL.

🎬 Real-World Scenarios

Scenario 1: End-of-week payout
  1. It's Friday May 15. Prema's week (Mon-Thu) had 50 polished sarees × ₹30 = ₹1,500 earned.
  2. Dad gives her ₹1,200 in cash as a partial advance.
  3. Dad logs into NOOL → Salary → Pay. Picks Prema, amount 1200, from 2026-05-11, to 2026-05-14, payment date today, mode CASH.
  4. Backend inserts a SalaryPayment row.
  5. Dashboard now shows: Prema · Earned ₹1,500 · Paid ₹1,200 · Pending ₹300.
Scenario 2: Same-day work, deferred pay
  1. Prema works through May 11-17 (a full week), earning ₹2,100.
  2. Dad doesn't pay her until May 20 (waited until the bank deposit cleared).
  3. On May 20, Dad records: from_date=11, to_date=17, payment_date=20, amount=2100, mode=BANK_TRANSFER.
  4. Two different reports can use this row:
    • "What did we pay Prema in May?" → SUM where payment_date is in May → ₹2,100.
    • "Pending salary as of May 17?" → SUM where from_date/to_date fall in the work period → ₹2,100 paid, so pending = 0.
Scenario 3: Festival bonus on top of regular wage
  1. It's Diwali week. Dad gives Prema ₹2,500 even though her actual earnings were ₹2,000.
  2. Dad records: amount=2500, remarks="₹500 Diwali bonus included".
  3. The dashboard now shows Prema as "overpaid" — earned ₹2,000 but paid ₹2,500 → pending = −₹500.
  4. This is fine. Negative pending = "we paid them in advance". Future earnings will offset it naturally.
Common mistakes around SalaryPayment
Using only one date field (e.g., dropping paymentDate and using toDate)
Work date and payment date are different things. Conflating them breaks reports — you can't answer "what did we pay in May?" if payment_date is missing.
Storing the salary calculation result on this entity (e.g., earnedAmount)
Earnings are computed from EmployeeDailyWork × polish rate, not stored here. SalaryPayment stores only what was actually paid out. Mixing them creates double-truth.
Using int or BigDecimal instead of Double
NOOL uses Double for consistency with polishRate and polishRatePerSaree on the other entities. Going forward with BigDecimal would mean changing all three together — not a one-off decision.
Hard-deleting a payment to "fix" an over-pay
Don't delete. Insert a corrective payment for the difference (negative amount, or a separate row with an explanatory remark). The audit trail matters.
Skipping the paymentMode field as "not important"
It's essential for reconciliation. At month-end Dad needs to confirm "we paid ₹15,000 in CASH this month — does that match what we took out of the cash box?". Without paymentMode, that reconciliation is impossible.
Interview questions about SalaryPayment
Why store BOTH a work-period (from/to) AND a payment date?
Because they answer different questions. fromDate/toDate tells us which work this payment was FOR; paymentDate tells us WHEN money left the till. These can be days or weeks apart, and reports need to slice by both axes — e.g., "what did we pay in May?" (paymentDate) vs "what's the pending for week 19's work?" (fromDate/toDate).
Why no UNIQUE constraint on (employee_id, from_date, to_date)?
Because a single work period can have multiple payments — e.g., a ₹500 advance on Monday and a ₹700 settlement on Friday, both for the same week. Both rows are legitimate. The salary calc SUMs them. A UNIQUE constraint would prevent this very natural pattern.
Should payments be immutable like ledger entries?
Arguably yes — money records should be auditable. But NOOL currently allows @PreUpdate because Dad might want to fix a typo (amount entered as 1200 instead of 1300) within a few minutes of recording. The audit timestamp tells you when an edit happened. If we ever needed stricter audit (for tax purposes), we'd remove the setters and require corrective rows instead — same pattern as the inventory ledger.
What does negative pending mean?
That we've paid the worker MORE than they've earned in the period — an advance or a bonus. Negative pending isn't an error; it's an "owed by the worker" state that naturally settles as future work earns more. The dashboard handles it gracefully (typically shows "₹500 advance" instead of "-₹500 pending").
Why use Double for the amount instead of BigDecimal?
Consistency with the rest of NOOL's money columns (polishRate, polishRatePerSaree). For whole-rupee values used in our context, the floating-point risk is negligible. BigDecimal would be safer for international currencies with sub-units (cents/paise), but adds boilerplate. Pragmatic trade-off accepted.
What's the difference between this entity and a "transaction log" for cash-out?
Conceptually similar — both record money leaving. The difference is purpose: this entity is specifically about payroll, with worker FK and work-period context. A general cash-out log would lack the from_date/to_date and the link to a specific employee. NOOL keeps them separate because mixing all "money out" into one table would lose semantic clarity.
