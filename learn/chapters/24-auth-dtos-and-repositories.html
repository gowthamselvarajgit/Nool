Auth DTOs & Repositories

1️⃣ AuthLoginRequestDto.java — The Login Form

auth/dto/AuthLoginRequestDto.javaJava
package com.nool.backend.auth.dto;

import jakarta.validation.constraints.NotBlank;
import jakarta.validation.constraints.Size;
import lombok.Getter;
import lombok.Setter;

@Getter @Setter
public class AuthLoginRequestDto {

    @NotBlank(message = "Mobile number is required")
    @Size(min = 10, max = 15,
           message = "Mobile number must be valid")
    private String mobileNumber;

    @NotBlank(message = "Password is required")
    private String password;
}

Validation in action:

AnnotationWhat it enforces
@NotBlankField cannot be null, empty, or whitespace-only
@Size(min=10, max=15)Mobile number length between 10 and 15 chars (covers +91 prefix)

When the frontend posts { "mobileNumber": "9876543210", "password": "Admin@123" }, Spring's @Valid annotation in the controller triggers these validators before any business code runs. If the mobile is empty, the request fails at this stage with HTTP 400 — GlobalExceptionHandler (Chapter 09) converts the MethodArgumentNotValidException into a friendly JSON error.

2️⃣ AuthLoginResponseDto.java — The Login Result

auth/dto/AuthLoginResponseDto.javaJava
package com.nool.backend.auth.dto;

import com.nool.backend.enums.Role;
import lombok.Builder;
import lombok.Getter;
import lombok.Setter;

@Getter @Setter
@Builder
public class AuthLoginResponseDto {
    private String token;       // the signed JWT
    private Role   role;        // ADMIN / WORKER / SAREE_OWNER
    private Long   employeeId;  // for workers; null otherwise
    private Long   ownerId;     // for owners; null otherwise
}

Sample JSON returned to the frontend after a successful login:

Login response for RahulJSON
{
  "token":      "eyJhbGciOiJIUzI1NiJ9.eyJ1c2VySWQiOjIs...",
  "role":       "SAREE_OWNER",
  "employeeId": null,
  "ownerId":    1
}
Why does the response duplicate info that's already inside the token? Convenience. The frontend would otherwise have to decode the JWT to know "what role did I just log in as?". Returning role, employeeId, ownerId alongside the token lets the frontend immediately render the correct sidebar and dashboard without parsing anything.

3️⃣ UserRepository.java — Queries on the users Table

repository/auth/UserRepository.javaJava
package com.nool.backend.repository.auth;

import com.nool.backend.auth.entity.User;
import com.nool.backend.enums.Role;
import org.springframework.data.jpa.repository.JpaRepository;
import org.springframework.stereotype.Repository;
import java.util.Optional;

@Repository
public interface UserRepository extends JpaRepository<User, Long> {

    Optional<User> findByMobileNumber(String mobileNumber);
    boolean existsByMobileNumber(String mobileNumber);
    boolean existsByRole(Role role);
}

Three custom finders + the entire JpaRepository toolkit:

MethodGenerated SQLUsed in
findByMobileNumber(mobile)SELECT * FROM users WHERE mobile_number = ?Login (Chapter 25)
existsByMobileNumber(mobile)SELECT 1 FROM users WHERE mobile_number = ? LIMIT 1Duplicate-check before insert (Chapter 26)
existsByRole(role)SELECT 1 FROM users WHERE role = ? LIMIT 1Seed admin check (Chapter 27)
(inherited from JpaRepository)save(), findById(), findAll(), deleteById(), count()Everywhere

How Spring Data generates the SQL

Spring Data JPA parses the method name at startup and generates the SQL automatically. findByMobileNumber breaks down as:

  • find → SELECT a User
  • By → WHERE clause follows
  • MobileNumber → match the field named mobileNumber (maps to column mobile_number)
No SQL written by hand. Spring generates the query, prepares the statement, executes it, and maps the result into a User entity. We just declare the method signature.

4️⃣ UserProfileRepository.java — Queries on user_profiles

repository/auth/UserProfileRepository.javaJava
package com.nool.backend.repository.auth;

import com.nool.backend.auth.entity.UserProfile;
import org.springframework.data.jpa.repository.JpaRepository;
import org.springframework.stereotype.Repository;
import java.util.Optional;

@Repository
public interface UserProfileRepository extends JpaRepository<UserProfile, Long> {
    Optional<UserProfile> findByUserId(Long userId);
    Optional<UserProfile> findByEmployeeId(Long employeeId);
    Optional<UserProfile> findByOwnerId(Long ownerId);
}
MethodUsed for
findByUserId(userId)"Given a User, what's their profile?"
findByEmployeeId(empId)"Given an Employee, find the User who can log in as them"
findByOwnerId(ownerId)"Given a SareeOwner, find their login profile"

Notice the second and third finders use plain Long IDs — that matches the entity from Chapter 12, where employeeId and ownerId are stored as Long not @ManyToOne relationships. The repository method names follow Spring Data's naming convention directly.

🎬 Real-World Scenarios

Scenario 1: Rahul submits the login form
  1. Frontend posts { "mobileNumber": "9000000001", "password": "rahul123" }.
  2. Spring deserializes into an AuthLoginRequestDto.
  3. @Valid kicks in — both fields are present and length is valid.
  4. Controller hands the DTO to AuthServiceImpl.login().
  5. Service calls userRepository.findByMobileNumber("9000000001") → returns the User row.
  6. Service calls user.getUserProfile() (LAZY-loaded), gets ownerId = 1.
  7. Service builds an AuthLoginResponseDto.builder()...build() with token + role + ownerId.
Scenario 2: Empty password — validation rejects early
  1. Frontend posts { "mobileNumber": "9876543210", "password": "" }.
  2. Spring deserializes into the DTO.
  3. @Valid runs @NotBlank on password → fails.
  4. Spring throws MethodArgumentNotValidException.
  5. GlobalExceptionHandler.handleValidation catches it.
  6. Returns HTTP 400: { "status": 400, "error": "Bad Request", "message": "password: Password is required", ... }.
  7. The auth service is never invoked — saves a wasted DB query.
Scenario 3: Seed-admin check at startup
  1. AdminBootstrapRunner (Chapter 27) runs on first boot.
  2. Calls userRepository.existsByRole(Role.ADMIN).
  3. SQL: SELECT 1 FROM users WHERE role = 'ADMIN' LIMIT 1.
  4. Returns false on first boot (empty DB).
  5. Runner inserts the seed admin User row.
  6. On subsequent boots, returns true, so the runner skips.
Common mistakes
Forgetting @Valid on the controller's @RequestBody parameter
Without @Valid, none of the @NotBlank / @Size annotations run. Empty inputs slip through to the service and cause cryptic errors later. Always add @Valid.
Returning the entity directly from the controller instead of a DTO
Returning User would expose the BCrypt-hashed password, timestamps, etc. The DTO is the security boundary — it exposes only what the client should see.
Using List<User> findByMobileNumber instead of Optional<User>
Mobile is UNIQUE in the schema — at most one match. Optional is the correct return type. List would force consumers to handle "what if there are multiple" — which never happens.
Adding business logic to the repository interface
Repositories should only describe queries. Business logic goes in services. Mixing the two makes the repository hard to test and reuse.
Interview questions
Why use Optional instead of returning null?
Optional explicitly forces the caller to handle the "not found" case at compile time. You can't accidentally dereference a null because orElseThrow() or ifPresent() makes the absence-handling explicit. With raw nulls, the caller might forget — leading to NullPointerException in production.
How does Spring Data generate the SQL from a method name?
At startup, Spring parses the interface's method names. findByMobileNumber becomes SELECT FROM User WHERE mobileNumber = ?. The parser knows about keywords like findBy, existsBy, countBy, And, Or, Like, etc. Method names must match field names exactly (camelCase) — typos like "MobilNumber" cause a startup error.
Why are the DTOs in a different package from the entities?
DTOs and entities serve different layers. Entities live near the database; DTOs live near the API. Separating packages signals that DTOs can change shape (for API versioning) without affecting persistence, and vice versa.
