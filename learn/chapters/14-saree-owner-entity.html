🧠 What This Entity Represents

A SareeOwner is one of my parents' clients — the cloth traders and shop owners who bring sarees to the workshop. The class is structurally similar to Employee (Chapter 13): a User link for login + business-specific fields. But the business fields reflect a different role.

Employee (worker)SareeOwner (client)
Earns money from the workshopPays money to the workshop
polishingRate = ₹ paid per saree polishedpolishRatePerSaree = ₹ charged per saree returned
Status: ACTIVE / LEFTStatus: ACTIVE / INACTIVE
Has joining dateNo joining date (we don't track when they became a client)
The workshop's profit per saree is essentially SareeOwner.polishRatePerSaree − Employee.polishingRate. If Rahul pays ₹65 and Prema earns ₹30, the workshop keeps ₹35 per saree. That's why both rates are first-class data, not global constants.

📜 The Full File — Every Line Explained

entity/owner/SareeOwner.java Java 
package com.nool.backend.entity.owner;

import com.nool.backend.auth.entity.User;
import com.nool.backend.enums.OwnerStatus;     // ACTIVE / INACTIVE
import jakarta.persistence.*;
import lombok.*;
import java.time.LocalDateTime;

@Entity
@Table(name = "saree_owner")              // SQL table (note: singular, by historical choice)
@Getter @Setter
@NoArgsConstructor
@AllArgsConstructor
@Builder
public class SareeOwner {

    // ── PRIMARY KEY ───────────────────────────────────────────────
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    @Column(name = "owner_id")
    private Long id;

    // ── OWNER NAME (business name) ────────────────────────────────
    @Column(name = "owner_name", nullable = false)
    private String ownerName;                // "Rahul Textiles", "Sharma Sarees"

    // ── MOBILE NUMBER (same as User's, for quick contact lookup) ──
    @Column(name = "mobile_number",
            unique = true, nullable = false, length = 15)
    private String mobileNumber;

    // ── STATUS ────────────────────────────────────────────────────
    @Enumerated(EnumType.STRING)
    @Column(name = "status", nullable = false)
    private OwnerStatus status;                // ACTIVE = current client; INACTIVE = no longer

    // ── POLISH RATE PER SAREE ★ ───────────────────────────────────
    // Rate (in INR) the owner pays the workshop per returned saree.
    // Different owners can negotiate different rates and the rate may change over time.
    @Column(name = "polish_rate_per_saree", nullable = false)
    @lombok.Builder.Default                     // preserve the default value when using builder()
    private Double polishRatePerSaree = 70.0;  // fallback default ₹70
    // ★ The single most important business field on this entity.
    //   Owner Payable = (sareesReturned × this rate). Each owner has
    //   their own negotiated price.

    // ── LOGIN LINK ────────────────────────────────────────────────
    @OneToOne
    @JoinColumn(name = "user_id", unique = true)
    private User user;
    // Unlike Employee, no explicit nullable=false here either — same pattern.
    // The FK column "user_id" on saree_owner points at users.user_id.

    // ── AUDIT TIMESTAMPS ──────────────────────────────────────────
    @Column(name = "created_at", nullable = false, updatable = false)
    private LocalDateTime createdAt;

    @Column(name = "updated_at")
    private LocalDateTime updatedAt;

    // ── LIFECYCLE HOOKS ───────────────────────────────────────────
    @PrePersist
    protected void onCreate() {
        LocalDateTime now = LocalDateTime.now();
        this.createdAt = now;
        this.updatedAt = now;
    }

    @PreUpdate
    protected void onUpdate() {
        this.updatedAt = LocalDateTime.now();
    }
}

💰 Why Each Owner Has Their Own Rate

Real businesses in India work on relationship-based pricing — not menu prices. My parents have been at this for years, and their pricing reflects every client's history:

Client typeTypical rateWhy
Long-time client (10+ years)₹62 / sareeLoyalty discount, bulk volume
Established client₹65 / sareeReliable monthly business (Rahul)
Standard client₹70 / sareeThe fallback default rate
Premium fabric handling₹75 / sareeExtra care, higher-end sarees
The fallback rate of ₹70 matches the nool.rate-per-saree=70.0 setting from application.properties (Chapter 06). If somehow an owner is created without a rate (shouldn't happen — the column is nullable=false), the application can fall back to that default in downstream calculations.

🧮 How polishRatePerSaree Drives Money

Owner Payable for a period
totalPayable = sareesReturnedInPeriod × owner.polishRatePerSaree

Example with Rahul:

  • Returned sarees this week: 80
  • Rahul's rate: ₹65/saree
  • Total payable: 80 × ₹65 = ₹5,200
  • Already paid (one UPI of ₹3,000 earlier): ₹3,000
  • Pending: ₹5,200 − ₹3,000 = ₹2,200

This calculation happens inside OwnerPaymentServiceImpl (Chapter 48). The service queries saree_ledger for RETURN entries in the period, multiplies the count by the rate, then subtracts the SUM of owner_payment rows in the same period to get the pending amount.

⚠️ The Tiny but Important @Builder.Default

One small annotation that's easy to miss but matters a lot:

The line that preserves the defaultJava
@Column(name = "polish_rate_per_saree", nullable = false)
@lombok.Builder.Default
private Double polishRatePerSaree = 70.0;

Without @Builder.Default, calling SareeOwner.builder().ownerName("...").mobileNumber("...").build() would create an owner with polishRatePerSaree = null — and the INSERT would then fail because the column is nullable=false. With the annotation, Lombok's builder respects the field's default (70.0) when the caller doesn't explicitly set it.

The general rule: any field with a default value AND a non-null DB column needs @Builder.Default.

🎬 Real-World Scenarios

Scenario 1: Dad adds Rahul as a new client
  1. Dad clicks "Add Owner". Form fields:
    • Owner name: Rahul Textiles
    • Mobile: 9000000001
    • Password: rahul123
    • Polish rate: ₹65 (negotiated lower than the ₹70 default)
  2. The owner-service code (Chapter 42):
    • Creates a User row with role SAREE_OWNER, hashed password.
    • Creates a SareeOwner row with status ACTIVE and the negotiated rate.
    • Creates a UserProfile linking them (Chapter 12).
  3. The owner's @PrePersist stamps creation timestamps.
  4. Rahul can now log in from his phone and see his own dashboard.
Scenario 2: Computing how much Rahul owes

It's Friday afternoon. Dad wants to know what Rahul owes for this week.

  1. Service queries: SELECT SUM(quantity) FROM saree_ledger WHERE owner_id=1 AND entry_type='RETURN' AND entry_date BETWEEN '2026-05-11' AND '2026-05-15' → returns 80.
  2. Service multiplies by Rahul's rate: 80 × Rahul.polishRatePerSaree (65.0) = ₹5,200.
  3. Service queries: SELECT SUM(amount_paid) FROM owner_payment WHERE owner_id=1 AND payment_date BETWEEN ... → returns ₹3,000.
  4. Pending = ₹5,200 − ₹3,000 = ₹2,200.
  5. Dashboard shows: "Rahul Textiles · Pending ₹2,200".
Scenario 3: Two clients, two different rates

Same day, two clients pick up sarees:

  • Rahul (rate ₹65) picks up 50 returned sarees → owes ₹3,250
  • Mrs. Sharma (rate ₹75) picks up 30 returned sarees → owes ₹2,250

Each calculation uses that specific owner's rate. The same Java code path serves both — only the rate field differs.

Scenario 4: A client stops doing business

Mr. Khan, an old client, has stopped bringing sarees over the past year. Dad marks him as INACTIVE:

UPDATE saree_owner
   SET status = 'INACTIVE',
       updated_at = '2026-05-15 17:30:00'
 WHERE owner_id = 7;

His ledger history stays in the DB forever (audit, tax). He disappears from the "Active clients" dropdown so Dad can't accidentally record a transaction against him. If he ever comes back, Dad just flips status to ACTIVE again.

Common mistakes around SareeOwner.java
Pulling the rate from application.properties (the global ₹70 default)
Each owner has their OWN rate. Use owner.getPolishRatePerSaree(). The nool.rate-per-saree=70.0 setting is purely a fallback for owners with no explicit rate — not the primary source.
Forgetting @Builder.Default on polishRatePerSaree
Without it, the builder would produce null for the rate, and the INSERT would fail with a NOT NULL violation.
Hard-deleting an owner who has ledger history
Use status = INACTIVE instead. Hard delete breaks the FK from saree_ledger.owner_id and owner_payment.owner_id — historical transactions get orphaned or cascade-destroyed.
Retroactively recomputing old invoices after a rate change
Old owner_payment rows were collected at the old rate. Don't go back and adjust them. New transactions use the new rate; old ones stay as historical fact.
Allowing two owners to share a mobile number
The unique = true on mobile_number blocks this at the DB level. If two owners share a phone, login lookup becomes ambiguous.
Interview questions about SareeOwner.java
Why is the polish rate per-owner instead of per-saree-type or global?
Real business pricing is relationship-based. Long-term clients get loyalty discounts; new clients pay the standard rate; volume buyers negotiate. Storing the rate on the owner gives my parents the flexibility to negotiate individually — and the dashboard automatically uses each owner's rate in calculations without any code changes.
What happens to old ledger entries if I change an owner's rate?
Ledger entries themselves don't change — they just record quantities and dates. But when we recompute totalPayable for that owner, the new rate is multiplied against the count of returned sarees, including older ones in the same period. If we wanted historical pricing to be locked in, we'd need to copy the rate onto each ledger row — but that's not what NOOL does today. The simpler "current rate" model is enough for my parents' workflow.
Why use Double instead of BigDecimal for money?
Yes, BigDecimal is technically safer because it avoids floating-point rounding. But polish rates are always clean whole-rupee values (₹62, ₹65, ₹70, ₹75), and totals come from multiplying integer saree counts. Real-world precision is fine. If NOOL ever supports cents/paise or international currencies, switching to BigDecimal would be the right move.
Why is the table called saree_owner (singular) when other tables are plural?
Honestly, a historical inconsistency. The convention across NOOL leans plural (users, employees, policies), but saree_owner ended up singular. It's a small naming quirk we accepted rather than risk a migration. A future cleanup pass would rename it to saree_owners.
Could I let the polish rate be NULL and just fall back to the global default?
That's how the nool.rate-per-saree setting is designed — it's a fallback. But the column itself is nullable=false because every owner SHOULD have an explicit rate at creation time. The default of 70.0 exists for the builder pattern when callers don't specify a rate. So in practice we always have a rate; the fallback config is just a safety net.
Could a SareeOwner ever exist without a User?
The current schema allows it — the FK on user_id doesn't have an explicit nullable=false on this side. So technically you could create a SareeOwner without a login row, for an offline-only client. The service code, however, always creates both together (User + SareeOwner + UserProfile in one transaction). Without a User, the client can't log in to see their own data — but their inventory still works because the admin operates everything.
What's the difference between Employee.LEFT and SareeOwner.INACTIVE?
Semantically the same — both mean "no longer active". The different word choices match how my parents naturally talk about each: a worker "left the job" but a client "isn't buying anymore". Domain language over technical uniformity. Both serve the same purpose: a soft-delete flag that preserves history.
