1️⃣ AttendanceRepository.java

repository/employee/AttendanceRepository.javaJava
package com.nool.backend.repository.employee;

import com.nool.backend.entity.employee.Attendance;
import com.nool.backend.enums.AttendanceStatus;
import org.springframework.data.domain.Page;
import org.springframework.data.domain.Pageable;
import org.springframework.data.jpa.repository.*;
import org.springframework.stereotype.Repository;
import java.time.LocalDate;
import java.util.List;

@Repository
public interface AttendanceRepository extends JpaRepository<Attendance, Long> {

    // ── A. Lookups by date ───────────────────────────────────────
    List<Attendance> findByAttendanceDate(LocalDate attendanceDate);
    long countByAttendanceDateAndAttendanceStatus(LocalDate attendanceDate,
                                                  AttendanceStatus attendanceStatus);

    // ── B. Per-employee date-range lookups ───────────────────────
    List<Attendance> findByEmployeeIdAndAttendanceDateBetween(Long employeeId,
                                                                LocalDate fromDate,
                                                                LocalDate toDate);

    // ── C. Duplicate-prevention check ────────────────────────────
    boolean existsByEmployeeIdAndAttendanceDate(Long employeeId, LocalDate attendanceDate);

    // ── D. Worker's own paginated history (with eager employee load) ─
    @EntityGraph(attributePaths = "employee")
    Page<Attendance> findByEmployeeId(Long employeeId, Pageable pageable);

    // ── E. Custom JPQL — count present/absent days for one employee ─
    @Query("""
           SELECT COUNT(a)
           FROM Attendance a
           WHERE a.employee.id = :employeeId
             AND a.attendanceStatus = :status
           """)
    long countPresentDaysByEmployee(Long employeeId, AttendanceStatus status);

    // ── F. Filter by employee + date range + status combo ────────
    long countByEmployeeIdAndAttendanceDateBetweenAndAttendanceStatus(
            Long employeeId, LocalDate fromDate, LocalDate toDate, AttendanceStatus attendanceStatus);

    List<Attendance> findByEmployeeIdAndAttendanceDateBetweenAndAttendanceStatus(
            Long employeeId, LocalDate fromDate, LocalDate toDate, AttendanceStatus attendanceStatus);

    // ── G. Bulk delete for an employee (used in deleteEmployee) ──
    @Transactional
    @Modifying
    @Query("DELETE FROM Attendance a WHERE a.employee.id = :employeeId")
    int deleteAllByEmployeeId(Long employeeId);

    // ── H. Workshop-wide aggregates for leave-productivity dashboard ─
    @Query("""
           SELECT a.employee.id, a.attendanceStatus, COUNT(a)
           FROM Attendance a
           WHERE a.attendanceDate BETWEEN :fromDate AND :toDate
           GROUP BY a.employee.id, a.attendanceStatus
           """)
    List<Object[]> aggregateByEmployeeAndStatus(LocalDate fromDate, LocalDate toDate);

    @Query("""
           SELECT COUNT(a)
           FROM Attendance a
           WHERE a.attendanceDate BETWEEN :fromDate AND :toDate
             AND a.attendanceStatus = :status
           """)
    long countByDateRangeAndStatus(LocalDate fromDate, LocalDate toDate, AttendanceStatus status);

    @Query("""
           SELECT a FROM Attendance a
           WHERE a.attendanceDate BETWEEN :fromDate AND :toDate
             AND a.attendanceStatus = :status
           """)
    List<Attendance> findByDateRangeAndStatus(LocalDate fromDate, LocalDate toDate, AttendanceStatus status);
}

A subtle but important annotation — @EntityGraph

The line that prevents N+1 queriesJava
@EntityGraph(attributePaths = "employee")
Page<Attendance> findByEmployeeId(Long employeeId, Pageable pageable);

Without this annotation, fetching a page of 10 attendance rows would cause 10 additional queries — one per row to lazy-load the Employee association. @EntityGraph tells Hibernate to JOIN the employee table in the original query. One query instead of 11.

The N+1 problem: a common JPA performance bug. Without @EntityGraph or JOIN FETCH, every lazy reference triggers a separate query. A page of 100 rows with 3 lazy references = 1 + 100×3 = 301 queries. Always check explain plans on list endpoints.

Modifying queries — the bulk delete

Bulk delete with @ModifyingJava
@Transactional
@Modifying
@Query("DELETE FROM Attendance a WHERE a.employee.id = :employeeId")
int deleteAllByEmployeeId(Long employeeId);
AnnotationPurpose
@ModifyingMarks the @Query as a write (DELETE/UPDATE/INSERT). Without it, JPA assumes SELECT and throws.
@TransactionalRequired for write queries. The transaction is auto-committed on success.
Return type intNumber of rows actually deleted. Useful for logging or sanity checks.

Aggregate queries — returning List<Object[]>

Grouped aggregateJava
@Query("""
       SELECT a.employee.id, a.attendanceStatus, COUNT(a)
       FROM Attendance a
       WHERE a.attendanceDate BETWEEN :fromDate AND :toDate
       GROUP BY a.employee.id, a.attendanceStatus
       """)
List<Object[]> aggregateByEmployeeAndStatus(LocalDate fromDate, LocalDate toDate);

Each Object[] in the result list contains 3 elements: [0]=employeeId (Long), [1]=AttendanceStatus, [2]=count (Long). The service typically iterates and groups these into a structured result.

Why Object[] and not a custom class? JPA returns raw column arrays for grouped aggregates by default. NOOL accepts this simpler form rather than defining a separate projection class. The service immediately maps these to domain DTOs — only the repo layer sees the raw arrays.

Notice employeeName is included even though only employeeId is the actual data. This is a deliberate "flattening" — the frontend renders "Prema · PRESENT" without making a second API call to translate ID 5 into "Prema". One request, all the info needed for display.

5️⃣ AttendanceSummaryDto.java

dto/attendance/AttendanceSummaryDto.javaJava
package com.nool.backend.dto.attendance;

import lombok.Builder;
import lombok.Data;

@Data @Builder
public class AttendanceSummaryDto {
    private Long    employeeId;
    private String  employeeName;
    private Long    totalWorkingDays;       // rows in the range
    private Long    absentDays;             // rows where status = ABSENT
    private Double  attendancePercentage;   // (totalWorkingDays - absentDays) / totalWorkingDays * 100
}

The "summary" view of attendance — aggregated stats over a date range. Sample response:

Sample summary response for Prema, May 1-15JSON
{
  "employeeId":           5,
  "employeeName":         "Prema",
  "totalWorkingDays":     15,
  "absentDays":           2,
  "attendancePercentage": 86.67
}

How attendancePercentage is computed

Attendance percentage
percentage = (totalWorkingDays − absentDays) ÷ totalWorkingDays × 100

With 15 working days and 2 absences: (15 − 2) / 15 × 100 = 86.67%.

"Total working days" here means days the worker has any attendance row. If Prema was hired on May 5 but the query asks about May 1-15, she only has 11 rows (May 5-15), not 15. So "totalWorkingDays" is really "rows that exist in this range for this worker", not "calendar days in the range". The terminology could be confusing — a clearer rename would be recordedDays.

🎬 Real-World Scenarios

Scenario 1: Mom marks Prema's attendance for today
  1. Frontend posts { "employeeId": 5, "attendanceDate": "2026-05-15", "status": "PRESENT" }.
  2. Validation passes (all three @NotNull pass).
  3. Service calls existsByEmployeeIdAndAttendanceDate(5, '2026-05-15').
  4. Returns false → safe to insert.
  5. Insert the row. DB also enforces the UNIQUE constraint as a backstop.
  6. Response: { "attendanceId": 142, "employeeId": 5, "employeeName": "Prema", "attendanceDate": "2026-05-15", "status": "PRESENT" }.
Scenario 2: Computing Prema's May summary
  1. Service queries findByEmployeeIdAndAttendanceDateBetween(5, '2026-05-01', '2026-05-15').
  2. Returns 15 Attendance rows.
  3. Stream filters: 2 with status=ABSENT.
  4. Computes percentage: (15-2)/15 * 100 = 86.67.
  5. Returns the AttendanceSummaryDto.
Scenario 3: Admin dashboard aggregate query
  1. Admin dashboard service calls aggregateByEmployeeAndStatus('2026-05-01', '2026-05-15').
  2. Single SQL: SELECT employee_id, attendance_status, COUNT(*) ... GROUP BY ....
  3. Returns: [[5, PRESENT, 13], [5, ABSENT, 2], [8, PRESENT, 14], [8, ABSENT, 1], ...].
  4. Service groups by employee, computes percentages per worker.
  5. Renders the "Workforce Productivity" table on Dad's dashboard.
Scenario 4: Cascade delete a worker
  1. Admin hard-deletes employee 5 (Prema).
  2. EmployeeServiceImpl.deleteEmployee calls attendanceRepository.deleteAllByEmployeeId(5).
  3. Single DELETE query runs: DELETE FROM attendance WHERE employee_id = 5.
  4. Returns count: 47 (Prema's lifetime attendance rows wiped).
  5. FK constraint now satisfied — the parent Employee row can be deleted next.
Common attendance-repo / DTO mistakes
Forgetting @EntityGraph on paginated list queries
Lazy-loaded associations trigger N+1 queries when rendered. Either add @EntityGraph or use JOIN FETCH in the JPQL.
Skipping the application-level existsBy check and relying only on the DB UNIQUE constraint
Defense in depth: check first in the service for a clean user-facing message; the DB constraint catches anything that slips through. Both layers matter.
Forgetting @Modifying on DELETE / UPDATE @Query methods
Without it, Spring Data thinks it's a SELECT and throws at startup. Always pair @Modifying with @Transactional for write queries.
Storing the computed attendancePercentage in the DB
Compute on demand. Storing derived values risks drift. The summary endpoint recomputes every time it's called — cheap and always accurate.
Returning the Attendance entity directly in a list response
Use AttendanceListResponseDto. Entities lazy-load the Employee and bring its internal fields — DTOs let you precisely control what the frontend sees.
Interview questions
Why does this repo have eleven methods when EmployeeRepository had five?
Attendance is queried in many different shapes — by date, by employee, by date range, by status, by combinations of all of these. The repo declares one method per shape so services don't have to filter in Java. Each method translates directly to an efficient SQL query.
What does @EntityGraph(attributePaths = "employee") actually do?
It modifies the generated SQL to add a JOIN to the employees table and eagerly load the Employee association along with each Attendance row. Without this, accessing attendance.getEmployee().getName() later would trigger a separate SELECT per row — the classic N+1 problem.
Why does the summary DTO not have a "PRESENT days" field?
Because it's derivable: presentDays = totalWorkingDays − absentDays. Including both is redundant. The percentage already encodes the same info more compactly. If the frontend wants the explicit count, it computes it locally.
What's the difference between method-name queries and @Query ones here?
Method-name parsing handles simple cases (existsBy, findBy with up to a couple of fields). @Query is used for grouped aggregates, COUNT with multiple conditions, and DELETE statements — places where the method name would become unreadable or where the parser doesn't support the SQL construct.
Why is deleteAllByEmployeeId a custom JPQL DELETE instead of using deleteByEmployeeId?
Spring Data's auto-derived deleteByX methods load the rows first, then delete them one-by-one — slow for large sets. The custom @Modifying @Query executes one bulk DELETE directly. For deleting a worker's 1000+ attendance rows, the bulk version is dramatically faster.
