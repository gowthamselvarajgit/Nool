A bird's-eye view of how NOOL is put together — the three big pieces, how they talk to each other, and what tools/libraries we use at each layer. Once you have this mental map, every code file will make sense.

🏗️ The Three Big Pieces

NOOL is split into three independent halves that talk to each other. Think of them as three separate "places", each with a different job:

🌐

1. Frontend

The website my parents see in the browser — buttons, forms, calendars, charts. Built with React. Lives in frontend/.

⚙️

2. Backend

The "brain". Stores data, checks who you are, calculates totals, enforces rules. Spring Boot. Lives in backend/.

🗄️

3. Database

The permanent memory. Stores every saree, every payment, every worker. MySQL. Database name: nool_db.

📡 How They Talk To Each Other

The three pieces communicate using just two technologies:

Frontend ↔ Backend: HTTP + JSON

When Dad clicks "Save" on a form, the React app makes an HTTP request to the backend, sending the form data as JSON. The backend processes it and sends back a JSON response. Every request also carries a JWT token (the "ID card") in the Authorization header so the backend knows who's asking.

A real HTTP request from the frontend HTTP 
POST /api/inventory/receipt HTTP/1.1
Host: localhost:8083
Content-Type: application/json
Authorization: Bearer eyJhbGciOiJIUzI1NiJ9.eyJ1c2VySWQiOjEsInJv...

{
  "ownerId": 1,
  "entryDate": "2026-05-11",
  "quantity": 100,
  "remarks": "Diwali batch"
}

Backend ↔ Database: JDBC (via Hibernate)

The backend never writes raw SQL by hand. Instead, it uses Hibernate (an ORM — Object-Relational Mapper) which translates Java method calls into SQL automatically. So when the service writes sareeLedgerRepository.save(entry), Hibernate produces an INSERT INTO saree_ledger ... statement and sends it to MySQL over JDBC. Saves us from writing hundreds of SQL strings by hand.

🎯 A Key Choice — Stateless Backend

One of the most important architectural choices in NOOL: the backend doesn't remember anyone. There's no "logged-in users" table. No session memory. After every request finishes, the backend forgets you exist.

How does login work then? Every request carries a JWT token that contains the user's identity. The backend just verifies the token (checks its signature), reads the user info out of it, and processes the request. This is called stateless authentication.

Why stateless matters:
  • I can run multiple backend instances behind a load balancer — they don't need to share state
  • If the server restarts, no one is "logged out" mid-action (their token still works)
  • No database writes on login (BCrypt comparison is pure CPU work, very fast)
  • The same auth model works for web, mobile apps, third-party integrations — all the same JWT

🥞 The Backend's Internal Layers

Inside the backend itself, code is organized into four layers. Each layer only talks to the one directly below it. This is the classic "MVC + Service" pattern.

LayerFolderJobExample file
Controller controller/ Receives HTTP request, returns HTTP response. Doesn't contain business logic. SareeInventoryController.java
Service service/ The business logic. Validations, calculations, workflows. Where the brain lives. SareeInventoryServiceImpl.java
Repository repository/ Talks to the database. Just CRUD operations. No business logic. SareeLedgerRepository.java
Entity entity/ Java classes that map to database tables. Plain data, no behavior. SareeLedgerEntry.java

Plus two supporting layers:

  • DTO (Data Transfer Object) — folder dto/ — the shape of data sent over HTTP. Separated from entities for security and validation.
  • Exception — folder exception/ — custom error types + a global handler that converts them to clean HTTP responses.

🔍 A Complete Request — Top to Bottom

Let's trace one complete request: Dad clicks "Add Receipt" to record 100 sarees from Rahul. Here's every layer that gets touched:

  1. React: The form's onSubmit handler calls inventoryService.addReceipt(data) in api.js.
  2. api.js: Attaches the JWT token as Bearer ... header, sends POST /api/inventory/receipt.
  3. Vite dev server: Proxies /api/* to localhost:8083.
  4. JwtAuthenticationFilter: Reads the token, verifies its signature, sets the security context to "admin".
  5. SecurityConfig: Checks the rule for POST /inventory/receipt — requires ADMIN. Passes.
  6. SareeInventoryController: Receives the deserialized DTO, calls sareeInventoryService.addReceipt(dto).
  7. SareeInventoryServiceImpl: Locks the owner row, validates the date, builds a SareeLedgerEntry.
  8. SareeLedgerRepository: Calls save(entry). Hibernate generates the SQL.
  9. MySQL: Executes INSERT INTO saree_ledger ... Returns the new row's ID.
  10. Response flows back up: Entity → DTO → JSON → HTTP 200 → React.
  11. React: Shows a green "Saved!" toast, refreshes the inventory list.

Total time: usually under 100 milliseconds. The user sees no delay. That's why we organize code into layers — each layer does its small job and delegates the rest.

💭 Why These Choices?

Why Spring Boot?

Spring Boot is the most popular Java backend framework. It has built-in support for security, database, REST APIs, validation — everything I needed for NOOL. The @SpringBootApplication annotation alone replaces hundreds of lines of manual server configuration. And every Indian tech company uses it, so it's job-friendly too.

Why MySQL (not PostgreSQL or MongoDB)?

MySQL is mature, free, fast, and runs on a laptop. Our data is highly relational (workers ↔ attendance ↔ daily work ↔ salary, all linked by foreign keys), so a SQL database is the right fit — not a document store like MongoDB. PostgreSQL would also work; MySQL is just more familiar in the Indian ecosystem.

Why React + Vite (not Next.js or Angular)?

NOOL is a single-page app (SPA), not a SEO-driven public website. We don't need server-side rendering. Vite + React is dramatically faster to develop with — hot reloads happen in milliseconds — and the bundle is small and fast. Angular felt too heavy for a focused tool like this.

Why Tailwind CSS?

Utility-first CSS lets me style components directly in JSX without flipping between CSS files. For a solo developer building 30+ pages quickly, this is a huge productivity win. Yes, the markup looks busy at first — but the trade-off is worth it.

Why JWT (not server-side sessions)?

Stateless. Easier to scale. Same model works for future mobile apps or third-party integrations. The only real downside (you can't revoke a token until it expires) is acceptable for a small workshop with trusted users.

📏 Important Conventions

A few project-wide rules to keep in mind as you read the code:

  • Time zone: Everything stored and shown in Asia/Kolkata (IST). No UTC anywhere visible to users.
  • Currency: All money in Indian Rupees (₹) using Intl.NumberFormat('en-IN') (which produces "1,23,456" not "123,456").
  • Date strings: The frontend uses a helper called toLocalISODate() to avoid a sneaky timezone bug (covered in Chapter 61).
  • Data integrity: Inventory operations use database row-level locking to prevent concurrent receipt/return races (covered in Chapter 45).
  • Soft delete: Disabling something (a worker, an owner) never deletes the row — we flip a status flag instead. History is preserved.
  • Ledger pattern: Inventory is an append-only ledger (one row per event). Balance is always derived, never stored.
