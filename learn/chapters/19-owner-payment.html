🧠 Mirror Image of SalaryPayment

Structurally this is very similar to SalaryPayment from the last chapter — but the flow of money is reversed, and the structure is slightly simpler.

SalaryPayment (out)OwnerPayment (in)
DirectionWorkshop → WorkerClient → Workshop
Linked entityEmployeeSareeOwner
Work-period range?Yes (fromDate, toDate)No — single date only
Amount column nameamount_paidamount_received
Java field nameamountPaidamountPaid ← same field name!
Payment modeCASH / UPI / BANK_TRANSFERCASH / UPI / BANK_TRANSFER
Curious quirk: The Java field is called amountPaid on both entities, but on OwnerPayment it maps to the SQL column amount_received. That's a small naming inconsistency between layers — the Java field name was inherited from a shared pattern, but the DB column name was renamed to reflect the direction. Most code only sees the Java name, so it rarely matters; but if you write a raw SQL query, remember it's amount_received.

Why no work-period range?

Owner payments don't naturally map to a "work period". When Rahul sends ₹3,000 via UPI, he's not necessarily paying for a specific batch — he might be paying down a running balance, an advance on the next batch, or just a partial settlement. So NOOL records just the single payment date, and the dashboard uses the SUM of payments vs. SUM of (returnedSarees × ratePerSaree) to compute the pending amount.

entity/owner/OwnerPayment.java Java 
package com.nool.backend.entity.owner;

import com.nool.backend.enums.PaymentMode;     // CASH / UPI / BANK_TRANSFER
import jakarta.persistence.*;
import lombok.*;
import java.time.LocalDate;
import java.time.LocalDateTime;

@Entity
@Table(name = "owner_payments")             // SQL table = owner_payments
@Getter @Setter
@NoArgsConstructor
@AllArgsConstructor
@Builder
public class OwnerPayment {

    // ── PRIMARY KEY ───────────────────────────────────────────────
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    @Column(name = "payment_id")
    private Long id;

    // ── PAYMENT DATE ──────────────────────────────────────────────
    @Column(name = "payment_date", nullable = false)
    private LocalDate paymentDate;
    // When the money actually arrived (UPI timestamp, cash hand-off date).

    // ── AMOUNT RECEIVED ───────────────────────────────────────────
    @Column(name = "amount_received", nullable = false)
    private Double amountPaid;
    // ★ Field is called amountPaid in Java but the DB column is amount_received.
    //   Java perspective: "the amount paid by the owner".
    //   DB perspective: "the amount we received". Both are accurate; both
    //   describe the same event from different sides.

    // ── HOW THEY PAID ─────────────────────────────────────────────
    @Enumerated(EnumType.STRING)
    @Column(name = "payment_mode", nullable = false)
    private PaymentMode paymentMode;        // CASH / UPI / BANK_TRANSFER

    // ── REMARKS (optional) ────────────────────────────────────────
    @Column(name = "remarks")
    private String remarks;
    // "UPI ref: 1234567890", "Cheque #45611", "Partial — balance next week".

    // ── OWNER LINK ────────────────────────────────────────────────
    @ManyToOne
    @JoinColumn(name = "owner_id", nullable = false)
    private SareeOwner owner;
    // Many payments belong to one owner — across time.

    // ── AUDIT TIMESTAMPS ──────────────────────────────────────────
    @Column(name = "created_at", nullable = false, updatable = false)
    private LocalDateTime createdAt;

    @Column(name = "updated_at")
    private LocalDateTime updatedAt;

    // ── LIFECYCLE HOOKS ───────────────────────────────────────────
    @PrePersist
    protected void onCreate(){
        LocalDateTime now = LocalDateTime.now();
        this.createdAt = now;
        this.updatedAt = now;
    }

    @PreUpdate
    protected void onUpdate(){
        this.updatedAt = LocalDateTime.now();
    }
}

CREATE TABLE owner_payments (
    payment_id      BIGINT      NOT NULL AUTO_INCREMENT,
    owner_id        BIGINT      NOT NULL,
    payment_date    DATE        NOT NULL,
    amount_received DOUBLE      NOT NULL,             -- note: column name differs from Java field
    payment_mode    VARCHAR(20) NOT NULL,
    remarks         VARCHAR(255),
    created_at      DATETIME(6) NOT NULL,
    updated_at      DATETIME(6),

    PRIMARY KEY (payment_id),
    FOREIGN KEY (owner_id) REFERENCES saree_owner(owner_id)
);

SELECT SUM(amount_received)
FROM   owner_payments
WHERE  owner_id = 1
  AND  payment_date BETWEEN '2026-05-01' AND '2026-05-31';

-- Total cash received across all clients today:
SELECT SUM(amount_received)
FROM   owner_payments
WHERE  payment_mode = 'CASH'
  AND  payment_date = CURDATE();

🧮 How Owner-Pending Is Computed

The formula combines THREE entities we've already covered: SareeLedgerEntry (for returned counts), SareeOwner (for the rate), and OwnerPayment (for the amount already received):

Pending amount owed by one client
pending = (returnedSarees × owner.polishRatePerSaree) − SUM(owner_payment.amount_received)

Worked example for Rahul as of May 17, 2026:

StepValue
Total sarees returned to Rahul (from saree_ledger)70
Rahul's rate per saree₹65
Total payable: 70 × ₹65₹4,550
Total payments received from Rahul₹3,000
Pending₹1,550

🎬 Real-World Scenarios

Scenario 1: Rahul sends a UPI payment
  1. 4:30 PM. Dad gets a UPI notification: "Rahul Textiles paid ₹3,000".
  2. Dad opens NOOL → Admin → Owner Payments → Add.
  3. Picks Rahul Textiles, amount 3000, date today, mode UPI, remark "UPI ref: 1234567890".
  4. Backend inserts a new row in owner_payments.
  5. Dashboard updates: Rahul's pending drops by ₹3,000 immediately.
Scenario 2: Partial cash payment
  1. Mrs. Sharma comes to the workshop on May 17 with ₹2,000 cash, says "I'll send the rest by Friday".
  2. Dad records: owner=Mrs Sharma, amount=2000, mode=CASH, remark="Partial — balance Friday".
  3. Two days later Friday arrives. She sends another ₹1,500 via UPI.
  4. Dad records a SECOND row: amount=1500, mode=UPI, remark="Balance payment".
  5. The dashboard SUMs both rows for her — total paid = ₹3,500.

Two rows for one client in one week is totally normal — no UNIQUE constraint on (owner_id, payment_date) for exactly this reason.

Scenario 3: Reconciling cash at month-end
  1. End of May. Mom wants to confirm "how much cash did we collect this month?".
  2. Query: SELECT SUM(amount_received) FROM owner_payments WHERE payment_mode='CASH' AND payment_date BETWEEN '2026-05-01' AND '2026-05-31'.
  3. Result: ₹18,500.
  4. Mom counts the cash box: ₹18,500. Match! Or if not, she has a row-by-row list to track the discrepancy.
Common mistakes around OwnerPayment
Naming the Java field amountReceived to match the DB column
The current Java field is amountPaid (mirroring SalaryPayment for code consistency). Renaming would mean updating service code, DTOs, controllers, frontend — a much bigger change. The slight asymmetry between Java name and SQL name is accepted.
Storing the "payable for this payment" amount
Don't pre-compute. Pending is always derived from ledger + payments. Storing it would invite drift.
Adding a UNIQUE constraint on (owner_id, payment_date)
Multiple payments per client per day are legitimate (partial settlements). UNIQUE would block real-world workflows.
Forgetting to record the payment_mode for a UPI receipt
Always set it. Mode drives reconciliation. nullable=false on the column actively prevents leaving it blank.
Hard-deleting a row to "remove a wrong payment"
Insert a corrective row with a negative amount + explanatory remark. The audit trail stays clean. (NOOL's current implementation allows admin delete, but conservative practice prefers compensating rows.)
Interview questions about OwnerPayment
Why does the Java field amountPaid map to the SQL column amount_received?
Two perspectives, both correct. The Java field name amountPaid mirrors SalaryPayment's field of the same name — useful for code consistency and shared DTO patterns. The SQL column amount_received describes it from the workshop's side: this is money we received. Both names accurately describe the same event. The mismatch is just historical and harmless in practice.
Why no fromDate/toDate like SalaryPayment?
Because client payments don't map cleanly to specific work periods. Rahul might pay for the running balance, advance on a future batch, or partial settlement. The pending calculation already uses the SareeLedger to compute "amount owed up to today", so there's no need to record per-payment work-period boundaries.
Can the same client pay twice on the same day?
Yes — there's no UNIQUE constraint stopping it. Partial payments are real-world common (₹2,000 cash on arrival, ₹1,500 UPI later). The salary calculation SUMs the rows, so multiple rows are fine. Each row records one discrete payment event.
How do you handle a refund (workshop pays the owner back)?
The current schema doesn't support negative amounts directly (no constraint, but no UI flow either). For a refund, the cleanest pattern is to insert a corrective row with a negative amount + explanatory remark. A more disciplined design would have a separate OwnerRefund entity, but that's beyond NOOL's current scope.
How is "pending amount" computed for the dashboard?
Three queries combined: (1) SUM of RETURN entries from saree_ledger for the owner; (2) the owner's polishRatePerSaree; (3) SUM of amount_received from this table for the owner. Pending = (1) × (2) - (3). The OwnerPaymentService (Chapter 48) does this in Java.
What happens to OwnerPayment rows when an owner is marked INACTIVE?
Nothing — the rows stay forever. Status flip on SareeOwner only affects UI dropdowns and active-list filters. The historical payment records remain for audit, tax, and settling any outstanding balance. The FK from owner_payments.owner_id stays valid.
