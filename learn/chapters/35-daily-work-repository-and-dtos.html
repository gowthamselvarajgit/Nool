🧠 Where This Module Fits

Recall from earlier chapters: Attendance answered "did Prema show up today?". Daily Work answers "what did she produce while she was here?" — two numbers per day: freshCount (brand-new sarees polished) and rePolishCount (touch-up polishes). Multiplied by her per-saree polish rate, this is the entire input to the salary formula. So this module is on the critical path from "Prema polishes" → "Prema gets paid".

The whole salary system depends on these rows being accurate. If Dad mis-types Prema's count by 1, her salary changes by ₹30. If he forgets to log a day entirely, she loses that day's earnings. The repository methods we'll see below include defensive aggregates (COALESCE(SUM(...), 0)) precisely because "Sum of nothing" is a real case we need to handle gracefully.

1️⃣ EmployeeDailyWorkRepository.java

repository/employee/EmployeeDailyWorkRepository.javaJava
package com.nool.backend.repository.employee;

import com.nool.backend.entity.employee.EmployeeDailyWork;
import org.springframework.data.domain.Page;
import org.springframework.data.domain.Pageable;
import org.springframework.data.jpa.repository.*;
import org.springframework.stereotype.Repository;
import java.time.LocalDate;
import java.util.List;

@Repository
public interface EmployeeDailyWorkRepository
        extends JpaRepository<EmployeeDailyWork, Long> {

    // ── A. Lookups by employee ──────────────────────────────────
    List<EmployeeDailyWork> findByEmployeeId(Long employeeId);

    @EntityGraph(attributePaths = "employee")
    Page<EmployeeDailyWork> findByEmployeeId(Long employeeId, Pageable pageable);

    // ── B. Lookups by date ──────────────────────────────────────
    List<EmployeeDailyWork> findByWorkDate(LocalDate workDate);
    List<EmployeeDailyWork> findByWorkDateBetween(LocalDate fromDate, LocalDate toDate);

    // ── C. Combined employee + date range ───────────────────────
    List<EmployeeDailyWork> findByEmployeeIdAndWorkDateBetween(
            Long employeeId, LocalDate fromDate, LocalDate toDate);

    // ── D. Sum aggregates by date (single-day stats) ────────────
    @Query("""
           SELECT COALESCE(SUM(w.freshCount), 0)
           FROM EmployeeDailyWork w
           WHERE w.workDate = :workDate
           """)
    Long sumFreshWorkByDate(LocalDate workDate);

    @Query("""
           SELECT COALESCE(SUM(w.rePolishCount), 0)
           FROM EmployeeDailyWork w
           WHERE w.workDate = :workDate
           """)
    Long sumRePolishWorkByDate(LocalDate workDate);

    // ── E. Sum aggregates by date range ─────────────────────────
    @Query("""
           SELECT COALESCE(SUM(w.freshCount), 0)
           FROM EmployeeDailyWork w
           WHERE w.workDate BETWEEN :fromDate AND :toDate
           """)
    Long sumFreshWorkByDateRange(LocalDate fromDate, LocalDate toDate);

    @Query("""
           SELECT COALESCE(SUM(w.rePolishCount), 0)
           FROM EmployeeDailyWork w
           WHERE w.workDate BETWEEN :fromDate AND :toDate
           """)
    Long sumRePolishWorkByDateRange(LocalDate fromDate, LocalDate toDate);

    // ── F. Bulk delete for cascade ──────────────────────────────
    @Transactional
    @Modifying
    @Query("DELETE FROM EmployeeDailyWork w WHERE w.employee.id = :employeeId")
    int deleteAllByEmployeeId(Long employeeId);

    // ── G. Per-employee aggregate for a date range ──────────────
    @Query("""
           SELECT w.employee.id,
                  COALESCE(SUM(w.freshCount), 0),
                  COALESCE(SUM(w.rePolishCount), 0),
                  COUNT(DISTINCT w.workDate)
           FROM EmployeeDailyWork w
           WHERE w.workDate BETWEEN :fromDate AND :toDate
           GROUP BY w.employee.id
           """)
    List<Object[]> aggregateByEmployee(LocalDate fromDate, LocalDate toDate);

    // ── H. All-time aggregate per employee ──────────────────────
    @Query("""
           SELECT w.employee.id,
                  COALESCE(SUM(w.freshCount), 0),
                  COALESCE(SUM(w.rePolishCount), 0)
           FROM EmployeeDailyWork w
           GROUP BY w.employee.id
           """)
    List<Object[]> aggregateByAllEmployees();
}

Why every SUM uses COALESCE

A typical SUM in this repoJPQL
SELECT COALESCE(SUM(w.freshCount), 0)
FROM EmployeeDailyWork w
WHERE w.workDate = :workDate

The problem: If there are no matching rows (e.g., a date with no polishing done), SQL's SUM returns NULL, not 0. JPA returns that null to Java as null Long, which then NullPointerExceptions the moment anything tries to use it for arithmetic.

The fix: COALESCE(SUM(...), 0) says: "if SUM is null, give me 0 instead". Now the Java code can safely do repository.sumFreshWorkByDate(today) * rate without any null guard.

Defensive aggregation pattern. Anywhere you SUM across rows that might be empty, wrap in COALESCE. The alternative (handling null in Java) clutters every call site. Put the safety in the query and call sites stay clean.

The aggregate query that returns Object[]

Per-employee aggregate for a date rangeJPQL
SELECT w.employee.id,                                  // Object[0] = Long  empId
       COALESCE(SUM(w.freshCount), 0),                // Object[1] = Long  totalFresh
       COALESCE(SUM(w.rePolishCount), 0),             // Object[2] = Long  totalRepolish
       COUNT(DISTINCT w.workDate)                     // Object[3] = Long  distinctWorkdays
FROM EmployeeDailyWork w
WHERE w.workDate BETWEEN :fromDate AND :toDate
GROUP BY w.employee.id

The four columns in the SELECT come back as a List<Object[]> — each row's array has 4 elements in the order shown. Callers iterate and cast:

How a caller reads the resultJava
List<Object[]> rows = repo.aggregateByEmployee(fromDate, toDate);
for (Object[] row : rows) {
    Long empId           = (Long) row[0];
    Long totalFresh      = (Long) row[1];
    Long totalRepolish   = (Long) row[2];
    Long distinctWorkdays = (Long) row[3];
    // build a per-employee summary DTO
}

Slightly verbose but pragmatic. The alternative — defining a projection interface or record — works too but adds a new class for a one-use shape. NOOL accepts the Object[] pattern here.

Why COUNT(DISTINCT w.workDate) matters

Notice the aggregate doesn't just count rows — it counts distinct work dates. Why? Because in the (rare) edge case where a worker has multiple rows for one day (which shouldn't happen due to the UNIQUE constraint, but defensive code never hurts), a plain COUNT(*) would inflate the workday count. DISTINCT workDate gives the true number of unique working days. Belt and braces.

package com.nool.backend.dto.dailywork;

import jakarta.validation.constraints.NotNull;
import jakarta.validation.constraints.PositiveOrZero;
import lombok.Data;
import java.time.LocalDate;

@Data
public class EmployeeDailyWorkRequestDto {

    @NotNull(message = "Employee id is required")
    private Long employeeId;

    @NotNull(message = "Word date is required")
    private LocalDate workDate;

    @NotNull(message = "Fresh saree count is required")
    @PositiveOrZero(message = "Fresh saree count cannot be negative")
    private Integer freshCount;

    @NotNull(message = "Re-polish saree count is required")
    @PositiveOrZero(message = "Re-polish saree count cannot be negative")
    private Integer rePolishCount;
}
FieldValidationWhy
employeeId@NotNullMust point at a worker
workDate@NotNullNeed to know which day
freshCount@NotNull, @PositiveOrZeroCannot be null or negative — but ZERO is valid (worker only did re-polish)
rePolishCount@NotNull, @PositiveOrZeroSame as fresh — zero is valid

Why @PositiveOrZero and not @Positive?

A worker might have done only fresh-polish on a given day (zero re-polish), or vice versa. @PositiveOrZero allows zero; @Positive would reject it, forcing Dad to skip entry entirely. Allowing zero gives Dad a way to mark "Prema worked today but did zero re-polish" without leaving an ambiguous gap.

The interesting bit — todayRevenue

Unlike most response DTOs which just expose stored fields, this one includes a computed value: todayRevenue. The service calculates it at response time using the worker's polish rate.

todayRevenue computation in the service
todayRevenue = freshCount × employee.polishRate
An interesting quirk — only freshCount is in the formula! Looking at the actual service code (Chapter 36), the revenue is computed as freshCount * polishRate, NOT (freshCount + rePolishCount) * polishRate. That might be intentional (re-polish doesn't earn revenue because the saree was already paid for), or it might be a subtle bug. Either way, the design choice currently treats rePolishCount as informational-only for revenue purposes. This is worth flagging in a code review.

🎬 Real-World Scenarios

Scenario 1: Dad logs Prema's Friday work
  1. End of day. Dad asks Prema "how many today?". She says "10 fresh, 2 re-polish".
  2. Frontend posts { "employeeId": 5, "workDate": "2026-05-15", "freshCount": 10, "rePolishCount": 2 }.
  3. Validation passes (all four @NotNull pass; both counts are ≥ 0).
  4. Service inserts a row. UNIQUE constraint on (employee_id, work_date) ensures no duplicate.
  5. Response includes todayRevenue: 300.0 (10 × ₹30).
Scenario 2: Prema views her own daily-work history
  1. Frontend POST /api/employee-daily-working/my-list with pagination.
  2. Service uses findByEmployeeId(5, pageable) — the @EntityGraph variant — to load rows + employee in one query.
  3. Each row mapped to ListDto with the computed todayEarning.
  4. Frontend renders a paginated table.
Scenario 3: Workshop-wide stats for Diwali week
  1. Admin dashboard service calls sumFreshWorkByDateRange('2026-10-19', '2026-10-25').
  2. SQL runs: SELECT COALESCE(SUM(fresh_saree_count), 0) FROM employee_daily_work WHERE work_date BETWEEN ....
  3. Returns: 540 (workshop-wide total fresh sarees in Diwali week).
  4. Same with re-polish. Dashboard cards display both.
  5. Even if no work was logged in that range, COALESCE saves us from a NullPointerException.
Scenario 4: Generating a per-employee productivity report
  1. Admin → Leave-productivity dashboard for May 1-15.
  2. Service calls aggregateByEmployee('2026-05-01', '2026-05-15').
  3. SQL returns one row per worker: [5, 104, 26, 13], [8, 96, 14, 12], ...
  4. Service casts each Object[] to typed values, builds a summary DTO per worker.
  5. Dashboard renders a sortable table — Prema 104 fresh, Ravi 96 fresh, etc.
Scenario 5: Cascade-delete when removing a worker
  1. Admin hard-deletes employee 5.
  2. EmployeeServiceImpl.deleteEmployee calls dailyWorkRepository.deleteAllByEmployeeId(5).
  3. Single SQL: DELETE FROM employee_daily_work WHERE employee_id = 5.
  4. All of Prema's daily-work history vanishes — required before the parent Employee row can be deleted.
Common daily-work-repo / DTO mistakes
Forgetting COALESCE on SUM queries
A SUM over empty rows returns NULL → JPA gives you a null Long → NullPointerException in callers. COALESCE(SUM(...), 0) defends against this at the query level.
Using @Positive instead of @PositiveOrZero on counts
Zero is a valid count. A worker might have done only re-polish (zero fresh) or vice-versa. @Positive would force Dad to fudge the numbers.
Computing revenue as (fresh + rePolish) × rate in the response DTO
Current code only multiplies freshCount × rate. If your business rule is different, change BOTH the response DTO computation AND the summary computation — they must agree.
Using COUNT(*) instead of COUNT(DISTINCT workDate)
If two rows ever existed for one (employee, date), COUNT(*) double-counts the workday. DISTINCT workDate is defensive — same result usually, but safe if the UNIQUE constraint ever lapses.
Forgetting @EntityGraph on the paginated findByEmployeeId
Without it, every row triggers a separate query to load the Employee — classic N+1. The annotation forces a single JOIN.
Returning the entity directly from a list endpoint
List DTOs include the computed todayEarning — not stored on the entity. Mapping in the service is the right place to compute it.
Interview questions
Why does this repo have BOTH single-day SUM methods and date-range SUM methods?
Different dashboards need different aggregates. The single-day version powers "today's total polishing" cards on the admin home; the range version powers monthly / weekly revenue charts. Separating them lets each query be optimised individually. Both use COALESCE for the same null-safety reason.
Why return Object[] from aggregate queries instead of a projection class?
Pragmatism. Defining a typed projection class (record or interface) makes the call site cleaner but adds a new class for a one-use shape. Object[] keeps the API minimal — the service immediately maps the array into a domain DTO. The trade-off: slightly less type-safe at the JPA boundary, but only one place needs to know the ordering.
If a worker submits 0 fresh and 0 repolish, should a row be inserted?
Technically yes — the validation allows it. In practice, Dad wouldn't normally do this; it's just informational. The salary calculation would credit zero earnings for that day. If you wanted to forbid all-zero rows, you'd add a custom validator (@AssertTrue on a derived "is non-zero" method).
What does @EntityGraph(attributePaths = "employee") actually generate?
A LEFT OUTER JOIN to the employees table in the same SQL that retrieves the EmployeeDailyWork rows. The associated Employee object is fully loaded for every returned row — no lazy loading later. For paginated list endpoints that render the worker's name, this avoids N+1 queries.
Why is todayRevenue a Double and not BigDecimal?
Consistency with the rest of NOOL — polishRate is Double, owner rates are Double, salary amounts are Double. For whole-rupee values multiplied by integer counts, the precision loss is negligible. If we ever support paise-level precision or multi-currency, switching to BigDecimal would be a project-wide migration.
