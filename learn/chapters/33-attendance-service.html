@Service
@RequiredArgsConstructor
public class AttendanceServiceImpl implements AttendanceService {

    private final AttendanceRepository attendanceRepository;
    private final EmployeeRepository   employeeRepository;

    // ── Private helper — turn Attendance entity into a list-row DTO ─
    private AttendanceListResponseDto mapToListDto(Attendance attendance) {
        return AttendanceListResponseDto.builder()
                .attendanceId(attendance.getId())
                .employeeId(attendance.getEmployee().getId())
                .employeeName(attendance.getEmployee().getName())
                .attendanceDate(attendance.getAttendanceDate())
                .status(attendance.getAttendanceStatus())
                .build();
    }
    // Tiny helper extracted because it's used in multiple list methods.
    // DRY pays off when you see the same 8 lines repeated otherwise.

3️⃣ markAttendance() — Recording PRESENT/ABSENT

AttendanceServiceImpl.markAttendance()Java
@Override
public AttendanceResponseDto markAttendance(AttendanceRequestDto requestDto) {

    // ── STEP 1: Verify employee exists ───────────────────────────
    Employee employee = employeeRepository.findById(requestDto.getEmployeeId())
            .orElseThrow(() -> new ResourceNotFoundException("Employee not found"));

    // ── STEP 2: Prevent double-marking ───────────────────────────
    if (attendanceRepository.existsByEmployeeIdAndAttendanceDate(
            employee.getId(), requestDto.getAttendanceDate())) {
        throw new DuplicateResourceException(
                "Attendance is already marked for " + employee.getName() +
                " on " + requestDto.getAttendanceDate());
    }

    // ── STEP 3: Build and save ───────────────────────────────────
    Attendance attendance = Attendance.builder()
            .employee(employee)
            .attendanceDate(requestDto.getAttendanceDate())
            .attendanceStatus(requestDto.getStatus())
            .build();

    Attendance savedAttendance = attendanceRepository.save(attendance);

    // ── STEP 4: Return the response DTO ──────────────────────────
    return AttendanceResponseDto.builder()
            .attendanceId(savedAttendance.getId())
            .employeeId(employee.getId())
            .employeeName(employee.getName())
            .attendanceDate(savedAttendance.getAttendanceDate())
            .status(savedAttendance.getAttendanceStatus())
            .build();
}
Two layers of duplicate protection: the explicit existsBy... check up front PLUS the database's UNIQUE constraint on (employee_id, attendance_date). The application-level check gives a friendly error message ("Attendance is already marked for Prema on 2026-05-15"); the DB constraint catches any race that slips past the check.

@Override
public AttendanceResponseDto getAttendanceById(Long attendanceId) {
    Attendance attendance = attendanceRepository.findById(attendanceId)
            .orElseThrow(() -> new ResourceNotFoundException("Attendance not found"));

    return AttendanceResponseDto.builder()
            .attendanceId(attendance.getId())
            .employeeId(attendance.getEmployee().getId())
            .employeeName(attendance.getEmployee().getName())
            .attendanceDate(attendance.getAttendanceDate())
            .status(attendance.getAttendanceStatus())
            .build();
}

Straightforward. Note: no role check here. SecurityConfig allows both ADMIN and WORKER to call GET /attendance/{id}. A WORKER could theoretically retrieve another worker's attendance row by guessing IDs — a potential weakness compared to the strict check in getEmployeeById. In practice, the frontend doesn't expose this capability to workers, but a stricter implementation would replicate the self-or-admin pattern here too.

5️⃣ getAttendanceList() — Admin View

AttendanceServiceImpl.getAttendanceList()Java
@Override
public PaginationResponseDto<AttendanceListResponseDto> getAttendanceList(PaginationRequestDto paginationRequestDto) {
    PageRequest pageRequest = PageRequest.of(
            paginationRequestDto.getPage(),
            paginationRequestDto.getSize(),
            Sort.Direction.valueOf(paginationRequestDto.getSortingDirection()),
            paginationRequestDto.getSortBy()
    );

    Page<Attendance> page = attendanceRepository.findAll(pageRequest);

    List<AttendanceListResponseDto> content = page.getContent()
            .stream()
            .map(this::mapToListDto)        // reuse the helper
            .collect(Collectors.toList());

    return PaginationResponseDto.<AttendanceListResponseDto>builder()
            .content(content)
            .page(page.getNumber())
            .size(page.getSize())
            .totalElements(page.getTotalElements())
            .totalPages(page.getTotalPages())
            .last(page.isLast())
            .build();
}

Admin endpoint — no filtering, returns the whole table paginated. Used for the global attendance browser.

6️⃣ getMyAttendanceList() — Worker's Own List

AttendanceServiceImpl.getMyAttendanceList()Java
@Override
public PaginationResponseDto<AttendanceListResponseDto> getMyAttendanceList(PaginationRequestDto paginationRequestDto) {
    Long employeeId = CurrentUserUtil.getEmployeeId();    // read from JWT
    if (employeeId == null) {
        throw new RuntimeException("Access denied");    // caller isn't a worker
    }

    PageRequest pageRequest = PageRequest.of(
            paginationRequestDto.getPage(),
            paginationRequestDto.getSize(),
            Sort.Direction.valueOf(paginationRequestDto.getSortingDirection()),
            paginationRequestDto.getSortBy()
    );

    Page<Attendance> page = attendanceRepository.findByEmployeeId(employeeId, pageRequest);
    // ★ This calls the @EntityGraph-annotated method (Chapter 32),
    //   so the Employee association loads in the same query — no N+1.

    List<AttendanceListResponseDto> content = page.getContent()
            .stream()
            .map(this::mapToListDto)
            .collect(Collectors.toList());

    return PaginationResponseDto.<AttendanceListResponseDto>builder()
            .content(content)
            .page(page.getNumber())
            .size(page.getSize())
            .totalElements(page.getTotalElements())
            .totalPages(page.getTotalPages())
            .last(page.isLast())
            .build();
}

The worker version reads their own ID from the JWT — they can never accidentally view someone else's rows.

7️⃣ getAttendanceSummaryByEmployee() — The Self-or-Admin Pattern

AttendanceServiceImpl.getAttendanceSummaryByEmployee()Java
@Override
public AttendanceSummaryDto getAttendanceSummaryByEmployee(Long employeeId, DateRangeDto dateRangeDto) {

    // ── STEP 1: Permission check — self or admin only ────────────
    String role = CurrentUserUtil.getRole();
    if (!"ADMIN".equals(role)) {
        Long caller = CurrentUserUtil.getEmployeeId();
        if (caller == null || !caller.equals(employeeId)) {
            throw new org.springframework.security.access.AccessDeniedException(
                    "You can only view your own attendance");
        }
    }

    // ── STEP 2: Verify employee exists ───────────────────────────
    Employee employee = employeeRepository.findById(employeeId)
            .orElseThrow(() -> new ResourceNotFoundException("Employee not found"));

    // ── STEP 3: Fetch all attendance rows in the range ───────────
    List<Attendance> attendances = attendanceRepository
            .findByEmployeeIdAndAttendanceDateBetween(
                    employeeId, dateRangeDto.getFromDate(), dateRangeDto.getToDate());

    // ── STEP 4: Compute stats with Java streams ──────────────────
    long totalWorkingDays = attendances.size();
    long absentDays = attendances.stream()
            .filter(a -> a.getAttendanceStatus() == AttendanceStatus.ABSENT)
            .count();
    double attendancePercentage = totalWorkingDays == 0
            ? 0
            : ((double) (totalWorkingDays - absentDays) / totalWorkingDays) * 100;

    // ── STEP 5: Build the response DTO ───────────────────────────
    return AttendanceSummaryDto.builder()
            .employeeId(employeeId)
            .employeeName(employee.getName())
            .totalWorkingDays(totalWorkingDays)
            .absentDays(absentDays)
            .attendancePercentage(attendancePercentage)
            .build();
}
The self-or-admin pattern. When a URL has an explicit employeeId parameter, we need to verify the caller is authorised to see that ID's data. ADMINs see anyone; WORKERs only see their own. This is the third time we've seen this exact pattern (Employee, here, soon Salary). Standard defense-in-depth.

Why the division cast to double?

The cast that mattersJava
((double) (totalWorkingDays - absentDays) / totalWorkingDays) * 100

Without the cast, Java would do integer division: 13 / 15 = 0 (truncated). The cast forces floating-point division: 13.0 / 15.0 = 0.8666.... Crucial for a percentage. The guard against division-by-zero (totalWorkingDays == 0 ? 0 : ...) is the other essential bit — a worker with no attendance rows shouldn't crash the query.

8️⃣ getMyAttendanceSummary()

AttendanceServiceImpl.getMyAttendanceSummary()Java
@Override
public AttendanceSummaryDto getMyAttendanceSummary(DateRangeDto dateRangeDto) {
    Long employeeId = CurrentUserUtil.getEmployeeId();
    if (employeeId == null) {
        throw new RuntimeException("Access denied");
    }

    Employee employee = employeeRepository.findById(employeeId)
            .orElseThrow(() -> new ResourceNotFoundException("Employee not found"));
    List<Attendance> attendances = attendanceRepository
            .findByEmployeeIdAndAttendanceDateBetween(
                    employeeId, dateRangeDto.getFromDate(), dateRangeDto.getToDate());

    long totalWorkingDays = attendances.size();
    long absentDays = attendances.stream()
            .filter(a -> a.getAttendanceStatus() == AttendanceStatus.ABSENT)
            .count();
    double attendancePercentage = totalWorkingDays == 0 ? 0
            : ((double) (totalWorkingDays - absentDays) / totalWorkingDays) * 100;

    return AttendanceSummaryDto.builder()
            .employeeId(employeeId)
            .employeeName(employee.getName())
            .totalWorkingDays(totalWorkingDays)
            .absentDays(absentDays)
            .attendancePercentage(attendancePercentage)
            .build();
}

Basically identical to getAttendanceSummaryByEmployee but with no employeeId parameter — it's always read from the JWT. The duplication is mild but could be refactored into a private helper accepting the resolved ID.

🎬 Real-World Scenarios

Scenario 1: Mom marks today's roll call
  1. For each worker, frontend posts: { "employeeId": X, "attendanceDate": "2026-05-15", "status": "PRESENT" }.
  2. Service finds the employee, checks existsBy → false → inserts.
  3. Returns the response DTO. Frontend ticks the row green in the UI.
Scenario 2: Mom accidentally tries to mark Prema present twice
  1. Second POST for same (employeeId=5, date=2026-05-15).
  2. existsByEmployeeIdAndAttendanceDate → true.
  3. Service throws DuplicateResourceException("Attendance is already marked for Prema on 2026-05-15").
  4. GlobalExceptionHandler → 409 Conflict with the message.
  5. Frontend shows the friendly error to Mom. No duplicate row created.
Scenario 3: Prema views her May calendar
  1. Frontend posts to /api/attendance/my-list with pagination.
  2. Service reads employeeId=5 from JWT.
  3. Repository query loads Prema's rows in a single SQL (thanks to @EntityGraph).
  4. Each row mapped to AttendanceListResponseDto.
  5. Frontend renders the calendar with green/red dots per day.
Scenario 4: Dad pulls Prema's May summary
  1. Dad on admin → "Prema's profile" → "Attendance summary for May".
  2. Frontend posts POST /api/attendance/employee/5/summary with body { "fromDate": "2026-05-01", "toDate": "2026-05-15" }.
  3. Service: role=ADMIN → permission check passes.
  4. Repository returns 15 rows. 2 are ABSENT.
  5. Computes percentage: (15-2)/15 × 100 = 86.67.
  6. Returns summary DTO: Prema · 15 working days · 2 absent · 86.67%.
Scenario 5: Ravi tries to view Prema's summary
  1. Ravi (WORKER, employeeId=8) calls POST /api/attendance/employee/5/summary.
  2. SecurityConfig allows the URL pattern for both ADMIN and WORKER.
  3. Service: role=WORKER → check caller(8).equals(5)? No.
  4. Throws AccessDeniedException("You can only view your own attendance").
  5. Returns 403 Forbidden. Defense-in-depth at the service layer.
Common attendance-service mistakes
Forgetting the cast to double in the percentage calc
(long) / (long) gives integer division — 13/15 = 0. Always cast at least one operand to double.
Skipping the division-by-zero check
If a worker has zero rows in the range, dividing by 0 throws ArithmeticException for integers or produces NaN for doubles — either way, broken response. The ternary guard handles it.
Counting absent days with a second DB query
We already fetched all rows. Java streams filter them locally — no extra round-trip. .stream().filter(...).count() is fast in-memory.
Not validating the employeeId belongs to a real worker before computing
The findById(...).orElseThrow(...) catches typos like "employeeId=999". Without it, the summary would silently return zero counts as if the worker exists with no attendance.
Omitting the role check in getAttendanceSummaryByEmployee
SecurityConfig allows WORKERs to call this endpoint (so they can view their own summary). Without the in-service check, a worker could pass any employeeId. The check narrows access to "self only for WORKERs".
Interview questions
Why isn't markAttendance annotated with @Transactional?
Single-write operations are wrapped in their own transaction by Spring Data automatically. @Transactional matters when one method does multiple writes that need to be atomic — like the cascade-delete in EmployeeService. Single inserts work fine without it.
Why count absent days in Java instead of in SQL?
We already have the list of rows in memory (we needed them for the total count anyway). Counting absentees in Java is one stream operation. Going back to the DB for a separate count would mean two queries instead of one — slower, more complex.
Why does the summary use Double for percentage but Long for counts?
Counts are integers (you can't have 1.5 absent days). Percentages aren't (86.67% would round badly as an integer). The types reflect the natural domain. Java Long and Double serialize cleanly to JSON via Jackson.
Should the percentage be rounded?
The service returns a raw double like 86.6666... The frontend handles formatting ("86.67%"). Doing rounding in the backend would lose precision for any further computations on the value. Format only at the display layer.
Why is the duplicate check done in both the service AND the DB?
Defense in depth. The service check (existsBy...) gives a friendly error message early. The DB UNIQUE constraint is the last line of defense — catches races, bypasses, and any path that skips the service. Both layers matter.
If two requests arrive at exactly the same time to mark Prema, what happens?
One wins, one loses. Both pass the existsBy check (they read at the same moment). Both try to INSERT. The DB UNIQUE constraint fails the second one with DataIntegrityViolationException. GlobalExceptionHandler (Chapter 09) translates it into the friendly "Attendance is already marked..." message. So the worst case is one user sees an error and retries — no data corruption.
