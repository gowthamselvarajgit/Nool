🧠 Why a Separate "User" Class?

You might wonder — why have a User class at all, when we also have Employee (for workers) and SareeOwner (for clients)? Why not just put a password on each of those?

Because login data is fundamentally different from business data.

Login data (User)Business data (Employee / SareeOwner)
Mobile number, password, roleName, joining date, polish rate, status
Changes rarely (only on password reset)Changes often (rate revisions, status flips)
Sensitive (password must be encrypted)Routine (just business facts)
Same for ALL three rolesDifferent per role
Used at every API call (auth check)Used only on relevant screens

Splitting them keeps each class focused on one job. User handles authentication; Employee handles "what's this worker's polish rate?". Mixing them would make each class confusingly multi-purpose.

📜 The Full File — Every Line Explained

auth/entity/User.java Java 
package com.nool.backend.auth.entity;
// ↑ Lives in the auth sub-package, alongside login-related classes
//   (UserProfile, JwtUtil, etc.).

import com.nool.backend.enums.Role;            // ADMIN, WORKER, SAREE_OWNER
import jakarta.persistence.*;                  // JPA annotations (@Entity, @Column, etc.)
import lombok.*;                                // Lombok shortcuts
import java.time.LocalDateTime;                 // for createdAt/updatedAt timestamps

@Entity                       // Tells Hibernate: "manage this as a database table"
@Table(name = "users")        // SQL table name is `users`
@Getter                       // Auto-generate getMobileNumber(), getRole(), etc.
@Setter                       // Auto-generate setMobileNumber(), setRole(), etc.
@NoArgsConstructor            // public User() {} — required by Hibernate for entity hydration
@AllArgsConstructor           // constructor with every field — used by the builder
@Builder                       // enables User.builder().mobileNumber("X").build() pattern
public class User {

    // ── PRIMARY KEY ────────────────────────────────────────────────
    @Id                                                // "this is the unique row identifier"
    @GeneratedValue(strategy = GenerationType.IDENTITY)  // MySQL auto-increment: 1, 2, 3...
    @Column(name = "user_id")
    private Long id;                                    // In Java we call it just `id`; column is user_id

    // ── MOBILE NUMBER (login username) ────────────────────────────
    @Column(name = "mobile_number",
            nullable = false,        // must never be NULL in DB
            unique = true,          // no two users can share the same mobile
            length = 15)             // max 15 chars (covers "+91" + 10 digits + buffer)
    private String mobileNumber;

    // ── PASSWORD (BCrypt hash, NEVER plain text) ──────────────────
    @Column(name = "password", nullable = false, length = 100)
    private String password;
    // length=100 is sized for a BCrypt hash (60 chars + room to spare).
    // The actual plain password from the user is NEVER stored.

    // ── ROLE (ADMIN / WORKER / SAREE_OWNER) ───────────────────────
    @Enumerated(EnumType.STRING)              // store as "ADMIN" not as 0/1/2
    @Column(name = "role", nullable = false)
    private Role role;

    // ── ACTIVE FLAG (soft delete) ─────────────────────────────────
    @Column(name = "active_status", nullable = false)
    @lombok.Builder.Default                    // preserve the default when using builder()
    private boolean active = true;             // false = blocked from logging in

    // ── LINK TO BUSINESS PROFILE (Employee or SareeOwner) ─────────
    @OneToOne(mappedBy = "user", fetch = FetchType.LAZY)
    private UserProfile userProfile;
    // One User has at most one UserProfile, which then points to either an
    // Employee or a SareeOwner. mappedBy = "user" means UserProfile owns
    // the FK column; we just navigate from this side. LAZY = don't load
    // it until someone actually calls user.getUserProfile().

    // ── AUDIT TIMESTAMPS ──────────────────────────────────────────
    @Column(name = "created_at", nullable = false, updatable = false)
    private LocalDateTime createdAt;
    // updatable=false → once set on INSERT, never overwritten by UPDATE

    @Column(name = "updated_at")
    private LocalDateTime updatedAt;
    // Refreshed on every UPDATE via @PreUpdate below

    // ── LIFECYCLE HOOKS ───────────────────────────────────────────
    @PrePersist                                  // fires once, just before the first INSERT
    protected void onCreate(){
        LocalDateTime now = LocalDateTime.now();
        this.createdAt = now;
        this.updatedAt = now;
    }

    @PreUpdate                                   // fires before every UPDATE
    protected void onUpdate(){
        this.updatedAt = LocalDateTime.now();
    }
}

🔍 The Annotations, In Detail

@Entity

Marks a class as a JPA-managed entity. At startup, Hibernate scans the classpath for @Entity classes and builds an in-memory metamodel of every column, type, relationship, etc. The same metamodel is what generates the CREATE TABLE statement when ddl-auto=update is on.

@Table(name = "users")

Sets the SQL table name. Without it, Hibernate would default to the class name ("User" → table user). We pluralize because that's the SQL convention and "user" is a reserved word in some databases (PostgreSQL, etc.). Always explicit.

@Id + @GeneratedValue(IDENTITY)

@Id says "this is the primary key column". GenerationType.IDENTITY tells JPA to let the database itself generate the value (via MySQL's AUTO_INCREMENT). Inserts simply omit the column; MySQL fills it in.

@Column properties

PropertyMeaning
name="..."The SQL column name (lets us use snake_case in DB and camelCase in Java)
nullable=falseMaps to NOT NULL in SQL — DB itself refuses NULL values
unique=trueAdds a UNIQUE constraint — DB refuses duplicates
length=NSets VARCHAR(N) sizing
updatable=falseOnce set, this field is excluded from UPDATE statements

@Enumerated(EnumType.STRING)

Tells Hibernate to store the enum's name (e.g., "ADMIN") as text, not its position (0). Why STRING is safer was covered fully in Chapter 08 — the short version: enum reordering doesn't silently corrupt data.

@OneToOne(mappedBy = "user", fetch = FetchType.LAZY)

This is the most subtle annotation in the file. It has two key pieces:

  • mappedBy = "user" — tells Hibernate the other side (UserProfile) owns the foreign-key column. UserProfile has @JoinColumn(name = "user_id") pointing at the User table; we navigate backward from User by declaring this side as "mapped by" that field. No extra column appears on the users table.
  • fetch = FetchType.LAZY — don't query the UserProfile table just because someone loaded a User. Only fetch it if and when code calls user.getUserProfile(). Saves an extra JOIN on the very common login lookup.

@PrePersist and @PreUpdate

JPA lifecycle hooks. @PrePersist runs once, immediately before the very first INSERT. @PreUpdate runs immediately before every subsequent UPDATE. They let me auto-stamp createdAt and updatedAt without remembering to set them in every service method.

🎬 Real-World Scenarios

Scenario 1: Dad registers Rahul as a new client
  1. Dad clicks "Add Owner", types name "Rahul Textiles", mobile 9000000001, password rahul123, rate ₹65.
  2. The owner-service code:
    • Creates a User with role SAREE_OWNER, BCrypt-hashes the password.
    • Creates a SareeOwner row with the business data.
    • Creates a UserProfile linking them.
  3. The User's @PrePersist fires — createdAt and updatedAt set to now.
  4. Hibernate runs INSERT INTO users (mobile_number, password, role, ...) VALUES (...).
  5. MySQL auto-generates user_id = 2.
  6. From now on, Rahul can log in with his mobile + password.
Scenario 2: A worker leaves the workshop
  1. Suresh, one of the polishers, decides to move to another city.
  2. Dad opens NOOL → Employees → Suresh's row → clicks "Mark as Left".
  3. The service code does TWO things:
    • Flips Suresh's Employee.status to LEFT.
    • Flips Suresh's linked User.active to false.
  4. Hibernate runs UPDATE users SET active_status = false, updated_at = NOW() WHERE user_id = 5.
  5. If Suresh tries to log in: AuthServiceImpl finds his row, BCrypt confirms the password — but then sees active = false and refuses.
  6. All of Suresh's history (attendance, daily-work, salary payments) stays in the DB. Nothing lost.
Scenario 3: Rahul logs in
  1. Rahul opens NOOL on his phone, types mobile 9000000001 and password rahul123.
  2. Backend runs SELECT * FROM users WHERE mobile_number = '9000000001'.
  3. Finds the row. Reads the stored hash $2a$10$YDg3JL....
  4. BCryptPasswordEncoder.matches("rahul123", "$2a$10$YDg3JL...") → returns true.
  5. Checks user.isActive() → also true.
  6. Reads user.getRole() → SAREE_OWNER.
  7. Lazily loads user.getUserProfile().getOwnerId() → 1.
  8. Builds a JWT with these values, signs it, returns to Rahul's phone.
  9. Rahul lands on the owner dashboard.
Common mistakes around the User entity
Storing the plain password directly in the password field
Always run it through passwordEncoder.encode(...) first. The 100-char column length is sized for the BCrypt hash, not the plain password.
Hard-deleting a user with userRepository.delete(...)
Use user.setActive(false) instead. Hard delete breaks foreign-key relationships from Employee/SareeOwner/UserProfile and can lose audit history.
Forgetting @Builder.Default on the active field
Without it, User.builder().build() would create a user with active = false — they could never log in. The annotation preserves the field's default value when used via the builder.
Removing updatable=false from createdAt
Without it, an UPDATE could silently overwrite the original creation date. Losing creation timestamps destroys audit value.
Setting userProfile to FetchType.EAGER
Login is the most-frequent operation. Eager-fetching the profile on every User load means an extra JOIN every time. LAZY pays off. Pull profile only when actually needed (e.g., when building the JWT claims).
Storing role as a free-text String instead of the Role enum
Strings allow typos. The enum gives compile-time spell-checking and a single source of truth for valid values.
Interview questions about User.java
Why is the password column 100 chars? Passwords are usually shorter than that.
Because the column stores the BCrypt hash, not the plain password. BCrypt hashes are exactly 60 characters (60-byte fixed format including version, cost, salt, hash). We sized 100 to leave buffer room in case we ever upgrade to a different hash algorithm with a longer output. Storing plain passwords would be a security disaster.
Why @Enumerated(EnumType.STRING) and not ORDINAL?
ORDINAL stores the enum's position as an integer (ADMIN=0, WORKER=1, ...). If I ever reorder the enum or insert a new value in the middle, all the historical data silently becomes wrong. STRING stores the name ("ADMIN"), which survives reordering and is human-readable in SQL queries.
What's the difference between nullable=false on the entity and @NotNull on a DTO?
nullable=false is enforced by the database (column-level NOT NULL). @NotNull is enforced by Bean Validation on incoming requests. They serve different layers — entity attributes describe storage rules; DTO attributes describe input rules. Both should exist for any required field, defense in depth.
Why does Hibernate need @NoArgsConstructor?
When Hibernate reads a row from the database, it uses reflection to create the entity object and then sets each field. Reflection-based instantiation requires a no-args constructor. Without it, you'd get cryptic errors at startup. @NoArgsConstructor from Lombok generates it automatically.
What happens if I forget @PrePersist?
createdAt never gets set. The column is nullable=false, so the INSERT throws a SQL error. NOOL's GlobalExceptionHandler would catch it as a DataIntegrityViolationException and return a 500. Adding @PrePersist fixes it permanently. Some teams use the cleaner pattern @CreationTimestamp (Hibernate-specific) instead — does the same thing without writing the method.
If User has a OneToOne to UserProfile, why don't I see a column for it in users?
Because of mappedBy = "user". That declares this side as the inverse — the FK column lives on the other side (UserProfile.user_id). Without mappedBy, Hibernate would create FK columns on both sides, duplicating the relationship. The mappedBy tells Hibernate "I don't own the column; navigate from there".
Could I use @Data instead of separate @Getter @Setter?
You could, but it's not recommended for JPA entities. @Data also generates equals() and hashCode() based on ALL fields, including relationships — which can cause infinite recursion if two entities reference each other (User ↔ UserProfile in our case). Using @Getter @Setter explicitly keeps hashCode/equals at their JVM-default behavior, which is what JPA expects.
