AdminBootstrapRunner —
Creating the very first admin

Every admin in NOOL is created by another admin. So who creates the first one? This file solves that chicken-and-egg problem by inserting a seed admin row the very first time NOOL boots on an empty database.

🧠 The Problem This Solves

NOOL's rule: only admins can create new admins. But on a brand-new install, there are zero admins. With no way to log in, my parents could never use the app. So we need a one-time bootstrap mechanism that creates the first admin automatically.

The trick: a Spring CommandLineRunner bean that runs once after the app starts. On the first boot, no admin exists → insert one using credentials from application.properties. On every subsequent boot, an admin already exists → do nothing. Idempotent: safe to run any number of times, only the first run has effect.

📜 The Full File — Every Line Explained

auth/bootstrap/AdminBootstrapRunner.java Java 
package com.nool.backend.auth.bootstrap;

import com.nool.backend.auth.entity.User;
import com.nool.backend.repository.auth.UserRepository;
import com.nool.backend.enums.Role;
import lombok.RequiredArgsConstructor;
import org.springframework.boot.CommandLineRunner;
import org.springframework.beans.factory.annotation.Value;
import org.springframework.security.crypto.password.PasswordEncoder;
import org.springframework.stereotype.Component;

@Component                                          // Spring picks this up automatically
@RequiredArgsConstructor                            // constructor for final fields
public class AdminBootstrapRunner implements CommandLineRunner {

    private final UserRepository  userRepository;
    private final PasswordEncoder passwordEncoder;

    @Value("${admin.mobile}")
    private String adminMobile;                       // from application.properties

    @Value("${admin.password}")
    private String adminPassword;                     // from application.properties

    @Override
    public void run(String... args){              // runs ONCE after Spring Boot starts

        // ── STEP 1: Is there already an admin? ───────────────────
        boolean adminExists = userRepository.existsByRole(Role.ADMIN);
        if (adminExists) {
            return;                                   // idempotent — do nothing
        }

        // ── STEP 2: First boot — create the seed admin ──────────
        User admin = new User();
        admin.setMobileNumber(adminMobile);                           // "9876543210"
        admin.setPassword(passwordEncoder.encode(adminPassword));     // BCrypt-hash "Admin@123"
        admin.setRole(Role.ADMIN);
        admin.setActive(true);

        userRepository.save(admin);                                   // INSERT into users

        // ── STEP 3: Friendly console message ─────────────────────
        System.out.println("✅ DEFAULT ADMIN ACCOUNT CREATED SUCCESSFULLY");
    }
}

⚙️ What's a CommandLineRunner?

It's a special Spring interface with one method: run(String... args). When Spring Boot finishes starting up (after all beans are created, Tomcat is listening, etc.), it calls run() on every bean implementing this interface. Perfect for one-time startup tasks like:

  • Seeding initial data (NOOL's case)
  • Validating configuration
  • Warming up caches
  • Logging startup info
You can have multiple CommandLineRunner beans — Spring runs them all. If order matters, annotate them with @Order(1), @Order(2), etc. NOOL has just this one, so order isn't a concern.

🔄 Why Idempotent?

The existsByRole(Role.ADMIN) check is what makes this safe to run on every startup. Let's walk through both cases:

Boot scenarioWhat happens
First ever boot — empty DB existsByRole returns false → create admin → log success → continue.
Every subsequent boot existsByRole returns true → early return → no insertion → no log.
Admin was disabled and re-enabled Doesn't matter — existsByRole only checks the role column, not active.
All admins deleted from DB manually existsByRole returns false → seed admin recreated. NOOL self-heals.

📋 Why Use the Direct Repository Instead of AdminUserService?

Notice this runner uses userRepository.save(admin) directly, NOT adminUserService.createEmployeeUser(...). Three reasons:

  1. No UserProfile needed. Admins don't have a UserProfile (they aren't workers or owners). AdminUserService always creates a profile too.
  2. No role check needed. AdminUserService hardcodes WORKER or SAREE_OWNER. We need ADMIN here.
  3. No duplicate check needed. We already know there's no admin (just checked existsByRole).

So the runner uses raw repository calls — minimal, direct, and bypasses logic that doesn't apply.

🗄️ The Single INSERT on First Boot

-- On the very first startup with empty DB:

-- Step 1: existsByRole check (empty table = no rows = false)
SELECT CASE WHEN COUNT(*) > 0 THEN 1 ELSE 0 END
  FROM users WHERE role = 'ADMIN';
-- → 0 (no admin yet)

-- Step 2: insert the seed admin
INSERT INTO users
    (mobile_number, password, role, active_status, created_at, updated_at)
VALUES
    ('9876543210',
     '$2a$10$abcdef1234567890...',  -- BCrypt hash of "Admin@123"
     'ADMIN',
     true,
     '2026-05-15 10:30:00',
     '2026-05-15 10:30:00');

-- Console prints: "✅ DEFAULT ADMIN ACCOUNT CREATED SUCCESSFULLY"

-- ───────────────────────────────────────────────────────────
-- On every subsequent startup:
SELECT CASE WHEN COUNT(*) > 0 THEN 1 ELSE 0 END
  FROM users WHERE role = 'ADMIN';
-- → 1 (admin exists), early return, no INSERT

📌 The Seed Credentials

The mobile and password come from application.properties (Chapter 06):

From application.propertiesproperties
admin.mobile=9876543210
admin.password=Admin@123
Production deployment must override these! The hard-coded Admin@123 is intentionally easy for local development. For a real deployment, set environment variables ADMIN_MOBILE and ADMIN_PASSWORD BEFORE first boot — Spring's property resolution will pick them up and Mom and Dad's first admin will have a strong, unique password.

🎬 Real-World Scenarios

Scenario 1: Brand-new install
  1. I clone the NOOL repo to a fresh laptop with empty MySQL.
  2. Run ./mvnw spring-boot:run.
  3. Spring Boot starts, Hibernate auto-creates the schema (ddl-auto=update).
  4. Tomcat starts on port 8083.
  5. Spring invokes every CommandLineRunner — AdminBootstrapRunner.run() fires.
  6. existsByRole(ADMIN) → false. Runner proceeds to insert.
  7. Console: "✅ DEFAULT ADMIN ACCOUNT CREATED SUCCESSFULLY".
  8. I open http://localhost:5173, log in with 9876543210 / Admin@123. Works!
Scenario 2: Routine restart
  1. I stop and restart NOOL after a code change.
  2. Same startup flow.
  3. existsByRole(ADMIN) → true (already exists from earlier boot).
  4. Runner returns immediately. No insert, no console log.
  5. App is up. Existing admin can log in.
Scenario 3: Production deploy with override
  1. Production server has environment variables: 
    ADMIN_MOBILE=9123456789
ADMIN_PASSWORD=Str0ngR@nd0mPass!2026
  2. NOOL starts. Spring's @Value resolves to those env vars (overrides the file).
  3. Runner inserts admin with the strong production password.
  4. Dad logs in with the production credentials only he knows.
  5. Local-dev Admin@123 never works in production. Safe.
Scenario 4: Database corrupted, admin row lost
  1. Some catastrophe wipes the User row for the admin.
  2. NOOL is restarted.
  3. existsByRole(ADMIN) → false (no admin row anymore).
  4. Runner inserts a fresh seed admin — recovery without any manual SQL.
  5. Note: in production, you'd hope to restore from a backup first. But the runner is the safety net.
Common mistakes
Hard-coding the password directly in this file
Read from application.properties via @Value — exactly what NOOL does. This lets production override without code changes.
Forgetting to BCrypt-hash the password before saving
passwordEncoder.encode(adminPassword) is essential. Without it, the password column would store plain text — a massive security hole.
Skipping the existsByRole check
Without the check, every restart would try to insert another admin — fails due to UNIQUE constraint on mobile, but you'd see ugly stack traces in the log on every boot.
Leaving the default Admin@123 password in production
Override via environment variable BEFORE first deploy. Once the admin is created with the seed password, it stays until manually changed — anyone with the README knows it.
Putting the runner in a deeply nested package that auto-scan might miss
The runner lives under com.nool.backend.auth.bootstrap — covered by @SpringBootApplication's component scan from com.nool.backend. Always verify your runner is in a scanned package.
Interview questions
When does CommandLineRunner.run() get called?
After Spring Boot's startup is fully complete — all beans created, dependency injection done, Tomcat listening — but before the JVM blocks on the main thread. So by the time run() fires, the app is fully ready to serve HTTP requests (though no real ones have arrived yet).
What's the difference between CommandLineRunner and ApplicationRunner?
Nearly identical. CommandLineRunner.run(String...) takes a raw String array; ApplicationRunner.run(ApplicationArguments) takes a parsed ApplicationArguments object that distinguishes option args from non-option args. For simple seeding tasks like NOOL's, CommandLineRunner is fine.
Could you do the same thing with a SQL migration script?
Yes — a Flyway or Liquibase migration could INSERT the admin row. The trade-off: you'd lose the BCrypt-at-startup hashing (you'd have to pre-compute the hash and paste it into the SQL), and password rotation via env vars wouldn't work. Java-side seeding is more flexible for credentials. SQL migrations are still preferred for schema changes.
Why does the runner write to System.out instead of using a logger?
Pragmatism. The message is a one-shot success notification, not regular operational logging. Using System.out.println guarantees it shows up in the console even if SLF4J is misconfigured. A more polished version would use log.info("...") via a Logger.
What if multiple instances start at the same time on a shared database?
Race condition possible but harmless. Both instances would see no admin, both try to INSERT. One succeeds; the other hits the UNIQUE mobile constraint and gets an exception (which would surface as a startup error). For NOOL's single-instance deployment this is irrelevant. For a clustered setup, you'd add a database advisory lock around the check + insert.
