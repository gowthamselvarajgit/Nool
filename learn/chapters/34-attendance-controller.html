🧠 The Six Endpoints

HTTPPathPurposeRequired role
POST/api/attendanceMark a single attendance rowADMIN
GET/api/attendance/{attendanceId}Fetch one rowADMIN or WORKER
POST/api/attendance/listPaginated list of ALL attendanceADMIN
POST/api/attendance/my-listWorker's own paginated historyADMIN or WORKER
POST/api/attendance/employee/{id}/summarySummary for one worker over date rangeADMIN or WORKER (self)
POST/api/attendance/summaryWorker's own summaryADMIN or WORKER
The admin-vs-worker pattern:
  • /list → admin sees everyone
  • /my-list → worker sees just self
  • /employee/{id}/summary → admin specifies whose summary
  • /summary → worker gets their own (ID inferred from JWT)
Twin endpoints, one with an explicit ID, one with implicit (caller's own). This URL split lets SecurityConfig assign distinct roles to each variant without complex predicate logic.

controller/employee/AttendanceController.javaJava
package com.nool.backend.controller.employee;

import com.nool.backend.dto.attendance.*;
import com.nool.backend.dto.common.DateRangeDto;
import com.nool.backend.dto.common.PaginationRequestDto;
import com.nool.backend.dto.common.PaginationResponseDto;
import com.nool.backend.service.employee.AttendanceService;
import jakarta.validation.Valid;
import lombok.RequiredArgsConstructor;
import org.springframework.web.bind.annotation.*;

@RestController
@RequestMapping("/attendance")            // every URL: /api/attendance/*
@RequiredArgsConstructor
public class AttendanceController {
    private final AttendanceService attendanceService;

    // ── 1. MARK ATTENDANCE ──────────────────────────────────────
    @PostMapping
    public AttendanceResponseDto markAttendance(
            @Valid @RequestBody AttendanceRequestDto requestDto){
        return attendanceService.markAttendance(requestDto);
    }

    // ── 2. GET ONE BY ID ────────────────────────────────────────
    @GetMapping("/{attendanceId}")
    public AttendanceResponseDto getAttendanceById(
            @PathVariable Long attendanceId){
        return attendanceService.getAttendanceById(attendanceId);
    }

    // ── 3. ADMIN PAGINATED LIST ─────────────────────────────────
    @PostMapping("/list")
    public PaginationResponseDto<AttendanceListResponseDto> getAttendanceList(
            @RequestBody PaginationRequestDto paginationRequestDto){
        return attendanceService.getAttendanceList(paginationRequestDto);
    }

    // ── 4. WORKER'S OWN PAGINATED LIST ──────────────────────────
    @PostMapping("/my-list")
    public PaginationResponseDto<AttendanceListResponseDto> getMyAttendanceList(
            @RequestBody PaginationRequestDto paginationRequestDto){
        return attendanceService.getMyAttendanceList(paginationRequestDto);
    }

    // ── 5. SUMMARY FOR A SPECIFIC EMPLOYEE ──────────────────────
    @PostMapping("/employee/{employeeId}/summary")
    public AttendanceSummaryDto getAttendanceSummary(
            @PathVariable Long employeeId,
            @Valid @RequestBody DateRangeDto dateRangeDto){
        return attendanceService.getAttendanceSummaryByEmployee(employeeId, dateRangeDto);
    }

    // ── 6. WORKER'S OWN SUMMARY ─────────────────────────────────
    @PostMapping("/summary")
    public AttendanceSummaryDto getMyAttendanceSummary(
            @Valid @RequestBody DateRangeDto dateRangeDto){
        return attendanceService.getMyAttendanceSummary(dateRangeDto);
    }
}

⚙️ Why POST for Reads That Take a Body?

Notice /list, /my-list, /employee/{id}/summary, and /summary all use POST even though they're conceptually read operations. Why?

EndpointBody shapeWhy not GET?
/list and /my-listPaginationRequestDto (5 fields)Search keyword could have special chars; pagination fields are awkward in query strings
/employee/{id}/summary and /summaryDateRangeDto (2 LocalDates)Dates could go in path params, but JSON body keeps the contract consistent

NOOL's convention: any endpoint with a request body uses POST, even for reads. This avoids ambiguity and keeps query strings clean.

🛡️ Where Authorization Is Enforced

The controller doesn't have explicit role checks. Authorization happens in TWO places:

  1. SecurityConfig URL rules (Chapter 22) — coarse-grained, by URL pattern. Marking attendance → ADMIN only; reading own attendance → ADMIN or WORKER.
  2. Service-layer self-or-admin check (Chapter 33) — fine-grained, at row level. For /employee/{id}/summary, a worker can call it only with their own ID.

The controller's job is to map HTTP → service method. Authorization is a cross-cutting concern handled elsewhere.

🎬 Real-World Scenarios

Scenario 1: Mom marks the morning roll call
  1. For each worker, frontend sends a POST: { "employeeId": X, "attendanceDate": "2026-05-15", "status": "PRESENT" }.
  2. JWT filter (Chapter 21) verifies → role=ADMIN.
  3. SecurityConfig: POST /attendance → hasAuthority("ADMIN") → passes.
  4. Body deserialized → validation passes.
  5. Controller delegates to service → row inserted → DTO returned.
  6. Frontend ticks each worker green.
Scenario 2: Prema views her own calendar
  1. Frontend sends POST to /api/attendance/my-list with pagination body.
  2. SecurityConfig: POST /attendance/my-list → hasAnyAuthority("ADMIN", "WORKER") → passes.
  3. Service reads employeeId=5 from Prema's JWT.
  4. Returns paginated rows.
  5. Frontend renders the month-view calendar with green/red dots.
Scenario 3: Dad views Prema's May summary
  1. Frontend POSTs to /api/attendance/employee/5/summary with { "fromDate": "2026-05-01", "toDate": "2026-05-15" }.
  2. SecurityConfig: matching rule → ADMIN or WORKER allowed → passes.
  3. Service self-or-admin check: role=ADMIN → passes immediately.
  4. Computes summary → returns DTO.
  5. Dashboard renders: "Prema · 86.67%".
Scenario 4: Ravi tries to view Prema's summary
  1. Ravi (WORKER, employeeId=8) crafts a POST to /api/attendance/employee/5/summary.
  2. SecurityConfig: URL allows ADMIN or WORKER → passes (this is intentional — workers CAN call the endpoint).
  3. Service self-or-admin check: role=WORKER, caller(8) != 5 → throws AccessDeniedException.
  4. Spring Security returns 403 Forbidden.
  5. Defense in depth: SecurityConfig allowed the URL but the service blocked the misuse.
Scenario 5: Worker views own summary via /summary
  1. Prema's app POSTs to /api/attendance/summary with just the date range.
  2. No need to pass her own employeeId — the service reads it from her JWT.
  3. Summary returned: her May percentages.
  4. The URL design (no path variable) makes it impossible to accidentally request someone else's data.

Common attendance-controller mistakes
Adding role checks inside the controller
Authorization belongs in SecurityConfig (URL rules) and the service (row-level). Controllers stay thin.
Forgetting @Valid on @RequestBody
Without it, the @NotNull annotations on DTOs don't run. Null inputs silently slip through to the service.
Using GET with a body for list/summary endpoints
Many HTTP clients (especially older ones) don't reliably support GET with a body. POST is the safe choice.
Putting the employeeId in the request body of /employee/{id}/summary
Path variable + body is the right pattern. The path identifies the target resource; the body provides the query parameters. Mixing them into one large body would be unconventional.
Adding /me-style endpoints that take any ID
/my-list and /summary deliberately take NO employeeId. The ID always comes from the JWT — making it impossible to accidentally request someone else's data.
Interview questions
Why does this controller have BOTH /list and /my-list?
Different audiences, different access scopes. /list is for admins browsing all attendance. /my-list is for workers seeing their own. They share most logic in the service but use different repository queries (findAll vs findByEmployeeId). Separate URLs make role rules in SecurityConfig trivial.
Why isn't there a PUT or PATCH endpoint to edit attendance?
Two reasons. First, attendance is meant to be immutable history — once marked for a day, it shouldn't change easily. Second, the UNIQUE constraint on (employee_id, attendance_date) prevents marking the same day twice anyway. If you really need to fix a wrong status, the practical option is to delete the row (admin only) and remark.
Why use POST for /summary when summaries are read operations?
The body contains the date range. GET with a body is technically allowed but inconsistently supported by HTTP clients. NOOL's convention is: any endpoint requiring a body uses POST. Consistency beats REST purity in a project this size.
How does the URL /employee/{employeeId}/summary get matched?
Spring's path-pattern matcher recognises {employeeId} as a path variable. The @PathVariable Long employeeId annotation in the method signature extracts it. Then the body (DateRangeDto) is deserialized separately. Spring handles both — path and body — and passes the typed values to the method.
Could you combine the two summary endpoints into one method?
Theoretically yes — a single method accepting an optional employeeId, defaulting to the JWT's caller ID. But the explicit split is clearer at the API contract level: /employee/{id}/summary says "specifying someone"; /summary says "self". Cleaner endpoint design beats DRY for HTTP APIs.
