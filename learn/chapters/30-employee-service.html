EmployeeService & Impl

2️⃣ createEmployee() — Adding a New Worker

This is the most interesting method. It does four things:

  1. Validates the mobile is unique among employees
  2. Handles "stale user" cleanup (the orphan-cleanup trick)
  3. Inserts the Employee row
  4. Calls AdminUserService to create the linked login account
EmployeeServiceImpl.createEmployee()Java
@Override
@Transactional                                  // all-or-nothing
public EmployeeResponseDto createEmployee(CreateEmployeeRequestDto requestDto) {
    String mobile = requestDto.getMobileNumber();

    // ── STEP 1: Reject if another Employee has this mobile ──────
    if (employeeRepository.existsByMobileNumber(mobile)) {
        throw new DuplicateResourceException(
            "Employee with this mobile number already exists");
    }

    // ── STEP 2: Stale-User cleanup (advanced) ───────────────────
    // If a login User exists for this mobile, decide whether it's a legitimate
    // conflict (admin, owner, or another linked employee) or a stale orphan
    // from a previously-failed create — orphans get cleaned up so the retry
    // can succeed.
    userRepository.findByMobileNumber(mobile).ifPresent(existingUser -> {
        UserProfile profile = userProfileRepository
                .findByUserId(existingUser.getId()).orElse(null);
        boolean linkedToSomething = profile != null
                && (profile.getEmployeeId() != null || profile.getOwnerId() != null);
        boolean isAdmin = existingUser.getRole() != null
                && existingUser.getRole().name().equals("ADMIN");

        if (linkedToSomething || isAdmin) {
            throw new DuplicateResourceException(
                "This mobile number is already in use by another account");
        }
        // Orphan user — drop it so we can re-register cleanly.
        if (profile != null) {
            userProfileRepository.delete(profile);
        }
        userRepository.delete(existingUser);
        userRepository.flush();
    });

    // ── STEP 3: Insert the Employee row ─────────────────────────
    Employee employee = Employee.builder()
            .name(requestDto.getEmployeeName())
            .mobileNumber(mobile)
            .joiningDate(requestDto.getJoiningDate())
            .polishRate(requestDto.getPolishingRate())
            .status(EmployeeStatus.ACTIVE)        // new workers are ACTIVE
            .build();
    Employee saved = employeeRepository.save(employee);

    // ── STEP 4: Create the User + UserProfile (delegate) ────────
    User user = adminUserService.createEmployeeUser(
        mobile,
        requestDto.getPassword(),
        saved.getId()
    );
    saved.setUser(user);                          // link User back to Employee
    saved = employeeRepository.save(saved);

    // ── STEP 5: Return the response DTO ─────────────────────────
    return EmployeeResponseDto.builder()
            .employeeId(saved.getId())
            .employeeName(saved.getName())
            .mobileNumber(saved.getMobileNumber())
            .joiningDate(saved.getJoiningDate())
            .polishingRate(saved.getPolishRate())
            .status(saved.getStatus().name())
            .build();
}

The orphan-cleanup trick — why it exists

Scenario: Dad tries to create a worker. The User row gets created successfully, but the next step (UserProfile insert) fails due to a DB hiccup. The @Transactional rolls back the UserProfile, but Hibernate sometimes leaves the User row committed before the rollback (depends on the transaction propagation). That orphan User row now blocks any retry with the same mobile.

The fix: on retry, the code detects "this User exists but has no profile and isn't an admin" → that's an orphan from a prior failure → safe to delete and re-create. This is defensive code born from real production debugging.

3️⃣ updateEmployee() — Editing a Worker

EmployeeServiceImpl.updateEmployee()Java
@Transactional
@Override
public void updateEmployee(UpdateEmployeeRequestDto requestDto) {

    // ── STEP 1: Fetch the existing row ──────────────────────────
    Employee employee = employeeRepository.findById(requestDto.getEmployeeId())
            .orElseThrow(() -> new ResourceNotFoundException("Employee not found"));

    // ── STEP 2: If mobile is changing, ensure new one is free ───
    if (!employee.getMobileNumber().equals(requestDto.getMobileNumber())
            && employeeRepository.existsByMobileNumber(requestDto.getMobileNumber())){
        throw new DuplicateResourceException("Mobile number already in use");
    }
    if (!employee.getMobileNumber().equals(requestDto.getMobileNumber())
            && userRepository.existsByMobileNumber(requestDto.getMobileNumber())) {
        throw new DuplicateResourceException(
            "A login account with this mobile number already exists");
    }

    // ── STEP 3: Apply the changes to the Employee row ───────────
    employee.setName(requestDto.getEmployeeName());
    employee.setPolishRate(requestDto.getPolishingRate());
    employee.setMobileNumber(requestDto.getMobileNumber());

    // ── STEP 4: Sync the mobile on the linked User row too ──────
    User user = employee.getUser();
    if (user == null) {                                // fallback: lookup via profile
        user = userProfileRepository.findByEmployeeId(employee.getId())
                .map(profile -> profile.getUser())
                .orElse(null);
    }
    if (user != null) {
        user.setMobileNumber(requestDto.getMobileNumber());
        userRepository.save(user);
        employee.setUser(user);
    }

    employeeRepository.save(employee);
}

Two important details:

  • Only check duplicates if mobile is actually changing. If Dad is editing Prema's name but keeping her mobile the same, we DON'T trigger a uniqueness check that would find her own existing row.
  • Mobile is synced to BOTH Employee and User. They're stored separately (denormalized) — if we only updated one, the two tables would drift. The login table is the source of truth for auth; the Employee mobile is for quick contact display. They must match.

4️⃣ updateEmployeeStatus() — Mark as Left / Reactivate

EmployeeServiceImpl.updateEmployeeStatus()Java
@Transactional
@Override
public void updateEmployeeStatus(EmployeeStatusUpdateDto requestDto) {
    Employee employee = employeeRepository.findById(requestDto.getEmployeeId())
            .orElseThrow(() -> new ResourceNotFoundException("Employee not found"));
    employee.setStatus(requestDto.getStatus());     // ACTIVE or LEFT
    employeeRepository.save(employee);
}

Tiny method but important — it's the "soft delete" mechanism for workers. When Suresh leaves the workshop, Dad calls this with { employeeId: 7, status: LEFT }. The row stays in the DB (preserving history); only the flag flips.

What this method does NOT do: it doesn't also disable the User's active flag. So even after status=LEFT, the worker could technically still log in. A stricter implementation would also flip User.active=false here. Right now NOOL relies on the status flag for UI filtering (the user would just see an empty dashboard).

5️⃣ getEmployeeById() — With Permission Check

EmployeeServiceImpl.getEmployeeById()Java
@Override
public EmployeeResponseDto getEmployeeById(Long employeeId) {

    // ── A WORKER can only read their own profile, never another employee's ─
    String role = CurrentUserUtil.getRole();
    if (!"ADMIN".equals(role)) {
        Long callerEmployeeId = CurrentUserUtil.getEmployeeId();
        if (callerEmployeeId == null || !callerEmployeeId.equals(employeeId)) {
            throw new org.springframework.security.access.AccessDeniedException(
                "You can only view your own profile");
        }
    }

    // ── Permission check passed — fetch and return ──────────────
    Employee employee = employeeRepository.findById(employeeId)
            .orElseThrow(() -> new ResourceNotFoundException("Employee not found"));

    return EmployeeResponseDto.builder()
            .employeeId(employee.getId())
            .employeeName(employee.getName())
            .joiningDate(employee.getJoiningDate())
            .polishingRate(employee.getPolishRate())
            .mobileNumber(employee.getMobileNumber())
            .status(employee.getStatus().name())
            .build();
}
Defense in depth at work. SecurityConfig already allows the endpoint to be called by either ADMIN or WORKER. But this service adds a finer check: "if you're a WORKER, the ID you're asking for must match YOUR id". Without this, Prema could call GET /api/employees/8 and read Ravi's polish rate. Two layers of protection: URL-level (SecurityConfig) and row-level (here).

6️⃣ getEmployeeList() — Paginated, Searchable

EmployeeServiceImpl.getEmployeeList()Java
@Override
public PaginationResponseDto<EmployeeListResponse> getEmployeeList(PaginationRequestDto paginationRequestDto) {

    // ── STEP 1: Build the Spring Data PageRequest ───────────────
    PageRequest pageRequest = PageRequest.of(
            paginationRequestDto.getPage(),
            paginationRequestDto.getSize(),
            Sort.Direction.valueOf(paginationRequestDto.getSortingDirection()),
            paginationRequestDto.getSortBy()
    );

    // ── STEP 2: Search if keyword present, else findAll ─────────
    String keyword = paginationRequestDto.getSearchKeyword();
    Page<Employee> page = (keyword != null && !keyword.isBlank())
            ? employeeRepository.searchEmployees(keyword, pageRequest)
            : employeeRepository.findAll(pageRequest);

    // ── STEP 3: Map entities to DTOs ────────────────────────────
    return PaginationResponseDto.<EmployeeListResponse>builder()
            .content(
                page.getContent().stream().map(employee ->
                    EmployeeListResponse.builder()
                            .employeeId(employee.getId())
                            .employeeName(employee.getName())
                            .mobileNumber(employee.getMobileNumber())
                            .joiningDate(employee.getJoiningDate())
                            .polishingRate(employee.getPolishRate())
                            .status(employee.getStatus().name())
                            .build()
                ).collect(Collectors.toList())
            )
            .page(page.getNumber())
            .size(page.getSize())
            .totalElements(page.getTotalElements())
            .totalPages(page.getTotalPages())
            .last(page.isLast())
            .build();
}

Pure orchestration — builds a PageRequest, calls the right repo method (search or findAll), maps entities to DTOs, wraps in a PaginationResponseDto.

The frontend can call this without sending any ID. The backend just reads employeeId from the JWT — guaranteed to be the caller's own. Cannot be tricked into returning someone else's profile.

8️⃣ deleteEmployee() — The Cascading Deletion

EmployeeServiceImpl.deleteEmployee()Java
@Override
@Transactional
public void deleteEmployee(Long employeeId) {
    Employee employee = employeeRepository.findById(employeeId)
            .orElseThrow(() -> new ResourceNotFoundException(
                    "Employee not found with id: " + employeeId));

    // ── 1. Cascade-delete all child records that reference employee_id ──
    //    MySQL won't let us drop the parent row while children exist.
    dailyWorkRepository.deleteAllByEmployeeId(employeeId);
    salaryPaymentRepository.deleteAllByEmployeeId(employeeId);
    attendanceRepository.deleteAllByEmployeeId(employeeId);

    // ── 2. Unlink the user from the employee, then delete profile + user ──
    UserProfile profile = userProfileRepository.findByEmployeeId(employeeId).orElse(null);
    User user = employee.getUser();
    if (user == null && profile != null) {
        user = profile.getUser();
    }
    if (user != null) {
        employee.setUser(null);
        employeeRepository.save(employee);
        employeeRepository.flush();
    }
    if (profile != null) {
        userProfileRepository.delete(profile);
        userProfileRepository.flush();
    }
    if (user != null) {
        userRepository.delete(user);
    }

    // ── 3. Finally drop the employee row ─────────────────────────
    employeeRepository.delete(employee);
}

Why is delete so complicated?

Because the Employee row is the parent of five dependent tables:

  • attendance — FK employee_id
  • employee_daily_work — FK employee_id
  • salary_payments — FK employee_id
  • user_profiles — has employee_id column (not strict FK, but logical link)
  • users — has the linked User row to clean up

MySQL's foreign-key constraints prevent dropping a row that has children. So we must delete the children first, in the right order, then the Employee. The flush() calls force Hibernate to actually run the DELETEs immediately instead of batching them — important because the order matters here.

Hard delete is RARE in NOOL. The normal flow is "mark as LEFT" via updateEmployeeStatus, which preserves history. Hard-delete should only be used to remove test data or fix mistakes — and it irreversibly destroys all the worker's history. The GlobalExceptionHandler (Chapter 09) even has a specific message for the foreign-key violation case, urging the admin to use "Mark as Left" instead.

🎬 Real-World Scenarios

Scenario 1: Dad adds Prema as a new worker
  1. Dad clicks "Add Employee" → form with name "Prema", mobile 9000000002, joining date, rate ₹30, password.
  2. Frontend posts to POST /api/employees with CreateEmployeeRequestDto.
  3. Controller → createEmployee().
  4. Step 1: existsByMobileNumber("9000000002") → false.
  5. Step 2: no existing User with that mobile → orphan-cleanup branch doesn't trigger.
  6. Step 3: insert Employee row → returns id=5.
  7. Step 4: adminUserService.createEmployeeUser("9000000002", "prema123", 5) creates User + UserProfile.
  8. Step 5: Employee row updated with User pointer. Saved again.
  9. Response DTO returned: { employeeId: 5, employeeName: "Prema", ..., status: "ACTIVE" }.
  10. Dad sees Prema in the employee list. Prema can now log in with her phone.
Scenario 2: Prema gets a raise
  1. Dad opens Prema's row, edits her polishing rate from 30.0 to 32.0.
  2. Frontend posts to PUT /api/employees with { employeeId: 5, employeeName: "Prema", polishingRate: 32.0, mobileNumber: "9000000002" }.
  3. Step 1: Employee #5 found.
  4. Step 2: mobile NOT changing → skip duplicate checks.
  5. Step 3: employee.setPolishRate(32.0).
  6. Step 4: User mobile NOT changing → no User update needed (but we still execute the save — small inefficiency).
  7. Hibernate UPDATE runs. From May 16 onwards, Prema's daily earnings use ₹32.
Scenario 3: Prema views her own profile
  1. Prema's app calls GET /api/employees/me on the worker dashboard.
  2. Controller → getMyProfile().
  3. Reads employeeId = 5 from her JWT via CurrentUserUtil.getEmployeeId().
  4. Fetches Employee #5, builds DTO, returns it.
  5. Prema's UI shows her name, joining date, rate, status.
Scenario 4: Prema tries to view Ravi's profile via URL manipulation
  1. Prema (employeeId=5) manually calls GET /api/employees/8 (Ravi's ID).
  2. SecurityConfig allows it (GET /employees/** → ADMIN or WORKER).
  3. Service: getEmployeeById(8).
  4. Permission check: role is WORKER (not ADMIN). Check callerEmployeeId(5) vs requestedId(8) → mismatch.
  5. Throws AccessDeniedException("You can only view your own profile").
  6. Spring Security returns 403 Forbidden. Cross-worker access prevented at the service layer.
Scenario 5: Dad tries to delete a worker who has history
  1. Dad accidentally clicks "Delete" on Prema instead of "Mark as Left".
  2. Controller → deleteEmployee(5).
  3. Step 1: Employee found.
  4. Step 2a: dailyWorkRepository.deleteAllByEmployeeId(5) — wipes ALL her daily work history.
  5. Step 2b: salaryPaymentRepository.deleteAllByEmployeeId(5) — wipes ALL her salary records.
  6. Step 2c: attendanceRepository.deleteAllByEmployeeId(5) — wipes ALL her attendance.
  7. Step 3: unlink User → delete UserProfile → delete User.
  8. Step 4: delete the Employee row.
  9. Everything is GONE. Permanently. This is why NOOL urges admins to use "Mark as Left" — soft delete preserves history.
Common service-layer mistakes
Forgetting @Transactional on createEmployee
Multiple inserts (Employee, User, UserProfile) — if one fails, the others must roll back. Without @Transactional you get half-created accounts.
Letting workers view any employee by ID
The self-or-admin check is critical. SecurityConfig allows WORKERs to call the endpoint, but the SERVICE narrows it to "self only". Two layers.
Not syncing mobile updates between Employee and User
The mobile is denormalized — exists on both rows. Editing one without the other causes login/contact mismatch. Always update both together.
Using hard delete as the standard removal flow
Default to updateEmployeeStatus(LEFT) — soft delete preserves history. Hard delete is the nuclear option, reserved for test data and accidents.
Calling employeeRepository.findAll() without pagination on the list endpoint
Loading thousands of rows blows up memory. The PageRequest pattern is mandatory for any potentially-large list.
Forgetting to flush() between unlink and delete during cascade-delete
Hibernate batches operations by default. If you delete the parent before flushing the child unlink, you can get FK violations. flush() forces the buffered changes to run immediately.
Interview questions
Why is the orphan-cleanup logic in createEmployee?
To handle retries after a previously-failed create. If the User row got committed before something downstream failed, that User row would block the retry (mobile is UNIQUE). The cleanup detects "an orphan exists with no business link" and removes it so the retry can succeed. Real production code — born from debugging actual failures.
Why two duplicate checks in updateEmployee — one against Employee and one against User?
Mobile is denormalized between the two tables. The new mobile could clash with another Employee's mobile OR with an unrelated User's mobile (e.g., an owner who happens to have the same number). Both checks are needed. The conditional ensures we don't self-clash with the row being edited.
Why don't workers see other workers via getEmployeeById?
The service does an explicit permission check: if the caller isn't an ADMIN, the requested employeeId must match the caller's own employeeId (from the JWT). If they don't match, AccessDeniedException is thrown. This is defense-in-depth on top of the SecurityConfig URL rules.
Why is getMyProfile a separate endpoint from getEmployeeById?
For UX clarity and security. The worker's frontend doesn't need to know its own employeeId — it just calls GET /employees/me and the backend infers identity from the JWT. This avoids leaking the ID in URLs and prevents the worker from constructing requests that could be confused with admin-style lookups.
When would you actually use deleteEmployee vs updateEmployeeStatus?
Almost always use status-update. Hard delete is for: (a) test data cleanup, (b) removing a worker that was added by mistake before they had any activity, or (c) legal compliance (e.g., GDPR right-to-erasure). For normal "they left the job" situations, status=LEFT preserves the history needed for audit, future salary reconciliation, and tax records.
Could createEmployee skip step 2 (orphan cleanup) and rely on @Transactional?
In theory yes — if @Transactional always rolls back perfectly. In practice, Spring transactions sometimes don't roll back as expected (REQUIRES_NEW propagation, nested transactions, exceptions caught mid-flow). The orphan-cleanup is belt-and-braces: if a previous run left an orphan, the retry self-heals.
Why does the response DTO not include the password (even hashed)?
Three reasons. (1) The password isn't on the Employee entity anyway — it's on the linked User. (2) Even if it were, exposing the hash is a security mistake — attackers can run offline BCrypt brute-force on hashes they obtain. (3) The DTO is the boundary between "what we store" and "what we expose" — only put what the frontend needs.
